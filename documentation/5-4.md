## 1. Core Concept — What is Pipeline Promotion?

**Pipeline promotion** is the process of moving a validated pipeline definition — along with all its dependencies, configuration, and code — from one environment (dev) to the next (staging, then production), in a controlled, reproducible way.

### The Problem Without Promotion Strategy
| Symptom | Cause |
|---|---|
| "It works in dev but not in prod" | Manual, inconsistent configuration between environments |
| Production pipeline breaks after a fix | No staging gate; code untested in production-like env |
| Unknown pipeline changes | No version control; changes made directly in UI |
| Cluster mismatches cause failures | Different compute configs between dev and prod |
| Credentials scattered across environments | No unified secrets/config management |

### The Solution: DABs-Based Promotion
Define your **entire pipeline as code** in a bundle, use **targets** for environment isolation, and deploy identically using CI/CD. The same YAML, the same source code — only the environment-specific values change.

## 2. Infrastructure as Code (IaC) with DABs

### Core Principles
1. **Single Source of Truth** — Everything lives in Git: notebooks, Python code, job definitions, pipeline definitions, cluster specs, permissions
2. **No Manual Changes** — All infrastructure changes flow through Git → CI/CD → `databricks bundle deploy`. UI changes are overwritten on next deploy
3. **Deterministic Deployments** — The same bundle + same `--target` always produces the same result
4. **Co-versioned Code and Config** — Source code and pipeline configuration are committed together and promoted together

### What Goes in Git vs. What Doesn't
| Commit to Git | Do NOT Commit to Git |
|---|---|
| `databricks.yml` and all resource YAML | Built artifacts (`.whl`, `.jar`) |
| Source notebooks, Python, SQL | Credentials, tokens, secrets |
| `tests/` directory | Auto-generated workspace files |
| `.github/workflows/` CI/CD definitions | `.databricks/` lock files (optional) |
| `variable-overrides.json` (non-sensitive only) | Personal `.databrickscfg` files |

## 3. The Promotion Architecture — Targets

The bundle **target** is the fundamental unit of environment promotion. Each target is a named environment slice with its own workspace, mode, variables, and overrides.

### Typical Target Hierarchy
```
Developer laptop  →  dev target     →  dev workspace
PR/CI pipeline   →  validate only
Merge to main    →  staging target  →  staging workspace
Manual approval  →  prod target     →  prod workspace
```

### Target Structure Template
```yaml
targets:
  dev:
    default: true                    # Used when no -t flag specified
    mode: development                # Auto-prefix, paused schedules, no lock
    workspace:
      host: https://dev-adb.azuredatabricks.net
    variables:
      catalog_name: dev_catalog
      num_workers: 2
      cluster_node_type: Standard_DS3_v2

  staging:
    mode: production                 # Treated like prod for config strictness
    workspace:
      host: https://staging-adb.azuredatabricks.net
    variables:
      catalog_name: staging_catalog
      num_workers: 4
      cluster_node_type: Standard_DS4_v2

  prod:
    mode: production
    workspace:
      host: https://prod-adb.azuredatabricks.net
    git:
      branch: main                   # Block deploys from non-main branches
    variables:
      catalog_name: prod_catalog
      num_workers: 8
      cluster_node_type: Standard_DS5_v2
    run_as:
      service_principal_name: "prod-sp@company.com"
    permissions:
      - level: CAN_MANAGE
        group_name: data-engineers
      - level: CAN_RUN
        group_name: analysts
```

### Deploying to Each Target
```bash
# Developer promotes to their own dev space (default target)
databricks bundle deploy

# CI/CD deploys to staging (after merge to main)
databricks bundle deploy -t staging

# Ops promotes to production (after approval gate)
databricks bundle deploy -t prod

# Emergency production fix (bypass branch check)
databricks bundle deploy -t prod --force
```

## 4. Environment-Specific Configuration Techniques

### Technique 1: Variables for Scalar Values
Best for: catalog names, schema names, email addresses, cluster sizes, row limits

```yaml
variables:
  catalog_name:
    description: "Unity Catalog catalog"
    default: dev_catalog
  schema_name:
    description: "Schema for pipeline output"
    default: etl_silver
  alert_email:
    description: "Alert recipient"
    default: dev-team@company.com
  num_workers:
    description: "Cluster worker count"
    default: 2

targets:
  dev:
    variables:
      catalog_name: dev_catalog
      schema_name: etl_silver
      alert_email: dev-team@company.com
      num_workers: 2

  staging:
    variables:
      catalog_name: staging_catalog
      schema_name: etl_silver
      alert_email: staging-alerts@company.com
      num_workers: 4

  prod:
    variables:
      catalog_name: prod_catalog
      schema_name: etl_silver
      alert_email: oncall@company.com
      num_workers: 8
```

### Technique 2: Substitutions for Dynamic Values
Built-in substitutions that are always environment-aware:

```yaml
workspace:
  # Unique path per user per environment — prevents dev collisions
  root_path: /Workspace/Users/${workspace.current_user.userName}/.bundle/${bundle.name}/${bundle.target}

resources:
  jobs:
    etl_job:
      name: "ETL Job [${bundle.target}]"    # Job name shows current env
      tags:
        environment: ${bundle.target}        # Tag with env name
        deployed_by: ${workspace.current_user.userName}

  pipelines:
    silver_pipeline:
      name: silver_pipeline
      schema: silver_${bundle.target}        # Env-specific schema
```

### Technique 3: Per-Target Resource Overrides
Best for: cluster sizes, schedule cron expressions, concurrent run limits

```yaml
# Base resources (top level)
resources:
  jobs:
    etl_job:
      name: "ETL Job"
      schedule:
        quartz_cron_expression: "0 0 8 * * ?"   # 8am UTC
        pause_status: PAUSED                      # Overridden per target
      tasks:
        - task_key: transform
          notebook_task:
            notebook_path: ./src/transform.ipynb
          job_cluster_key: main_cluster
      job_clusters:
        - job_cluster_key: main_cluster
          new_cluster:
            spark_version: "15.4.x-scala2.12"
            node_type_id: Standard_DS3_v2        # Default (dev)
            num_workers: 2

# Per-target overrides
targets:
  staging:
    resources:
      jobs:
        etl_job:
          schedule:
            pause_status: UNPAUSED               # Active in staging
          job_clusters:
            - job_cluster_key: main_cluster
              new_cluster:
                node_type_id: Standard_DS4_v2    # Bigger for staging
                num_workers: 4

  prod:
    resources:
      jobs:
        etl_job:
          schedule:
            pause_status: UNPAUSED               # Active in prod
          email_notifications:
            on_failure:
              - oncall@company.com               # Prod-only alerting
          job_clusters:
            - job_cluster_key: main_cluster
              new_cluster:
                node_type_id: Standard_DS5_v2    # Largest for prod
                num_workers: 8
```

> **Precedence rule**: Settings in `targets.<target>.resources` always override the top-level `resources` block.

## 5. Resource-Level Target Overrides

### Job Cluster Override Pattern
```yaml
resources:
  jobs:
    my-job:
      job_clusters:
        - job_cluster_key: shared-cluster
          new_cluster:
            node_type_id: i3.xlarge        # Dev/default: smaller
            num_workers: 2

targets:
  prod:
    resources:
      jobs:
        my-job:
          job_clusters:
            - job_cluster_key: shared-cluster
              new_cluster:
                node_type_id: i3.4xlarge   # Prod override: larger
                num_workers: 16
```

### DLT Pipeline Cluster Override Pattern
```yaml
resources:
  pipelines:
    silver_pipeline:
      clusters:
        - label: default
          num_workers: 2
          node_type_id: Standard_DS3_v2    # Dev default

targets:
  prod:
    resources:
      pipelines:
        silver_pipeline:
          clusters:
            - label: default
              num_workers: 8
              node_type_id: Standard_DS5_v2  # Prod: production-sized
```

### Override Fields Reference
| What to Override | How |
|---|---|
| Cluster size | `targets.<t>.resources.jobs.<job>.job_clusters[]` |
| DLT cluster | `targets.<t>.resources.pipelines.<p>.clusters[]` |
| Schedule | `targets.<t>.resources.jobs.<job>.schedule` |
| Email notifications | `targets.<t>.resources.jobs.<job>.email_notifications` |
| Max concurrent runs | `targets.<t>.resources.jobs.<job>.max_concurrent_runs` |
| Catalog / schema | Via `${var.<name>}` variables |
| Workspace | `targets.<t>.workspace.host` |
| Run identity | `targets.<t>.run_as` |
| Permissions | `targets.<t>.permissions` |

## 6. Unity Catalog — Catalog & Schema Promotion

### Pattern 1: Variables for Catalog/Schema (Most Common)
```yaml
variables:
  catalog:
    default: dev_catalog
  schema:
    default: silver

targets:
  dev:
    variables:
      catalog: dev_catalog
      schema: silver
  staging:
    variables:
      catalog: staging_catalog
      schema: silver
  prod:
    variables:
      catalog: prod_catalog
      schema: silver

resources:
  pipelines:
    silver_pipeline:
      catalog: ${var.catalog}
      schema: ${var.schema}
      name: silver_pipeline_${bundle.target}
```

### Pattern 2: Schema per Environment
Useful when environments should write to entirely separate schemas:

```yaml
variables:
  catalog:
    default: shared_catalog
  silver_schema:
    default: silver_dev

targets:
  dev:
    variables:
      silver_schema: silver_dev
  staging:
    variables:
      silver_schema: silver_staging
  prod:
    variables:
      silver_schema: silver_prod    # e.g., just 'silver'

resources:
  pipelines:
    silver_pipeline:
      catalog: ${var.catalog}
      schema: ${var.silver_schema}
```

### Pattern 3: Managing Catalogs as Bundle Resources
With CLI v0.287.0+, you can define Unity Catalog catalogs and schemas directly in the bundle:

```yaml
resources:
  catalogs:
    my_catalog:
      name: ${var.catalog}
      comment: "Managed by bundle ${bundle.name}"

  schemas:
    silver_schema:
      catalog_name: ${var.catalog}
      name: ${var.silver_schema}
      comment: "Silver layer schema for ${bundle.target}"
```

## 7. Deployment Modes in Multi-Environment Context

### How Mode Affects Promotion

| Behavior | `mode: development` (dev) | `mode: production` (staging/prod) |
|---|---|---|
| Resource name prefix | `[dev <username>]` prepended | No prefix — clean names |
| Job schedules | Auto-paused | Active (as configured) |
| DLT pipeline mode | `development: true` auto-set | `development: false` enforced |
| Deployment lock | Disabled (fast iteration) | Enabled (prevents concurrent deploys) |
| Git branch check | Not enforced | Enforced if `git.branch` is set |
| Cluster override | `--cluster-id` allowed | `--cluster-id` blocked |

### Why `mode: production` on Staging?
Setting `mode: production` on the staging target (not just prod) provides:
- Production-like validation at the staging gate
- Active schedules run and validate timing
- Real DLT pipeline behavior (not dev mode)
- Stricter naming (no prefix) — mirrors production naming exactly

```yaml
targets:
  staging:
    mode: production    # Intentional — validate prod-like behavior
  prod:
    mode: production
```

## 8. Branching Strategy & GitOps

### Recommended: Trunk-Based Development
One main branch (`main`) is always deployment-ready. Feature branches are short-lived.

```
feature/my-feature  →  PR  →  main
                              │
                    ┌─────────┼──────────────────────┐
                    ▼         ▼                       ▼
                validate   deploy staging         deploy prod
                (on PR)    (on merge)           (manual gate)
```

### Branching Rules
| Branch | Purpose | Auto-Deploys to |
|---|---|---|
| `feature/*` | Development work | Dev (manual: `bundle deploy`) |
| `main` | Production-ready code | Staging (automatic after merge) |
| Git tags | Release markers | Prod (manual approval + `bundle deploy -t prod`) |

### GitOps Enforcement
Use `git.branch` in your prod target to ensure production is always deployed from `main`:

```yaml
targets:
  prod:
    mode: production
    git:
      branch: main    # CLI validates current branch == main before deploy
```

If a developer accidentally runs `databricks bundle deploy -t prod` from `feature/my-branch`, the deploy fails automatically.

## 9. Full Promotion CI/CD Workflow

### GitHub Actions — Complete Pipeline
```yaml
name: Databricks Bundle Promotion

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  # ──────────────── STAGE 1: Validate on PR ────────────────
  validate:
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: databricks/setup-cli@v1

      - name: Validate bundle configuration
        env:
          DATABRICKS_HOST: ${{ vars.STAGING_HOST }}
          DATABRICKS_CLIENT_ID: ${{ vars.DATABRICKS_CLIENT_ID }}
        run: |
          databricks bundle validate -t staging
          echo "Validation passed"

      - name: Run unit tests
        run: |
          pip install pytest
          pytest tests/unit/ -v

  # ──────────────── STAGE 2: Deploy to Staging on merge ────
  deploy-staging:
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - uses: actions/checkout@v4
      - uses: databricks/setup-cli@v1

      - name: Deploy to staging
        env:
          DATABRICKS_HOST: ${{ vars.STAGING_HOST }}
          DATABRICKS_CLIENT_ID: ${{ vars.STAGING_CLIENT_ID }}
        run: databricks bundle deploy -t staging

      - name: Run integration tests
        env:
          DATABRICKS_HOST: ${{ vars.STAGING_HOST }}
          DATABRICKS_CLIENT_ID: ${{ vars.STAGING_CLIENT_ID }}
        run: |
          databricks bundle run -t staging integration_test_job
          echo "Integration tests passed"

  # ──────────────── STAGE 3: Promote to Production ─────────
  deploy-prod:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment: production               # Requires manual approval in GitHub
    steps:
      - uses: actions/checkout@v4
      - uses: databricks/setup-cli@v1

      - name: Deploy to production
        env:
          DATABRICKS_HOST: ${{ vars.PROD_HOST }}
          DATABRICKS_CLIENT_ID: ${{ vars.PROD_CLIENT_ID }}
        run: |
          databricks bundle validate -t prod
          databricks bundle deploy -t prod
          echo "Production deployment complete"
```

### Promotion Flow Summary
```
1. Developer → creates feature branch
2. Developer → databricks bundle deploy (dev, local)
3. Developer → opens Pull Request to main
4. CI Pipeline → databricks bundle validate -t staging
5. CI Pipeline → pytest tests/unit/
6. Team → code review & approve PR
7. Merge to main →
8. CI Pipeline → databricks bundle deploy -t staging
9. CI Pipeline → runs integration tests on staging
10. Ops/Lead → approves production gate
11. CI Pipeline → databricks bundle deploy -t prod
```

## 10. Configuration Drift — Prevention & Detection

### What is Configuration Drift?
**Configuration drift** occurs when the actual state of a deployed resource (job, pipeline, cluster) diverges from the desired state defined in your YAML files.

**Common causes:**
- Manual changes made directly in the Databricks UI
- Hotfixes applied via CLI without updating the YAML
- Multiple developers deploying different local versions

### Prevention with DABs
```bash
# Every bundle deploy is idempotent — it reconciles the actual state
# back to the desired state defined in databricks.yml
databricks bundle deploy -t prod

# Running deploy on a schedule or on every PR merge
# acts as a continuous drift reconciliation
```

> **Key concept**: `databricks bundle deploy` is **idempotent** — running it repeatedly produces the same result and corrects any manual changes made outside the bundle.

### Detection
If someone manually changes a job in the Databricks UI, the next `bundle deploy` will overwrite it. To detect drift before it is corrected:

```bash
# Preview what changes deploy would make (without applying)
databricks bundle plan -t prod

# Output shows: create / update / delete for each resource
# No changes = no drift; changes listed = drift detected
```

### Governance Rules
- Grant `CAN_MANAGE` only to the CI/CD service principal on prod resources
- Remove `CAN_MANAGE` for individual user accounts on production
- All production changes must come through a PR and the CI/CD pipeline

## 11. Security Across Environments

### Identity Model per Environment

| Environment | Deployer Identity | Job Run Identity |
|---|---|---|
| Dev | Individual developer (OAuth U2M) | Developer's own identity |
| Staging | CI/CD service principal | Staging service principal |
| Prod | CI/CD service principal (OIDC) | Prod service principal |

### `run_as` Configuration
```yaml
targets:
  dev:
    run_as:
      user_name: "${workspace.current_user.userName}"   # Developer's identity

  staging:
    run_as:
      service_principal_name: "staging-sp@company.com"

  prod:
    run_as:
      service_principal_name: "prod-sp@company.com"
```

### Secrets Management Across Environments
Never put secret values in YAML. Use secret scopes with environment-specific values:

```yaml
# In notebook tasks — reference secrets, not values
resources:
  jobs:
    etl_job:
      tasks:
        - task_key: ingest
          notebook_task:
            base_parameters:
              db_password: "{{secrets/my-scope/db-password}}"
```

```bash
# Different scopes per environment
# Dev:
databricks secrets put-secret dev-scope db-password --string-value "DevPass!" --profile dev

# Prod:
databricks secrets put-secret prod-scope db-password --string-value "ProdPass!" --profile prod
```

## 12. Artifact Promotion — Wheels and JARs

Build artifacts once per commit; promote the same artifact across environments.

### Bundle-Managed Build
```yaml
artifacts:
  my_package:
    type: whl
    path: ./src/my_package
    build: "pip wheel . --no-deps -w dist/"
```

```bash
# bundle deploy builds + uploads the wheel before deploying
databricks bundle deploy -t staging
databricks bundle deploy -t prod
# Both envs receive the same wheel built from the same commit
```

### Best Practice: Build Once in CI/CD
```yaml
# In GitHub Actions:
- name: Build wheel once
  run: pip wheel ./src/my_package -w dist/ --no-deps

- name: Deploy to staging (uses pre-built wheel)
  run: databricks bundle deploy -t staging

- name: Deploy to prod (uses same pre-built wheel)
  run: databricks bundle deploy -t prod
```

### Uploading to Unity Catalog Volumes for Promotion
```bash
# Upload wheel to a central volume accessible by all environments
databricks fs cp dist/my_package-0.1.0-py3-none-any.whl \
  /Volumes/shared_catalog/libraries/releases/my_package-0.1.0-py3-none-any.whl
```

## 13. Testing Strategy per Environment

### Test Types per Stage

| Stage | Test Type | Tool | Triggered By |
|---|---|---|---|
| Dev (local) | Unit tests | `pytest` | Developer locally |
| PR validation | Unit + config validation | `pytest` + `bundle validate` | PR open |
| Staging | Integration tests | `bundle run integration_test_job` | Auto after staging deploy |
| Pre-prod | Smoke tests | `bundle run smoke_test_job` | Post-staging approval |
| Production | Monitoring | Data quality monitors | Scheduled |

### Integration Test as a Bundle Job
```yaml
resources:
  jobs:
    integration_test_job:
      name: "Integration Tests [${bundle.target}]"
      tasks:
        - task_key: run_tests
          python_wheel_task:
            package_name: my_package
            entry_point: run_integration_tests
            parameters:
              - "--catalog"
              - ${var.catalog_name}
              - "--env"
              - ${bundle.target}
          job_cluster_key: test_cluster
      job_clusters:
        - job_cluster_key: test_cluster
          new_cluster:
            spark_version: "15.4.x-scala2.12"
            node_type_id: Standard_DS3_v2
            num_workers: 1
```

```yaml
        group_name: data-engineers
      - level: CAN_RUN

```

### `resources/pipelines/silver_pipeline.yml`
```yaml
resources:
  pipelines:
    silver_pipeline:
      name: "Silver Pipeline [${bundle.target}]"
      catalog: ${var.catalog_name}
      schema: ${var.silver_schema}
      libraries:
        - notebook:
            path: ../../src/pipelines/silver.ipynb
      clusters:
        - label: default
          num_workers: ${var.num_workers}
          node_type_id: ${var.node_type}
      configuration:
        bundle.sourcePath: ${workspace.file_path}/src
```

### `resources/jobs/etl_job.yml`
```yaml
resources:
  jobs:
    etl_job:
      name: "ETL Pipeline [${bundle.target}]"
      schedule:
        quartz_cron_expression: "0 0 6 * * ?"
        timezone_id: "America/Chicago"
      email_notifications:
        on_failure:
          - ${var.alert_email}
      tasks:
        - task_key: ingest
          notebook_task:
            notebook_path: ../../src/notebooks/ingest.ipynb
            base_parameters:
              catalog: ${var.catalog_name}
              schema: ${var.silver_schema}
              env: ${bundle.target}
          job_cluster_key: main_cluster
        - task_key: transform
          depends_on:
            - task_key: ingest
          notebook_task:
            notebook_path: ../../src/notebooks/transform.ipynb
          job_cluster_key: main_cluster
      job_clusters:
        - job_cluster_key: main_cluster
          new_cluster:
            spark_version: "15.4.x-scala2.12"
            node_type_id: ${var.node_type}
            num_workers: ${var.num_workers}
```

## 15. Anti-Patterns to Avoid

| Anti-Pattern | Why It's Wrong | Correct Approach |
|---|---|---|
| Maintaining separate YAML files per environment | Duplication, divergence, inconsistent configs | One YAML with `targets` for all environments |
| Hardcoding workspace URLs in source code | Breaks in other environments | Use `${workspace.host}` substitution |
| Hardcoding catalog names in notebook code | Cannot promote between environments | Pass as `base_parameters` using `${var.catalog}` |
| Committing PAT tokens to Git | Critical security vulnerability | Use OIDC or environment variable secrets |
| Making UI changes directly to production | Configuration drift, loss of version control | All changes via Git → CI/CD → bundle deploy |
| Deploying to prod from a feature branch | Bypasses staging gate, risk of untested code | Enforce `git.branch: main` in prod target |
| Using `mode: development` for staging | Paused schedules, different naming — not production-like | Use `mode: production` for staging too |
| Building artifacts per deploy | Non-deterministic if code changes mid-pipeline | Build once, promote the same artifact |
| Giving individual users `CAN_MANAGE` on prod | Enables manual UI changes, drift | Grant only service principal `CAN_MANAGE` on prod |

## 1. Overview & CLI Versions

### What is the Databricks CLI?
The Databricks CLI is a command-line tool that wraps the Databricks REST API and provides first-class support for Declarative Automation Bundles (DABs). It is the primary tool for CI/CD automation, asset management, and workspace administration.

### CLI Version History
| Version | Status | Auth |
|---|---|---|
| v0.18 and below | **Deprecated / Legacy** | PAT-based only |
| v0.205+ (current) | **Current — use this** | OAuth, M2M, OIDC |

> **Key rule**: Always use the current CLI (v0.205+). The legacy CLI is no longer supported.

### What the CLI can do
- Manage **Declarative Automation Bundles** — validate, deploy, run, destroy
- Manage **jobs, clusters, pipelines, SQL warehouses**
- Manage **files** (DBFS, Volumes) and **workspace notebooks**
- Manage **secrets** (scopes, keys)
- Execute **direct REST API calls**
- Authenticate with multiple workspaces via **profiles**

## 3. Authentication

### Method 1: OAuth User-to-Machine (U2M) — Interactive Login
The recommended method for local development. Opens a browser window.

```bash
# Authenticate to a workspace
databricks auth login --host https://<workspace-url>

# Example
databricks auth login --host https://adb-111111.1.azuredatabricks.net

# Authenticate to an account console
databricks auth login --host https://accounts.azuredatabricks.net --account-id <account-id>
```

### Method 2: OAuth Machine-to-Machine (M2M) — Service Principal
Used for CI/CD. Requires a service principal with a client secret.

```bash
# Using environment variables (CI/CD)
export DATABRICKS_HOST=https://<workspace-url>
export DATABRICKS_CLIENT_ID=<service-principal-app-id>
export DATABRICKS_CLIENT_SECRET=<service-principal-secret>
```

Or in `~/.databrickscfg`:
```ini
[prod]
host            = https://adb-111111.1.azuredatabricks.net
client_id       = <sp-application-id>
client_secret   = <sp-secret>
```

### Method 3: Personal Access Token (PAT) — Legacy
Still functional but not recommended for new projects.

```ini
[DEFAULT]
host  = https://adb-111111.1.azuredatabricks.net
token = dapiXXXXXXXXXXXXXXXX
```

### Authentication Precedence (Unified Auth)
When the CLI resolves credentials, it checks in this order:
1. Environment variables (`DATABRICKS_HOST`, `DATABRICKS_TOKEN`, etc.)
2. Named profile via `--profile` flag
3. `DEFAULT` profile in `~/.databrickscfg`

### Check Authentication Status
```bash
# List all configured profiles
databricks auth profiles

# Verify a specific profile
databricks auth describe --profile prod

# Get current user info
databricks current-user me
```

## 4. Configuration Profiles — `.databrickscfg`

The CLI stores all workspace connections in `~/.databrickscfg` (Linux/macOS) or `%USERPROFILE%\.databrickscfg` (Windows).

### Profile File Format
```ini
[DEFAULT]
host  = https://dev-adb.azuredatabricks.net
token = dapi...

[staging]
host            = https://staging-adb.azuredatabricks.net
client_id       = staging-sp-id
client_secret   = staging-sp-secret

[prod]
host            = https://prod-adb.azuredatabricks.net
client_id       = prod-sp-id
client_secret   = prod-sp-secret
```

### Using a Profile
```bash
# Use the DEFAULT profile (implicit)
databricks clusters list

# Use a named profile explicitly
databricks clusters list --profile prod
databricks clusters list -p staging

# Override host for a single command
databricks jobs list --host https://other-workspace.net
```

## 5. General CLI Syntax & Help System

### Command Structure
```
databricks <command-group> <command> <subcommand> [arguments] [flags]
```

### Help at Every Level
```bash
# List all top-level command groups
databricks -h

# List commands within a group
databricks jobs -h
databricks bundle -h
databricks clusters -h

# Get full usage for a specific command
databricks bundle deploy -h
databricks jobs create -h
databricks secrets put-secret -h
```

### Common Global Flags
| Flag | Description |
|---|---|
| `-h`, `--help` | Show help for any command |
| `-p`, `--profile <name>` | Use a named config profile |
| `--host <url>` | Override workspace URL |
| `--output json` | Output results as JSON |
| `--output text` | Output results as human-readable text (default) |
| `-t`, `--target <name>` | Bundle target (dev/staging/prod) |
| `--var="key=value"` | Override a bundle variable |
| `--debug` | Enable debug logging |

## 6. Bundle Commands — Core CI/CD Workflow

The `databricks bundle` command group is the backbone of CI/CD workflows with Declarative Automation Bundles.

### 6.1 `bundle init` — Scaffold New Project
```bash
# Interactive template selection
databricks bundle init

# Built-in templates:
#   default-python  — Jobs + notebooks (Python)
#   dlt             — Delta Live Tables pipeline
#   ml              — Machine learning project
#   default-sql     — SQL-based workflows
```

### 6.2 `bundle validate` — Verify Configuration
Checks YAML syntax, schema, variable resolution. Makes **zero changes** to the workspace.

```bash
# Validate default target
databricks bundle validate

# Validate a specific target
databricks bundle validate -t prod

# Validate with variable overrides
databricks bundle validate -t prod --var="catalog_name=prod_catalog"

# Export the resolved config as JSON (useful for debugging)
databricks bundle validate --output json
```

### 6.3 `bundle deploy` — Deploy Resources
Deploys all resources defined in the bundle to the target workspace.

```bash
# Deploy to default target
databricks bundle deploy

# Deploy to specific target
databricks bundle deploy -t staging
databricks bundle deploy -t prod

# Deploy with variable override
databricks bundle deploy -t prod --var="num_workers=8"

# Force deploy (bypass Git branch check in production mode)
databricks bundle deploy -t prod --force

# Dev: override all cluster definitions with a specific cluster
databricks bundle deploy -t dev --cluster-id 1234-567890-abcde123

# Use a specific auth profile
databricks bundle deploy -t prod --profile prod-service-principal
```

### 6.4 `bundle run` — Execute a Resource
Triggers a specific job or pipeline in the bundle.

```bash
# Run a job by its key name
databricks bundle run my_etl_job

# Run for a specific target
databricks bundle run -t prod my_etl_job

# Run a pipeline
databricks bundle run silver_pipeline

# Pass Python parameters to the job
databricks bundle run my_etl_job --python-params '["--env", "prod"]'

# Pass notebook parameters
databricks bundle run my_etl_job --notebook-params '{"catalog": "prod_catalog"}'
```

### 6.5 `bundle destroy` — Remove Deployed Resources
Permanently deletes all resources deployed by the bundle. **Irreversible.**

```bash
# Destroy with confirmation prompt
databricks bundle destroy -t dev

# Destroy without prompting (CI/CD automation)
databricks bundle destroy -t dev --auto-approve
```

### 6.6 Other Bundle Commands

```bash
# Generate YAML from existing workspace resources
databricks bundle generate

# Preview changes without deploying
databricks bundle plan -t prod

# Show summary of deployed resources
databricks bundle summary

# Export the full JSON schema for IDE autocomplete
databricks bundle schema > bundle_config_schema.json
```

### Bundle Command Quick Reference
| Command | Action |
|---|---|
| `bundle init` | Scaffold new project from template |
| `bundle validate` | Check syntax + schema (no deploy) |
| `bundle deploy -t <target>` | Deploy to workspace |
| `bundle run <resource-key>` | Trigger job or pipeline |
| `bundle destroy -t <target>` | Delete deployed resources |
| `bundle generate` | Generate YAML from existing workspace resources |
| `bundle plan -t <target>` | Preview deployment changes |
| `bundle summary` | Show deployed resource status |
| `bundle schema` | Export JSON schema for IDE |

## 7. Managing Assets — Jobs, Clusters, Pipelines

### Jobs
```bash
# List all jobs
databricks jobs list

# Get job details (by job ID)
databricks jobs get --job-id 12345

# Create a job from a JSON file
databricks jobs create --json @job_definition.json

# Update (reset) a job
databricks jobs reset --job-id 12345 --json @updated_job.json

# Delete a job
databricks jobs delete --job-id 12345

# Run a job immediately (ad hoc)
databricks jobs run-now --job-id 12345

# List recent job runs
databricks jobs list-runs --job-id 12345

# Get output/status of a run
databricks jobs get-run --run-id 678901

# Cancel a running job run
databricks jobs cancel-run --run-id 678901
```

### Clusters
```bash
# List all clusters
databricks clusters list

# Get cluster details
databricks clusters get --cluster-id 1234-567890-abcde

# Create a cluster from JSON
databricks clusters create --json @cluster_spec.json

# Start a stopped cluster
databricks clusters start --cluster-id 1234-567890-abcde

# Restart a cluster
databricks clusters restart --cluster-id 1234-567890-abcde

# Delete (terminate) a cluster
databricks clusters delete --cluster-id 1234-567890-abcde

# List available Spark versions
databricks clusters spark-versions

# List available node types
databricks clusters list-node-types
```

### Pipelines (DLT / Lakeflow)
```bash
# List all pipelines
databricks pipelines list-pipelines

# Get pipeline details
databricks pipelines get --pipeline-id abc123

# Create a pipeline from JSON
databricks pipelines create --json @pipeline_spec.json

# Start a pipeline update (full refresh)
databricks pipelines start-update --pipeline-id abc123 --full-refresh

# Stop a running pipeline
databricks pipelines stop --pipeline-id abc123

# Delete a pipeline
databricks pipelines delete --pipeline-id abc123
```

### SQL Warehouses
```bash
# List all SQL warehouses
databricks warehouses list

# Get warehouse details
databricks warehouses get --id abc123

# Start a warehouse
databricks warehouses start --id abc123

# Stop a warehouse
databricks warehouses stop --id abc123
```

## 8. Managing Files — `fs` Command Group

The `fs` command group manages files in **DBFS** and **Unity Catalog Volumes**.

### File Path Prefixes
| Path Type | Prefix | Example |
|---|---|---|
| DBFS | `dbfs:/` | `dbfs:/tmp/data.csv` |
| Unity Catalog Volume | `/Volumes/` | `/Volumes/catalog/schema/vol/file.csv` |
| Local filesystem | None | `./local-file.csv` |

### Commands
```bash
# List files in DBFS
databricks fs ls dbfs:/tmp/
databricks fs ls dbfs:/user/hive/warehouse/

# List with details (size, modification time)
databricks fs ls dbfs:/tmp/ --long

# Copy local file to DBFS
databricks fs cp ./local-data.csv dbfs:/tmp/data.csv

# Copy DBFS file to local
databricks fs cp dbfs:/tmp/output.parquet ./local-output.parquet

# Copy recursively (directory)
databricks fs cp ./data-dir/ dbfs:/mnt/data/ --recursive

# Remove a file
databricks fs rm dbfs:/tmp/old-data.csv

# Remove a directory recursively
databricks fs rm dbfs:/tmp/old-dir/ --recursive

# Create a directory in DBFS
databricks fs mkdirs dbfs:/tmp/new-folder/

# Copy between Unity Catalog Volumes
databricks fs cp /Volumes/catalog/schema/src/file.txt /Volumes/catalog/schema/dest/
```

## 9. Managing Workspace Objects

The `workspace` command group manages notebooks, files, and directories in the Databricks workspace.

```bash
# List workspace contents
databricks workspace list /Workspace/Users/alice@company.com/

# Get object status (type, language, path)
databricks workspace get-status /Workspace/Users/alice@company.com/my-notebook

# Import a notebook from local filesystem
databricks workspace import \
  ./local-notebook.py \
  /Workspace/Users/alice@company.com/my-notebook \
  --language PYTHON \
  --format SOURCE

# Export a notebook
databricks workspace export \
  /Workspace/Users/alice@company.com/my-notebook \
  --format SOURCE \
  --file ./local-notebook.py

# Export as Jupyter notebook format
databricks workspace export \
  /Workspace/Users/alice@company.com/my-notebook \
  --format JUPYTER \
  --file ./my-notebook.ipynb

# Create a directory
databricks workspace mkdirs /Workspace/Users/alice@company.com/new-project/

# Delete a file or notebook
databricks workspace delete /Workspace/Users/alice@company.com/temp-notebook

# Delete a directory recursively
databricks workspace delete /Workspace/Users/alice@company.com/old-project/ --recursive
```

### Import Format Options
| Format | Value | Use for |
|---|---|---|
| Source code | `SOURCE` | `.py`, `.scala`, `.sql`, `.r` notebooks |
| Jupyter | `JUPYTER` | `.ipynb` notebooks |
| HTML | `HTML` | Exported HTML (import not supported) |
| DBC | `DBC` | Databricks archive (batch import/export) |

## 10. Managing Secrets

Secrets allow you to store credentials and sensitive values securely. They are accessed in notebooks via `dbutils.secrets.get(scope, key)`.

```bash
# List all secret scopes in the workspace
databricks secrets list-scopes

# Create a new secret scope
databricks secrets create-scope my-scope

# Create a scope with Azure Key Vault backing
databricks secrets create-scope my-akv-scope \
  --scope-backend-type AZURE_KEYVAULT \
  --resource-id /subscriptions/.../vaults/my-vault \
  --dns-name https://my-vault.vault.azure.net/

# Add or update a secret (string value)
databricks secrets put-secret my-scope db-password --string-value "MySecret123!"

# Add a secret from stdin (avoids shell history)
echo "MySecret123!" | databricks secrets put-secret my-scope db-password

# Add a secret from a file (for binary/multi-line secrets)
databricks secrets put-secret my-scope tls-cert --bytes-value "$(cat cert.pem | base64)"

# List all secrets in a scope (names only — values are never shown)
databricks secrets list-secrets my-scope

# Delete a specific secret
databricks secrets delete-secret my-scope db-password

# Delete an entire scope
databricks secrets delete-scope my-scope
```

### Accessing Secrets in Notebooks
```python
# In a Databricks notebook
password = dbutils.secrets.get(scope="my-scope", key="db-password")

# List available scopes (from notebook)
dbutils.secrets.listScopes()

# List keys in a scope (from notebook)
dbutils.secrets.list("my-scope")
```

## 11. Direct REST API Calls — `api` Command

When a native CLI command doesn't exist, use `databricks api` to call the REST API directly.

```bash
# Syntax
databricks api <HTTP-METHOD> <api-path> [--json <payload>]

# GET — List clusters
databricks api get /api/2.0/clusters/list

# GET — Get job details
databricks api get /api/2.1/jobs/get --json '{"job_id": 12345}'

# POST — Create a cluster
databricks api post /api/2.0/clusters/create --json @cluster_spec.json

# POST — Run a job now
databricks api post /api/2.1/jobs/run-now --json '{"job_id": 12345}'

# DELETE — Delete a job run
databricks api delete /api/2.1/jobs/runs/delete --json '{"run_id": 678901}'

# PATCH — Update a Git folder (Repos)
databricks api patch /api/2.0/repos/123456 --json '{"branch": "main"}'
```

> **When to use `api`**: Use when a specific operation isn't yet available as a native CLI command, or when you need to call a REST endpoint that is not covered by the named command groups.

## 12. CI/CD Workflow with GitHub Actions

### Recommended Authentication: OIDC (Workload Identity Federation)
No stored secrets — GitHub issues short-lived tokens automatically.

**Step 1: Create Service Principal in Databricks and configure federation policy**
- Issuer: `https://token.actions.githubusercontent.com`
- Subject: `repo:my-org/my-repo:environment:Prod`

**Step 2: Store non-sensitive values as GitHub Actions variables (not secrets)**
```
DATABRICKS_HOST = https://adb-111111.1.azuredatabricks.net
DATABRICKS_CLIENT_ID = <service-principal-app-id>
```

**Step 3: GitHub Actions Workflow**
```yaml
name: Deploy to Databricks

on:
  push:
    branches: [main]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: databricks/setup-cli@v1      # Install Databricks CLI
      - name: Validate Bundle
        env:
          DATABRICKS_HOST: ${{ vars.DATABRICKS_HOST }}
          DATABRICKS_CLIENT_ID: ${{ vars.DATABRICKS_CLIENT_ID }}
        run: databricks bundle validate -t staging

  deploy:
    needs: validate
    runs-on: ubuntu-latest
    environment: Prod                       # Must match federation policy subject
    steps:
      - uses: actions/checkout@v4
      - uses: databricks/setup-cli@v1
      - name: Deploy to Production
        env:
          DATABRICKS_HOST: ${{ vars.DATABRICKS_HOST }}
          DATABRICKS_CLIENT_ID: ${{ vars.DATABRICKS_CLIENT_ID }}
        run: databricks bundle deploy -t prod
```

### Alternative: OAuth M2M (Service Principal + Client Secret)
```yaml
- name: Deploy with OAuth M2M
  env:
    DATABRICKS_HOST: ${{ vars.DATABRICKS_HOST }}
    DATABRICKS_CLIENT_ID: ${{ vars.DATABRICKS_CLIENT_ID }}
    DATABRICKS_CLIENT_SECRET: ${{ secrets.DATABRICKS_CLIENT_SECRET }}
  run: databricks bundle deploy -t prod
```

### Installing the CLI in CI/CD
```yaml
# Option 1: Official action (always latest)
- uses: databricks/setup-cli@v1

# Option 2: Pin to a specific version
- uses: databricks/setup-cli@v0.205.0

# Option 3: Manual install (Linux)
- name: Install Databricks CLI
  run: |
    curl -fsSL https://raw.githubusercontent.com/databricks/setup-cli/main/install.sh | sh
```

### Recommended CI/CD Pattern
```
┌─────────────────────────────────────────────────────────┐
│                   CI/CD WORKFLOW                        │
├──────────────────────────┬──────────────────────────────┤
│ Event: PR opened         │ databricks bundle validate   │
│ Event: Merge to main     │ databricks bundle deploy -t staging │
│ Event: Manual approval   │ databricks bundle deploy -t prod    │
│ Event: Rollback needed   │ databricks bundle destroy -t prod   │
└──────────────────────────┴──────────────────────────────┘
```

### Authentication Methods Comparison
| Method | Security | Use Case |
|---|---|---|
| **OIDC / Workload Identity** | Highest — no stored secrets | GitHub Actions (recommended) |
| **OAuth M2M (client_secret)** | High — requires secret rotation | CI/CD without OIDC support |
| **PAT (Personal Access Token)** | Low — long-lived, user-linked | Local dev only (not CI/CD) |
| **OAuth U2M (browser login)** | High — interactive only | Local development |

## 13. Output Formats & Flags

```bash
# JSON output (machine-readable, for scripting)
databricks clusters list --output json
databricks jobs list --output json | jq '.jobs[] | .job_id'

# Text output (human-readable, default)
databricks clusters list --output text

# Debug mode (shows raw API requests/responses)
databricks clusters list --debug

# Limit number of results
databricks jobs list --limit 20

# Filter by name
databricks clusters list --filter-by.cluster-name my-cluster
```

### Piping and Scripting
```bash
# Get job ID by name using jq
JOB_ID=$(databricks jobs list --output json | jq '.jobs[] | select(.settings.name=="My ETL Job") | .job_id')

# Wait for a job run to complete
RUN_ID=$(databricks jobs run-now --job-id $JOB_ID --output json | jq '.run_id')
databricks jobs get-run --run-id $RUN_ID

# Deploy and capture exit code
databricks bundle deploy -t prod
if [ $? -ne 0 ]; then
  echo "Deployment failed!"
  exit 1
fi
```

## 14. Best Practices

### Authentication
- Use **OIDC** for GitHub Actions — no stored secrets, automatically rotated
- Use **OAuth M2M** (service principal + client secret) when OIDC is not available
- Never commit credentials to Git — use environment variables or CI/CD secrets stores
- Use **`--profile`** to separate dev/staging/prod workspaces in local development

### Bundle Workflow
- Always run `databricks bundle validate -t <target>` before `deploy` — catch errors early
- Use `databricks bundle generate` to onboard existing workspace resources into bundle management
- Use `databricks bundle plan -t prod` to preview changes before deploying to production
- Use `--auto-approve` flag with `destroy` only in fully automated pipelines

### Asset Management
- Use `databricks fs` for DBFS and Unity Catalog Volume file operations
- Use `databricks workspace import/export` for migrating notebooks between workspaces
- Never expose secret values — `databricks secrets list-secrets` shows key names only
- Prefer `--string-value` or stdin piping over environment variables for `put-secret` (avoids shell history)

### CI/CD
- Pin the `databricks/setup-cli` action to a specific version for reproducibility
- Use the **Unified Authentication** environment variables (`DATABRICKS_HOST`, `DATABRICKS_CLIENT_ID`) — they are automatically detected by the CLI
- Structure workflows: **validate on PR**, **deploy staging on merge**, **deploy prod on approval**

## 1. Core Concepts: Two Trigger Philosophies

A Lakeflow Jobs **trigger** can be **time-based**, such as running a job at a scheduled time (for example, every day at 2 AM), or **event-based** (data-driven), such as running a job when new data arrives in cloud storage.

| Philosophy | Trigger Types | Drives Off Of |
|---|---|---|
| **Time-based** | Scheduled (simple interval / Quartz cron) | The clock |
| **Data-driven** | File arrival, Table update | The actual state of your data |

Historically, many data teams have relied on cron jobs to *approximate* when data is available — but that guesswork leads to wasted compute when data is late, and stale insights when data arrives early but the job hasn't fired yet. Data-driven triggers exist specifically to remove that guesswork by letting jobs run automatically as soon as specified files land or tables update.

> [!NOTE]
> A **Continuous** trigger is a third category — neither clock-based nor event-triggered, but an always-on loop that restarts as soon as the previous run finishes. It's covered briefly in [Section 6](#6-hybrid-patterns) but is out of scope for the core comparison since it doesn't "wait" for anything.

## 2. Decision Framework

Ask these questions, roughly in order:

1. **Is the upstream data arrival pattern predictable and regular?**
   → Predictable (e.g., a nightly batch load that always lands by 5 AM) favors **Scheduled**.
   → Irregular, bursty, or unknown timing favors **File arrival** or **Table update**.

2. **Do you control or have visibility into the upstream producer?**
   → If you own the upstream job/schedule, you can sometimes just chain jobs (`depends_on`, Run Job task) instead of either trigger type.
   → If the upstream is owned by another team, another system, or an external partner (classic **Data Mesh** scenario), a data-driven trigger decouples you from needing to know their schedule.

3. **Is the upstream source a file landing in cloud storage, or a Unity Catalog table changing?**
   → Files in S3/ADLS/GCS → **File arrival trigger**.
   → Delta/Iceberg tables, materialized views, streaming tables, UC views → **Table update trigger**.
   → Neither (pure time cadence, e.g., "always refresh at 6 AM regardless of data") → **Scheduled**.

4. **How latency-sensitive is the downstream consumer?**
   → Dashboards/alerts that need to reflect data "as soon as possible" benefit from data-driven triggers, which start a run as soon as data is ready instead of waiting for the next scheduled slot.
   → Reporting that's only consumed once a day at a fixed time gains little from data-driven triggering — a schedule is simpler and just as effective.

5. **What's your tolerance for wasted compute?**
   → A schedule set conservatively early to "be safe" wastes compute on empty/incomplete runs when data is late; set too late, it delays insights. Data-driven triggers run only when there's actually something to process.

6. **Do you need a hard ceiling on run frequency or batch debouncing?**
   → Both file arrival and table update triggers offer **minimum time between triggers** (cooldown) and **wait after last change** (debounce) — useful when data arrives in bursts and you don't want a run per file/commit.

## 3. Time-Based (Scheduled) Triggers — When They Win

### Best Fit Scenarios
- Upstream data has a **known, reliable landing time** (e.g., a vendor feed that always completes by 4 AM).
- The job's purpose is itself inherently time-bound (e.g., "compute end-of-day snapshot," "send the weekly digest every Monday").
- You need **predictable, auditable run times** for compliance or SLA reporting, independent of when data technically becomes available.
- The workload doesn't have a meaningful upstream "trigger event" at all (e.g., a periodic health-check job, a cache-warming job, retraining on a fixed cadence).
- You want the **simplest possible configuration** — no Unity Catalog volume/external location dependency, no enabling file events, no list of tables to maintain.

### Trade-offs
- **Pros**: Simple, predictable, easy to reason about and audit; no dependency on Unity Catalog storage configuration; works for any task type including those with no natural "data event."
- **Cons**: Inherently a guess about when data will be ready — too early wastes compute on incomplete data, too late delays downstream consumers; doesn't adapt if upstream timing shifts (e.g., DST, upstream job runs late).

### Key Behavior Notes
- **Simple** schedules let you pick an interval/unit only; the scheduler chooses the first-run time. **Advanced** schedules give full Quartz cron control, including time zone.
- Databricks enforces a **minimum interval of 10 seconds** between scheduled runs.
- Prefer **UTC** if you need a truly fixed absolute hourly cadence — DST-observing time zones can shift or skip hourly runs around DST transitions.
- The scheduler is **not built for low-latency** triggering; runs can be delayed by several minutes during network/cloud issues.

## 4. Data-Driven Triggers — When They Win

### File Arrival Trigger — Best Fit
- Data lands as **files** in a Unity Catalog external location or volume (S3, ADLS, GCS) on an **irregular** cadence.
- You're ingesting from an external system that drops files whenever it has new data, with no fixed schedule.
- A scheduled job's efficiency is compromised by irregular new-data arrivals — i.e., you'd otherwise have to poll frequently "just in case," wasting compute on empty runs.

### Table Update Trigger — Best Fit
- The upstream "event" is a **Unity Catalog table, view, materialized view, or streaming table changing** (insert, merge, delete) rather than a raw file landing.
- You want jobs to run **automatically as soon as specified tables are updated**, without needing to guess when an upstream pipeline finishes.
- You operate in a **decentralized / Data Mesh** environment where you don't own or have visibility into the upstream producer's internal schedule, and want to react to the data itself instead of coupling to someone else's cron expression.
- A downstream **dashboard refresh** or **alert** should reflect "freshest insights" rather than wait for a fixed time of day.

### Trade-offs
- **Pros**: Reacts to data as it actually becomes available — reduces wasted compute (no more guess-and-check polling) and reduces latency between data landing and downstream processing; decouples consumers from producers' schedules; built-in observability passes table metadata (commit timestamp/version) into downstream tasks via job parameters for consistent snapshots.
- **Cons**: Requires Unity Catalog; file arrival and table update both check on an interval (file arrival is best-effort, roughly every minute) so they are not instantaneous; more moving parts to configure (storage location or table list, file events setup for best performance); subject to documented limits (job counts per workspace, table counts per trigger, file counts without file events).

### Shared Rate-Control Options
Both file arrival and table update triggers expose the same two advanced controls to prevent run storms:
- **Minimum time between triggers in seconds** — a cooldown; caps the job to at most one run per interval.
- **Wait after last change in seconds** — a debounce; waits this long after the most recent change before starting a run, and resets on every new arrival/update, so a whole batch is processed in one run.
- If both are set, the minimum-time cooldown is honored first, then the debounce window applies on top of that.

## 5. Side-by-Side Comparison

| Dimension | Scheduled (Time-Based) | File Arrival (Data-Driven) | Table Update (Data-Driven) |
|---|---|---|---|
| **Fires on** | Clock time / cron expression | New files in a UC external location or volume | Updates (insert/merge/delete) to monitored UC tables/views |
| **Latency to react to data** | Fixed — only as fast as the next scheduled slot | Best-effort check every minute | Near-real-time, faster with file events enabled |
| **Setup complexity** | Low | Medium (UC volume/external location, optional file events) | Medium (table list, optional file events, view-dependency rules) |
| **Wasted-compute risk** | Higher if schedule is set conservatively early | Low — only fires when files actually arrive | Low — only fires when tables actually change |
| **Coupling to upstream** | None needed, but assumes you know upstream timing | None — reacts to storage state | None — reacts to table state, ideal for Data Mesh |
| **Predictability for audits/SLAs** | High — exact times known in advance | Lower — fires whenever data lands | Lower — fires whenever data lands |
| **Cost beyond compute** | None | Cloud provider file-listing costs only | Cloud provider listing/metadata-read costs only |
| **Rate control** | N/A (controlled by cron interval itself) | Min time between triggers, wait after last change | Min time between triggers, wait after last change |
| **Dynamic parameters exposed** | None inherent to the trigger | None inherent to the trigger | `updated_tables`, `commit_timestamp`, `version` per table |
| **Best for** | Predictable batch cadences, time-bound reports | Irregular file drops from external systems | Reactive pipelines chained to UC table changes |

## 6. Hybrid Patterns

You are not limited to exactly one trigger philosophy per job — common hybrid approaches:

### Pattern A — Scheduled Safety Net + Event-Driven Primary Path
Use a table update or file arrival trigger as the primary mechanism, and keep a much-less-frequent **scheduled** run (e.g., once daily) as a safety net in case an event is somehow missed (e.g., trigger evaluation failure). Pair this with **failure notifications** on the trigger itself so missed evaluations are visible.

### Pattern B — Chained Jobs Instead of Either Trigger
If you own both the upstream and downstream jobs, sometimes neither a schedule nor a data-driven trigger is the right tool — instead add a **Run Job task** or `depends_on` relationship so the downstream job runs immediately after the upstream job's task DAG completes, without polling anything.

### Pattern C — Continuous Trigger for Always-On Workloads
For workloads that need to process data essentially continuously (rather than in discrete batches), a **Continuous** trigger keeps a job always running, starting a new run as soon as the previous one finishes or fails. This sits outside the time-based/data-driven dichotomy — it's neither waiting for a clock nor for a specific event, just looping. Keep pipeline tasks that already run continuously (e.g., a continuous-mode Lakeflow Spark Declarative Pipeline) in **triggered** pipeline mode when wrapped by a job-level continuous trigger, to avoid double-continuous execution.

### Pattern D — Table Update Trigger Driving a Time-Sensitive Refresh
Instead of scheduling a dashboard refresh at a fixed time like 8 a.m. daily, use a table update trigger so the dashboard refreshes as soon as new data lands — this is explicitly called out as a canonical use case for replacing fixed-time refreshes with event-driven ones.

## 7. Migrating from Cron to Event-Driven

A common modernization path is replacing an existing `schedule`-based job with a `trigger.table_update` or `trigger.file_arrival` configuration once the upstream dependency is known and lives in Unity Catalog.

### Steps
1. **Identify the real dependency.** Determine which table(s) or storage path your job is actually waiting on when it runs on its current schedule.
2. **Confirm Unity Catalog coverage.** Table update triggers require the source to be a UC Delta/Iceberg managed table, UC external Delta table, materialized view, streaming table, or a UC view/metric view depending only on supported tables. File arrival triggers require a UC volume or external location.
3. **Enable file events** on the backing external location for best performance (recommended, not required) — this benefits both trigger evaluation speed and Auto Loader.
4. **Add the new trigger alongside the old schedule temporarily**, if you want to validate before cutting over (e.g., compare run timestamps for a few days), then remove the `schedule` block once confident.
5. **Set rate controls** (`min_time_between_triggers_seconds`, `wait_after_last_change_seconds`) to match the old schedule's intent — e.g., if the cron job ran once an hour at most, set a comparable cooldown to avoid a flood of runs if the upstream table updates more frequently than expected.
6. **Add failure notifications** for trigger evaluation failures, since a silently-failing event trigger has no equivalent to a missed cron tick showing up obviously in logs.

### Example: Before (Cron) → After (Table Update)

**Before:**
```yaml
resources:
  jobs:
    refresh_sales_dashboard:
      name: refresh_sales_dashboard
      schedule:
        quartz_cron_expression: '0 0 8 * * ?'   # daily at 8am, hoping data is ready
        timezone_id: UTC
      tasks:
        - task_key: refresh_dashboard
          dashboard_task:
            dashboard_id: 33333333-3333-3333-3333-333333333333
```

**After:**
```yaml
resources:
  jobs:
    refresh_sales_dashboard:
      name: refresh_sales_dashboard
      trigger:
        pause_status: UNPAUSED
        table_update:
          table_names:
            - main.sales.orders
          condition: ANY_UPDATED
          wait_after_last_change_seconds: 300   # let the full nightly batch land first
      tasks:
        - task_key: refresh_dashboard
          dashboard_task:
            dashboard_id: 33333333-3333-3333-3333-333333333333
```

## 8. REST API & DAB — Pattern Examples

### REST API: Job With Both a Schedule and a Safety-Net Comment (Pattern A)
> The Jobs API trigger object accepts exactly one of `periodic`, `file_arrival`, or `table_update`. To implement a "safety net," create two separate jobs (a primary event-driven job and a secondary scheduled reconciliation job) rather than combining both in a single trigger object.

```json
{
  "name": "primary_event_driven_job",
  "trigger": {
    "pause_status": "UNPAUSED",
    "table_update": {
      "table_names": ["main.sales.orders"],
      "condition": "ANY_UPDATED",
      "wait_after_last_change_seconds": 300
    }
  }
}
```
```json
{
  "name": "daily_reconciliation_job",
  "schedule": {
    "quartz_cron_expression": "0 0 23 * * ?",
    "timezone_id": "UTC"
  }
}
```

### DAB: Choosing the Trigger via a Bundle Variable
```yaml
variables:
  trigger_mode:
    description: "scheduled or event_driven"
    default: event_driven

resources:
  jobs:
    sales_etl:
      name: sales_etl
      # Use only one of the two blocks below per deployment target
      trigger:
        pause_status: UNPAUSED
        table_update:
          table_names:
            - main.raw.sales_feed
          condition: ANY_UPDATED
          min_time_between_triggers_seconds: 120
      tasks:
        - task_key: transform
          notebook_task:
            notebook_path: ../src/transform_sales.ipynb
```

## 9. Anti-Patterns to Avoid

- **Scheduling "just in case" at a very tight interval** (e.g., every 5 minutes) to approximate event-driven behavior. This wastes compute on empty runs and is exactly the guesswork data-driven triggers are designed to remove — use a file arrival or table update trigger instead.
- **Using a file arrival trigger to detect file overwrites.** Overwriting an existing file with the same name does **not** trigger a run; if your pipeline relies on in-place file updates, use a table update trigger or redesign the ingestion pattern to write new file names.
- **Pointing a file arrival trigger at a high-churn root path** (e.g., the root of a busy external location) instead of a scoped subdirectory or dedicated volume — this risks trigger evaluation timeouts under heavy unrelated file churn.
- **Wrapping an already-continuous Lakeflow Spark Declarative Pipeline in a job-level continuous trigger** — this layers two continuous execution loops; keep pipeline tasks orchestrated by jobs in triggered mode.
- **Ignoring rate-control settings on bursty sources.** Without `min_time_between_triggers_seconds` or `wait_after_last_change_seconds`, a table that receives many small commits in a short window can launch a run per commit instead of one consolidated run.
- **Skipping failure notifications on data-driven triggers.** A trigger that fails to evaluate fails silently from the job-run history's perspective — without notifications, you may not notice the job has simply stopped firing.

## 10. Decision Checklist

```
┌──────────────────────────────────────────────────────────────────────────┐
│      TIME-BASED vs. DATA-DRIVEN TRIGGER — QUICK DECISION CHECKLIST       │
├──────────────────────────────────────────────────────────────────────────┤
│ Upstream timing predictable & stable?         → YES → Scheduled          │
│ Upstream timing irregular / unknown?          → YES → File arrival or    │
│                                                         Table update      │
│ Source is raw files in cloud storage?         → YES → File arrival       │
│ Source is a UC table / view / MV / ST?        → YES → Table update       │
│ Job has no real "data event" (e.g., cache     │
│   warm, health check, fixed weekly digest)?   → YES → Scheduled          │
│ Don't own/see upstream producer's schedule    │
│   (Data Mesh, cross-team)?                    → YES → Table update       │
│ Need lowest possible latency to fresh data?   → YES → File arrival /     │
│                                                         Table update      │
│ Need exact, auditable run times for SLAs?     → YES → Scheduled          │
│ Want an always-on streaming-style workload?   → YES → Continuous         │
│ Want a safety net under an event trigger?     → YES → Add a separate,    │
│                                                  low-frequency Scheduled  │
│                                                  reconciliation job       │
└──────────────────────────────────────────────────────────────────────────┘
```

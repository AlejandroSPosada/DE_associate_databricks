## 1. Overview — Lakeflow Jobs UI Navigation

Access the Lakeflow Jobs UI by clicking the **Workflows** icon in the Databricks sidebar. The interface has two primary levels:

| Level | What You See |
|---|---|
| **Jobs list** | All jobs in the workspace — name, creator, trigger type, last 5 run outcomes |
| **Job detail** | Tabs: Configuration, Runs, Tasks (DAG) |
| **Run detail** | DAG view, Timeline view, Run Events, task-level drill-down |

The UI provides two complementary monitoring modes:
- **Reactive** — inspect a specific run after it starts or fails
- **Proactive** — use system tables and dashboards to track trends over time

### What the Job List Shows

## 3. Job Lifecycle States

Lifecycle states describe the **current phase** of a run. Visible in the API and reflected in the UI:

| State | Description |
|---|---|
| `PENDING` | Run triggered but cluster not yet provisioned or started |
| `QUEUED` | Run is waiting due to maximum concurrent run limit being reached |
| `RUNNING` | Job/task is actively executing |
| `TERMINATING` | Execution completed; cluster is shutting down |
| `TERMINATED` | Run has finished (check `result_state` for outcome) |
| `SKIPPED` | Run or task was not executed (dependency or condition not met) |
| `BLOCKED` | Run prevented from starting due to upstream dependencies or concurrency constraints |
| `WAITING_FOR_RETRY` | Task failed and is waiting before the next automatic retry attempt |
| `INTERNAL_ERROR` | An unexpected platform-level error occurred |

### Active vs. Terminal States
| Category | States |
|---|---|
| **Active (not finished)** | `PENDING`, `QUEUED`, `RUNNING`, `TERMINATING`, `WAITING_FOR_RETRY`, `BLOCKED` |
| **Terminal (finished)** | `TERMINATED`, `SKIPPED`, `INTERNAL_ERROR` |

> For automation and API polling: **stop polling when the state reaches a terminal value** (`TERMINATED`, `SKIPPED`, `INTERNAL_ERROR`).

## 4. Result States — Terminal Outcomes

When a run reaches `TERMINATED`, the `result_state` provides the final outcome:

| Result State | Meaning |
|---|---|
| `SUCCESS` | All tasks completed successfully |
| `FAILED` | At least one task failed and no retry resolved it |
| `CANCELED` | Run was manually stopped by a user or via API |
| `TIMEDOUT` | Run exceeded the configured `timeout_seconds` |
| `MAXIMUM_CONCURRENT_RUNS_REACHED` | New run was blocked because max concurrency was already reached |
| `EXCLUDED` | Run was excluded by a condition (e.g., continuous job suppression) |

### UI Color Mapping for Result States

| UI Color | Lifecycle + Result State |
|---|---|
| **Blue (animated)** | `RUNNING` — actively in progress |
| **Green** | `TERMINATED` + `SUCCESS` |
| **Red** | `TERMINATED` + `FAILED` |
| **Gray** | `TERMINATED` + `CANCELED` or `TIMEDOUT` |
| **Pink** | `SKIPPED` — task/run skipped due to conditions |
| **Yellow** | `WAITING_FOR_RETRY` — paused between retry attempts |

## 5. DAG View — Visual Task Dependency Monitoring

The **DAG view** is the primary visual interface for monitoring the real-time and historical status of a multi-task job.

### What the DAG Shows
- Each **task** is represented as a node (box/card)
- **Arrows** between nodes represent dependencies (upstream → downstream)
- Each task card shows: task name, task type icon, current status (color-coded), duration
- **Parallel tasks** are displayed side-by-side — tasks with no dependency between them can run concurrently
- **Bottleneck tasks** (on the critical path) are visually identifiable as those with the longest duration

### DAG Task Colors (Same as Matrix Colors)
| Color | State |
|---|---|
| Green | Succeeded |
| Red | Failed |
| Pink | Skipped |
| Yellow | Waiting/Retrying |
| Blue | Running |
| Gray | Pending / Canceled |

### Clicking a Task in the DAG
Opens the **task run detail panel** (right side) showing:
- Task type, cluster info
- Start time, end time, duration
- Status and attempt number
- Output, error message, logs
- Link to Spark UI for this run

### DAG vs. Matrix View — When to Use Each
| Use Case | Best View |
|---|---|
| Monitor a single run in real time | **DAG view** (in Run detail) |
| Identify recurring failures across runs | **Matrix view** (in Runs tab) |
| Compare task durations across multiple runs | **Matrix view** + system tables |
| Understand dependency structure | **DAG view** |
| Spot cascading failures in one run | **DAG view** |

## 6. Timeline View — Duration & Bottleneck Analysis

The **Timeline view** shows each task as a horizontal bar spanning its execution time, all rendered on a shared time axis within the same job run.

```
Task:         |─────────────────────────────────────────────|
              0min          10min        20min        30min

ingest        |═══════|
transform              |═════════════════|
validate                                  |════|
load                                           |════════════|
```

### What the Timeline Reveals
| Pattern | Interpretation |
|---|---|
| Wide bar | Long-running task — potential bottleneck |
| Gap before bar starts | Task was waiting for dependencies (correct) or for cluster |
| Tasks overlapping | Tasks running in parallel (no dependency between them) |
| Very short bar before long gap | Task completed quickly; downstream task has a long wait |

### When Timeline View is Most Useful
- Optimizing task ordering for parallel execution
- Identifying which task on the **critical path** limits overall job duration
- Seeing cluster startup time as a gap before the first task bar begins

## 7. Run Events Panel

The **Run Events** panel provides a chronological event log for the entire job run, reducing the need to parse raw logs.

### Types of Events
| Event | Description |
|---|---|
| `Run triggered` | When the run was initiated and by what trigger |
| `Cluster starting` | Cluster provisioning began (shows cluster ID) |
| `Cluster ready` | Cluster is available — tasks can start |
| `Task started` | A specific task began execution |
| `Task retrying` | A task failed and will retry after the interval |
| `Task succeeded` | A task completed successfully |
| `Task failed` | A task failed with error details |
| `Run completed` | Final run outcome |

### Value of Run Events
- **Cluster startup time**: The gap between "Cluster starting" and "Cluster ready" shows how much startup time is adding to job latency
- **Retry timeline**: Seeing "Task retrying" events shows how many retries occurred and how much time they consumed
- **Root cause**: "Task failed" event includes the error class (e.g., `SparkException`, `ValueError`) without opening the full log

## 8. Task-Level Inspection

From any run's detail page, clicking a task opens a dedicated side panel or full-page view:

### Task Run Detail Panel Contents
| Section | What's Included |
|---|---|
| **Status** | Lifecycle state + result state |
| **Duration** | Wall-clock time for this task attempt |
| **Attempt** | Attempt number (1 = first, 2 = first retry, etc.) |
| **Cluster** | Cluster ID and type used for this task |
| **Output** | Notebook cell outputs, `print()` statements, return values |
| **Error** | Full exception class and stack trace (for failures) |
| **Logs** | Links to: Driver stdout, Driver stderr, Log4j |
| **Spark UI** | Direct link to the Spark UI for this run's cluster session |
| **Metrics tab** | For streaming tasks: backlog size, processed records/sec |

### Diagnosing Common Failure Patterns

| Error Pattern | Where to Look |
|---|---|
| Python exception | Error section → full stack trace |
| Out of memory (OOM) | Logs → Driver stderr + Spark UI → Executor tab |
| Cluster startup failure | Run Events panel → "Cluster starting" event details |
| Timeout | Result state = `TIMEDOUT`; `timeout_seconds` limit was exceeded |
| Upstream dependency issue | DAG view → upstream task is red; this task is pink (skipped) |

## 9. Advanced Control Flow in the DAG

### If/Else Tasks (Branching)
Databricks supports **conditional branching** directly in the DAG:

```
         task_a
           │
       ┌───▼───┐
       │if/else│
       └───────┘
      True │    │ False
           ▼    ▼
      task_b  task_c
```

- The If/else task evaluates a condition at runtime
- Only one branch (True or False) executes
- The other branch's tasks are **skipped** (pink in DAG)
- Both branches merge back at a downstream task if configured

### For Each Tasks (Looping)
- Iterates over a list of inputs and runs a nested task for each item
- Each iteration is a sub-run visible within the parent run's DAG
- Useful for: per-table processing, per-file ingestion, parameter sweeps

### How Control Flow Appears in the DAG
| Feature | Visual |
|---|---|
| If/else | Diamond node in the DAG connecting to two task branches |
| For each | Task node with a loop indicator; click to see sub-runs |
| Disabled task | Grayed out in the DAG; downstream tasks evaluate their `Run if` condition |

## 11. Failure Rate Monitoring — UI vs. System Tables

### UI Capabilities (Real-Time, Reactive)
| Feature | Location |
|---|---|
| See last 5 run outcomes | Job list page — mini color dots |
| See all runs with filter by `Failed` | Runs tab → filter by status |
| See which task failed in the most recent run | DAG view → red task |
| See full error message for a failed task | Task run detail → Error section |

### UI Limitations
- **No built-in failure rate percentage** — the UI shows individual run outcomes but does not compute aggregate failure rates
- **60-day maximum history** — cannot see trends beyond 2 months
- **No cross-job comparison** — cannot compare failure rates across 10 different jobs in one view

### System Tables for Failure Rate Analysis
To calculate failure rates across jobs or over time, query `system.lakeflow.job_run_timeline`:

```sql
-- Failure rate per job, last 30 days
SELECT
  job_id,
  COUNT(*) AS total_runs,
  SUM(CASE WHEN state = 'FAILED' THEN 1 ELSE 0 END) AS failures,
  ROUND(SUM(CASE WHEN state = 'FAILED' THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 2) AS failure_pct
FROM system.lakeflow.job_run_timeline
WHERE start_time >= CURRENT_TIMESTAMP - INTERVAL 30 DAYS
GROUP BY job_id
ORDER BY failure_pct DESC;
```

## 12. Notifications & Webhooks

Configure alerts for job events from the job's **Notifications** settings tab:

### Email Notifications
```yaml
email_notifications:
  on_start:       [team@company.com]
  on_success:     [team@company.com]
  on_failure:     [oncall@company.com]
  no_alert_for_skipped_runs: true
```

### Webhook Notifications
Connect to Slack, PagerDuty, Microsoft Teams, or custom HTTP endpoints:
1. Create a **Notification destination** in workspace Settings (admin)
2. Reference it in the job via its destination ID
3. Events supported: `on_start`, `on_success`, `on_failure`, `on_duration_warning`

### Health Rules — SLA Alerting
Alert when a job exceeds its expected duration:

```yaml
health:
  rules:
    - metric: RUN_DURATION_SECONDS
      op: GREATER_THAN
      value: 3600    # Alert if job takes > 1 hour (SLA breach)
```

### Notification Events
| Event | When It Fires |
|---|---|
| `on_start` | The run begins (cluster ready, first task starting) |
| `on_success` | All tasks complete with `SUCCESS` result state |
| `on_failure` | Run ends with `FAILED` result state |
| `on_duration_warning` | Run duration exceeds the configured health rule threshold |

## 13. External Monitoring Integration

### Option 1: System Tables → Databricks SQL Dashboard
Query `system.lakeflow` from a SQL Warehouse, build a Lakeview dashboard:
- Recommended for Databricks-native monitoring
- Supports real-time refreshes, alerts, and drill-downs

### Option 2: Jobs REST API → External Tools
Poll the Jobs API for run status:
```bash
GET /api/2.1/jobs/runs/list?job_id=12345&limit=25
GET /api/2.1/jobs/runs/get?run_id=678901
```

Feed status data into Grafana, Datadog, Prometheus, or custom tooling.

### Option 3: Webhooks → External Alerting
Configure webhook notifications to push run events to:
- **Slack** — team alerting channels
- **PagerDuty** — on-call incident management
- **Microsoft Teams** — enterprise notification pipelines

## 14. Monitoring Best Practices

| Scenario | Recommended Action |
|---|---|
| Investigate a specific failure right now | DAG view → click red task → read Error section |
| Find which task causes most failures | Matrix view → look for column with most red |
| Track SLA compliance over weeks | `system.lakeflow.job_run_timeline` → duration trend query |
| Reduce alert fatigue from skipped runs | Set `no_alert_for_skipped_runs: true` in notifications |
| Monitor cluster startup time impact | Timeline view + Run Events — measure gap before first task |
| Detect cascading failures | DAG view — upstream red + downstream pink pattern |
| Compare performance across 20 jobs | Databricks SQL dashboard from system tables |
| Get notified of SLA breach in real time | Health rules → `RUN_DURATION_SECONDS > threshold` |
| Integrate with external incident tool | Webhooks → PagerDuty or Slack notification destination |

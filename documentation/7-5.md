## 1. Core Concepts Overview

### What is Data Lineage in Unity Catalog?
Data lineage shows **where data came from and where it goes**: which queries and files populate a table, which jobs and notebooks transform it, and which dashboards consume the results.

Unity Catalog captures lineage **automatically** for queries run on Databricks, down to the **column level**, and aggregates it across **all workspaces** attached to the same metastore.

### Why Use Lineage?

| Use Case | Description |
|---|---|
| **Impact Analysis** | Before changing or deleting a table/column, identify downstream tables, jobs, and dashboards that depend on it |
| **Root Cause Investigation** | When a downstream report shows unexpected results, trace upstream sources to find where data diverged |
| **Sensitive Data Tracking** | For compliance audits, see where regulated data originates, how it is transformed, and what consumes it |
| **Cross-Team Dependencies** | Discover which teams own upstream sources you rely on, or which teams consume your tables |

### What Can Appear as Lineage Nodes?
- Unity Catalog tables and views
- ML model versions
- External assets (Salesforce, MySQL, Tableau, Power BI, etc.)
- File paths

### Two Lineage Layers

| Layer | Type | Description |
|---|---|---|
| **Automatic** | Runtime Lineage | UC captures lineage from queries, jobs, notebooks, pipelines, dashboards automatically |
| **Manual / API** | External Lineage | You register external upstream/downstream assets and define relationships manually or via API |

> [!IMPORTANT]
> **External Lineage** extends the lineage graph beyond Databricks. Register upstream sources (e.g., MySQL, Salesforce) and downstream consumers (e.g., Tableau, Power BI) as external metadata objects in Unity Catalog -- they appear alongside UC tables in a single unified graph.

## 2. Requirements

### To Capture Lineage

| Requirement | Details |
|---|---|
| **Tables registered** | Tables must be registered in a Unity Catalog metastore |
| **External assets** | Non-UC assets must be added as *external metadata* objects in UC |
| **Query interface** | Queries must use Spark DataFrame API or Databricks SQL (notebooks, SQL editor) |

### Compute Requirements
- **Streaming lineage** between Delta tables: requires **Databricks Runtime 11.3 LTS or above**
- **Column lineage** in Lakeflow Spark Declarative Pipelines: requires **Databricks Runtime 13.3 LTS or above**

### Networking Requirements
> [!NOTE]
> You may need to update outbound firewall rules to allow connectivity to the **Amazon Kinesis endpoint** in the Databricks control plane (relevant for customer-managed VPCs or AWS PrivateLink environments).

### To View Lineage
- Must have at least **`BROWSE`** privilege on the parent catalog of the table or view
- The parent catalog must be accessible from the workspace (workspace-catalog binding)
- For notebooks, jobs, dashboards: requires the corresponding object-level permissions
- For Lakeflow Spark Declarative Pipelines: requires **`CAN VIEW`** permission on the pipeline

## 3. Viewing Lineage in Catalog Explorer

### Step-by-Step: View Table Lineage
```
1. In the Databricks workspace sidebar, click  Catalog
2. Search or browse for your table
3. Select the  Lineage  tab
   -> The lineage panel appears with related tables listed
4. Click  See Lineage Graph
   -> Interactive graph is displayed (default: 1 level shown)
5. Click the  +  icon on a node to reveal more connections
6. Click an edge in the graph to open the  Lineage Details  panel
   -> Shows source and target tables, notebooks, jobs, pipelines, queries
7. Use the filter buttons to show: Notebooks | Jobs | Pipelines | Queries
```

### View Job Lineage
1. Go to a table's **Lineage** tab
2. Select **Jobs** > **Downstream**
3. Job names appear under **Job Name** as consumers of the table

### View Dashboard Lineage
1. Go to a table's **Lineage** tab
2. Click **Dashboards**
3. Dashboard names appear under **Dashboard Name** as consumers

### Lineage Graph -- Key Actions

| Action | How |
|---|---|
| Expand node | Click `+` icon on the node |
| See connection details | Click an edge between nodes |
| Filter by asset type | Use filter buttons (Jobs, Notebooks, etc.) |
| View column-level lineage | Click a column name within the graph |

## 4. Column-Level Lineage

Unity Catalog tracks lineage down to the **individual column level**, showing which source columns contributed to each target column.

### View Column Lineage
In the Lineage Graph, click any **column name** within a table node -- links to related upstream/downstream columns are highlighted.

### How Column Lineage is Captured
```sql
-- Example: column lineage is captured for this query
CREATE OR REPLACE TABLE car_features
AS SELECT *, in1 + in2 AS premium_feature_set
FROM car_features_exterior
JOIN car_features_interior
USING (id, model);
```

**What UC records in `system.access.column_lineage`:**

| entity_type | source_table | source_column | target_table | target_column |
|---|---|---|---|---|
| `NOTEBOOK` | `car_features_interior` | `in1` | `car_features` | `premium_feature_set` |
| `NOTEBOOK` | `car_features_interior` | `in2` | `car_features` | `premium_feature_set` |

### Column Lineage -- What Is NOT Captured

| Scenario | Reason |
|---|---|
| Source/target referenced as path (`delta."s3://..."`) | Only table-name references are supported |
| User-defined functions (UDFs) | UDFs can obscure source-to-target column mapping |
| RDDs | Not captured at all |
| Global temp views | Not captured |

## 5. Get Lineage Using Genie Code (AI)

**Genie Code** is the Databricks AI assistant. It can answer lineage questions in natural language directly inside Catalog Explorer.

### How to Use
```
1. In the workspace sidebar, click  Catalog
2. Browse or search for the catalog, click its name
3. Click the  Genie Code  icon in the upper-right corner
4. At the prompt, type one of the following slash commands:
```

| Command | Description |
|---|---|
| `/getTableLineages` | View upstream and downstream dependencies for the selected table |
| `/getTableInsights` | Access metadata-driven insights such as user activity and query patterns |

### Example Natural Language Queries
```
"Show me downstream lineages of this table"
"Who queries this table most often?"
"What tables depend on revenue_fact?"
```

> [!TIP]
> Genie Code answers are generated from the same lineage data visible in the Lineage tab -- it simply lets you ask questions about it in plain English.

## 6. Lineage System Tables

Unity Catalog provides two **system tables** for programmatic access to lineage data. These are in the `system.access` schema.

> [!IMPORTANT]
> System tables must be **enabled** in your `system` catalog before use. See *Enable system tables* in the admin docs.

### 6.1 Table Lineage Table

**Path**: `system.access.table_lineage`

Records a row for **each read or write event** on a Unity Catalog table or path, including job runs, notebook runs, and dashboard queries.

#### Schema -- `system.access.table_lineage`

| Column | Type | Description |
|---|---|---|
| `account_id` | string | Databricks account ID |
| `metastore_id` | string | Unity Catalog metastore ID |
| `workspace_id` | string | Workspace ID |
| `entity_type` | string | `NOTEBOOK`, `JOB`, `PIPELINE`, `DASHBOARD_V3`, `DBSQL_QUERY`, or NULL |
| `entity_id` | string | ID of the entity (notebook, job, etc.) |
| `entity_run_id` | string | Unique run ID of the entity |
| `source_table_full_name` | string | Three-part name: `catalog.schema.table` |
| `source_table_catalog` | string | Catalog of the source table |
| `source_table_schema` | string | Schema of the source table |
| `source_table_name` | string | Name of the source table |
| `source_path` | string | Cloud storage path (e.g., `s3://mybucket/table1`) |
| `source_type` | string | `TABLE`, `PATH`, `VIEW`, `MATERIALIZED_VIEW`, `METRIC_VIEW`, `STREAMING_TABLE` |
| `target_table_full_name` | string | Three-part name: `catalog.schema.table` |
| `target_table_catalog` | string | Catalog of the target table |
| `target_table_schema` | string | Schema of the target table |
| `target_table_name` | string | Name of the target table |
| `target_path` | string | Cloud storage path of the target table |
| `target_type` | string | Same values as `source_type` |
| `created_by` | string | User, service principal, group, or `"System-User"` |
| `event_time` | timestamp | When lineage was generated (UTC, e.g. `2025-04-20T19:47:21.194+00:00`) |
| `event_date` | date | Date of the event -- **partition column** |
| `record_id` | string | Unique auto-generated record ID |
| `event_id` | string | Unique event ID (multiple rows can share the same event) |
| `statement_id` | string | FK to query history system table (SQL warehouse queries only) |
| `entity_metadata` | struct | Structured metadata about the entity (see below) |
| `direct_access` | boolean | `true` = source directly referenced; `false` = intermediate dependency via view expansion |

---

### 6.2 Column Lineage Table

**Path**: `system.access.column_lineage`

Same schema as table lineage **plus two additional columns**:

| Column | Type | Description |
|---|---|---|
| `source_column_name` | string | Name of the source column |
| `target_column_name` | string | Name of the target column |

> [!NOTE]
> Column lineage only records events **with a source**. If you insert using explicit values (e.g., `INSERT INTO t VALUES (1,2,3)`), no column lineage is captured. Read-only operations ARE captured.

---

### 6.3 Entity Metadata Struct (`entity_metadata`)

```json
{
  "job_info": {
    "job_id": "string",
    "job_run_id": "string"
  },
  "dashboard_id": "string",
  "legacy_dashboard_id": "string",
  "notebook_id": "string",
  "sql_query_id": "string",
  "dlt_pipeline_info": {
    "dlt_pipeline_id": "string",
    "dlt_update_id": "string"
  },
  "genie_space_id": "string",
  "alert_id": "string"
}
```

> Multiple values can be populated depending on the event type. For example, a **job running a notebook task** populates both `job_info` and `notebook_id`.
> If all values are `null`, no Databricks entity was involved (e.g., JDBC query or clicking Sample Data in the UI).

**Supported entity sources:**
- Lakeflow Spark Declarative Pipelines
- Notebooks
- Jobs
- Databricks SQL queries
- Dashboards
- Genie Spaces
- Alerts

### 6.4 Reading Lineage System Tables -- Determining Read vs. Write

| Event Type | source_type | target_type |
|---|---|---|
| **Read-only** | NOT NULL | NULL |
| **Write-only** | NULL | NOT NULL |
| **Read and write** | NOT NULL | NOT NULL |

### 6.5 Example SQL Queries

#### Find all downstream tables of a source table
```sql
SELECT
  target_table_full_name,
  entity_type,
  entity_id,
  created_by,
  event_time
FROM system.access.table_lineage
WHERE source_table_full_name = 'main.finance.revenue_raw'
  AND event_date >= CURRENT_DATE - INTERVAL 90 DAYS
ORDER BY event_time DESC;
```

#### Find all column dependencies for a target column
```sql
SELECT
  source_table_full_name,
  source_column_name,
  target_table_full_name,
  target_column_name,
  created_by,
  event_time
FROM system.access.column_lineage
WHERE target_table_full_name = 'main.finance.revenue_fact'
  AND target_column_name = 'net_revenue'
ORDER BY event_time DESC;
```

#### Query lineage for an external table referenced by path
```sql
-- Use source_path / target_path instead of table name for path-referenced tables
SELECT *
FROM system.access.table_lineage
WHERE
  source_path = 's3://mybucket/table1' OR
  target_path = 's3://mybucket/table1';
```

#### Python helper -- get lineage by table name (resolves path automatically)
```python
def getLineageForTable(table_name):
    table_path = spark.sql(f"describe detail {table_name}").select("location").head()[0]
    return spark.sql(f"""
        SELECT *
        FROM system.access.table_lineage
        WHERE source_path = '{table_path}'
           OR target_path = '{table_path}'
           OR source_table_full_name = '{table_name}'
           OR target_table_full_name = '{table_name}'
    """)

getLineageForTable("main.finance.revenue_fact").display()
```

## 7. External Lineage

Unity Catalog automatically captures lineage for queries that run **on** Databricks. But many pipelines have:
- **First-mile ETL** from external sources (e.g., Salesforce, MySQL, PostgreSQL)
- **Last-mile BI** consumption in external tools (e.g., Tableau, Power BI)

**External Lineage** lets you register these external assets and define relationships, so they appear in the same unified lineage graph alongside UC tables.

### 7.1 Two Ways to Add External Lineage

| Method | Description |
|---|---|
| **Manual (UI / API)** | Use Catalog Explorer, the External Metadata API, the External Lineage API, or the Databricks SDK for Python |
| **Automatic (Lakeflow Connect)** | Managed ingestion pipelines automatically write UC external metadata objects and create lineage relationships (including 1:1 column-level mapping) from source systems to destination Delta tables |

### 7.2 Required Privileges for External Lineage

| Operation | Required Privilege |
|---|---|
| Create an external metadata object | `CREATE EXTERNAL METADATA` on the metastore |
| Define lineage relationships (external <-> UC object) | `MODIFY` on the external metadata object |
| Create a downstream relationship to a UC table | `SELECT` (read) on the target table |
| Create an upstream relationship to a UC table | `MODIFY` (write) on the source table |

---

### 7.3 Step-by-Step: Add External Lineage (UI)

#### Step 1 -- Create an External Metadata Object
```
1. In the workspace sidebar, click  Catalog
2. Click  External data  ->  External Metadata  tab
3. Click  Create external metadata
4. Fill in required fields:
   - Name: human-readable (must follow UC naming rules)
   - System type: pick from list (Salesforce, MySQL, Tableau, etc.) or "Custom"
   - Entity type: e.g., "table", "dashboard", "report"
5. Optional fields:
   - URL: click-through URL for the external asset
   - Description
6. Advanced fields:
   - Columns: for column-level lineage mapping (UI or comma-delimited text)
   - Properties: JSON key-value pairs for extra metadata
7. Click  Create
```

#### Step 2 -- Create a Lineage Relationship
```
1. From the external metadata object page, click  Create lineage relationship
2. Select direction: Upstream or Downstream
3. Select Object type:
   - Table    -> use search dialog
   - Model    -> use search dialog + select version
   - Path     -> enter path (volumes, external locations)
   - External metadata -> select from dropdown
4. (Optional) Advanced:
   - Column mappings: map source <-> target columns
   - JSON metadata: query text, workflow annotations, etc.
5. Click  Create
```

The external asset now appears in the **Lineage graph** of the related UC table, model, or path.

### 7.4 Programmatic External Lineage (REST API)

#### Create External Metadata Object
```http
POST /api/2.0/external-metadata
Authorization: Bearer <token>
Content-Type: application/json
```
```json
{
  "name": "salesforce_opportunities",
  "system_type": "Salesforce",
  "entity_type": "object",
  "url": "https://mycompany.salesforce.com/Opportunity",
  "description": "Salesforce Opportunities object -- source for revenue pipeline",
  "columns": ["id", "amount", "stage", "close_date", "account_id"],
  "properties": {
    "owner_team": "revenue-ops"
  }
}
```

#### Create Lineage Relationship
```http
POST /api/2.0/external-lineage
Authorization: Bearer <token>
Content-Type: application/json
```
```json
{
  "source_external_metadata_name": "salesforce_opportunities",
  "target_table_full_name": "main.finance.revenue_raw",
  "column_mappings": [
    {"source_column": "amount",     "target_column": "gross_revenue"},
    {"source_column": "close_date", "target_column": "transaction_date"},
    {"source_column": "account_id", "target_column": "account_key"}
  ],
  "properties": {
    "ingestion_tool": "Lakeflow Connect",
    "schedule": "hourly"
  }
}
```

### 7.5 Python SDK -- External Lineage

```python
from databricks.sdk import WorkspaceClient

w = WorkspaceClient(
    host="https://<workspace-url>",
    token="<personal-access-token>"
)

# --- Create external metadata object ---
external_obj = w.external_metadata.create(
    name="tableau_revenue_dashboard",
    system_type="Tableau",
    entity_type="dashboard",
    url="https://tableau.mycompany.com/views/RevenueDashboard",
    description="Revenue dashboard consuming finance data",
    columns=["net_revenue", "region", "quarter"]
)
print(f"Created external metadata: {external_obj.name}")

# --- Create downstream lineage relationship ---
relationship = w.external_lineage.create(
    source_table_full_name="main.finance.revenue_fact",
    target_external_metadata_name="tableau_revenue_dashboard",
    column_mappings=[
        {"source_column": "net_revenue", "target_column": "net_revenue"},
        {"source_column": "region",      "target_column": "region"},
        {"source_column": "quarter",     "target_column": "quarter"}
    ]
)
print(f"Created lineage relationship: {relationship.id}")
```

### 7.6 Lakeflow Connect -- Automatic Source Lineage

When using **Lakeflow Connect** managed ingestion pipelines:
- Source lineage is recorded **automatically** (no manual configuration needed)
- Creates UC external metadata objects **and** lineage relationships
- Captures **1:1 column-level mapping** from source system columns to Delta table columns
- Supported sources: SaaS apps (Salesforce, etc.) and databases (MySQL, PostgreSQL, etc.)

---

### 7.7 External Lineage Modeling Patterns

| Pattern | How |
|---|---|
| **Connect two UC tables** | Create an external metadata object between them; set one table upstream and the other downstream |
| **Multi-level lineage** | Create multiple external metadata objects and chain them with relationships |
| **Column-level external lineage** | Add column names to the external metadata object, then map source/target columns in the relationship |

---

### 7.8 External Lineage -- Limits

> [!WARNING]
> External lineage is **NOT** recorded in the lineage system tables (`system.access.table_lineage` or `system.access.column_lineage`). It is only accessible via Catalog Explorer or the External Lineage API.

| Limit | Value |
|---|---|
| Max external metadata objects per metastore | **10,000** |
| Max external lineage relationships per metastore | **100,000** |

## 8. Permissions Model

Lineage graphs share the same **Unity Catalog permission model** as all other UC objects.

### Access Rules

| Privilege | What it grants |
|---|---|
| `BROWSE` on a catalog | Can view the lineage graph for any table in that catalog |
| `SELECT` on a table | Can see that table as a node in lineage |
| No privilege on a table | Table appears as a **masked** node -- cannot expand it or see downstream tables from it |

### Example -- Granting Lineage Access

```sql
-- Grant userA read access to a specific table (sees that table only)
GRANT USE SCHEMA ON lineage_data.lineagedemo TO `userA@company.com`;
GRANT SELECT ON lineage_data.lineagedemo.menu TO `userA@company.com`;

-- Grant userB BROWSE on the catalog (sees the full lineage graph)
GRANT BROWSE ON lineage_data TO `userB@company.com`;
```

> [!NOTE]
> Lineage is **aggregated across all workspaces** attached to the same metastore. Lineage captured in one workspace is visible in any other workspace sharing that metastore -- but **detailed information about workspace-level objects** (notebooks, dashboards) in *other* workspaces is masked.

### Permissions for Workspace Objects

| Object | Required Permission |
|---|---|
| Notebooks | Workspace notebook access control |
| Jobs | Job-level access control |
| Dashboards | Dashboard-level access control |
| Lakeflow Pipelines | `CAN VIEW` on the pipeline |

## 9. Retention Policy

| Where | Retention |
|---|---|
| **Catalog Explorer** | Indefinitely -- all lineage captured **after September 1, 2024** is available |
| **Lineage System Tables** | Rolling **1-year window** -- events older than 1 year removed on a rolling basis |
| **Lineage API** | Indefinitely -- all lineage captured after September 1, 2024 |

### Time-Range Options in Catalog Explorer

| Metastore Created | Dropdown Options |
|---|---|
| After September 1, 2024 | **All time** (full history from creation) |
| Before September 1, 2024 | **All available** (starts from Sep 1, 2024) + other ranges |
| Default selection | **1 year** |

> [!CAUTION]
> Lineage data captured **before September 1, 2024** is not available anywhere. The 1-year window for system tables is the supported governance contract -- do not rely on data older than 1 year in system tables.

## 10. Limitations

| Limitation | Details |
|---|---|
| **Pre-Sep 2024 data** | Lineage data before September 1, 2024 is not available |
| **Renamed objects** | Lineage is NOT preserved when catalogs, schemas, tables, views, or columns are renamed |
| **`runs submit` / `spark submit` jobs** | These do not appear in lineage views (table/column lineage IS still captured, but the job run link is not) |
| **Spark SQL dataset checkpointing** | Lineage is not captured |
| **Lakeflow PRIVATE tables** | Column lineage from Lakeflow Spark Declarative Pipelines is incomplete for PRIVATE tables |
| **RDDs** | Resilient Distributed Datasets are not captured |
| **Global temp views** | Not captured in lineage |
| **Transactions (rollback)** | Lineage events persist even if the transaction is rolled back |
| **`system.information_schema`** | Tables under this schema are not captured |
| **Path-referenced external tables** | Column lineage requires both source AND target referenced by table name, not cloud path |
| **UDFs** | User-defined functions can obscure column-to-column mapping |
| **External lineage in system tables** | External lineage relationships do NOT appear in `system.access.table_lineage` or `system.access.column_lineage` |

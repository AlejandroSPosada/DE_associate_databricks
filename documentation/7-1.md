## 1. Core Distinction — Managed vs. External

The fundamental difference between managed and external tables is **who controls the data lifecycle and storage location**.

| Aspect | Managed Table | External Table |
|---|---|---|
| **Storage location** | Controlled by Unity Catalog (automatically assigned) | Controlled by you (specified with `LOCATION`) |
| **Data lifecycle** | UC manages — DROP removes data files | You manage — DROP only removes metadata |
| **VACUUM eligibility** | Fully managed by UC / Predictive Optimization | You are responsible |
| **Optimization** | Automatic (Predictive Optimization eligible) | Manual |
| **Recommendation** | Default — use for most workloads | Use only for specific interoperability needs |

## 2. CREATE Behavior

### Creating a Managed Table
```sql
-- No LOCATION clause — UC assigns storage automatically
CREATE TABLE catalog.schema.orders (
  order_id    BIGINT,
  customer_id BIGINT,
  order_date  DATE,
  amount      DECIMAL(10,2)
) USING DELTA;

-- CTAS (Create Table As Select) — managed by default
CREATE TABLE catalog.schema.orders_2024
AS SELECT * FROM catalog.schema.orders
WHERE YEAR(order_date) = 2024;

-- Storage location: automatically within the catalog's managed storage location
-- e.g., abfss://container@storage.dfs.core.windows.net/unity-catalog/catalog/schema/orders/
```

### Creating an External Table
```sql
-- LOCATION clause required — must point to a registered External Location
CREATE TABLE catalog.schema.raw_events
USING DELTA
LOCATION 'abfss://raw@myaccount.dfs.core.windows.net/events/';

-- External table pointing to Parquet files (non-Delta)
CREATE TABLE catalog.schema.legacy_data
USING PARQUET
LOCATION 'abfss://landing@myaccount.dfs.core.windows.net/legacy/';
```

### External Location Requirement
Before creating an external table in Unity Catalog, the storage path must be registered as an **External Location**:
```sql
-- Admin task: register external location
CREATE EXTERNAL LOCATION my_raw_location
URL 'abfss://raw@myaccount.dfs.core.windows.net/'
WITH (STORAGE CREDENTIAL my_credential);
```

## 3. DROP Behavior — The Critical Difference

This is the most exam-tested distinction between managed and external tables.

### DROP TABLE — Managed Table
```sql
DROP TABLE catalog.schema.orders;
-- Result:
-- ✓ Metadata removed from Unity Catalog
-- ✓ Data files are DELETED from cloud storage (after 8-day retention window)
-- ✓ Delta log and all Parquet files are removed
-- ✓ Table can be recovered with UNDROP within 7 days
```

### DROP TABLE — External Table
```sql
DROP TABLE catalog.schema.raw_events;
-- Result:
-- ✓ Metadata removed from Unity Catalog (table no longer in catalog)
-- ✗ Data files REMAIN in cloud storage (unaffected)
-- ✗ Files must be manually deleted from cloud storage if no longer needed
-- ✗ UNDROP is NOT supported for external tables
```

### Key Exam Points
- **Managed DROP**: deletes both metadata AND data → data loss risk
- **External DROP**: deletes ONLY metadata → data is safe but orphaned files accumulate
- External tables cannot be recovered with `UNDROP`
- Managed table data has an 8-day retention before permanent deletion

## 4. UNDROP — Recovering Managed Tables

Unity Catalog supports recovering accidentally dropped **managed tables** within a 7-day window.

### Syntax
```sql
-- Recover a dropped managed table
UNDROP TABLE catalog.schema.orders;

-- Verify the table was recovered
SELECT COUNT(*) FROM catalog.schema.orders;
```

### UNDROP Constraints
| Constraint | Detail |
|---|---|
| **Time window** | 7 days from the DROP command |
| **Table type** | Managed tables only — external tables cannot be UNDROPped |
| **Condition** | VACUUM must not have run after the DROP (VACUUM permanently deletes files) |
| **Permission** | Requires ownership or `CREATE TABLE` privilege on the schema |

### What Happens to Data After UNDROP
- The Delta log entry for the DROP is rolled back
- Data files (Parquet) are still in the managed storage location
- The table appears in the catalog again with full history

## 5. Modifying Tables — ALTER TABLE

### Renaming a Table
```sql
-- Works for both managed and external tables
ALTER TABLE catalog.schema.old_name
RENAME TO catalog.schema.new_name;
```

### Adding Columns
```sql
ALTER TABLE catalog.schema.orders
ADD COLUMNS (
  discount DECIMAL(5,2),
  promo_code STRING
);
```

### Changing Column Data Type
```sql
-- Supported for compatible type widening in Delta Lake
ALTER TABLE catalog.schema.orders
ALTER COLUMN amount TYPE DOUBLE;
```

### Changing Storage Location (External Tables Only)
```sql
-- Change where an external table's data files are read from
ALTER TABLE catalog.schema.raw_events
SET LOCATION 'abfss://raw@newaccount.dfs.core.windows.net/events/';
-- Note: existing data at the OLD location is not moved — only the pointer changes
```

### Setting Table Properties
```sql
ALTER TABLE catalog.schema.orders
SET TBLPROPERTIES (
  'delta.enableChangeDataFeed' = 'true',
  'owner' = 'data-engineering-team'
);
```

### Adding/Changing Comments
```sql
ALTER TABLE catalog.schema.orders
SET TBLPROPERTIES ('comment' = 'Processed orders for the APAC region');

-- Or using COMMENT ON
COMMENT ON TABLE catalog.schema.orders IS 'Processed orders for the APAC region';
```

## 6. Storage Location Hierarchy

Managed table storage locations are assigned based on a hierarchy:

```
Unity Catalog Metastore
    └── Catalog (has its own managed storage path)
        └── Schema (can have its own managed storage path — overrides catalog)
            └── Table (uses schema path, or catalog path if schema has no path)
```

### Setting Managed Storage at Different Levels
```sql
-- Set managed storage at catalog level (recommended for data isolation)
ALTER CATALOG my_catalog
SET MANAGED LOCATION 'abfss://managed@myaccount.dfs.core.windows.net/catalog1/';

-- Set managed storage at schema level (overrides catalog-level path for this schema)
ALTER SCHEMA my_catalog.my_schema
SET MANAGED LOCATION 'abfss://managed@myaccount.dfs.core.windows.net/schema_specific/';
```

> **Important**: These commands only affect **NEW** managed tables and volumes created after the change. Existing tables are NOT moved.

### Storage Path Resolution for New Managed Tables
| Configuration | Where New Tables Are Stored |
|---|---|
| Schema has managed location | Schema managed location |
| Schema has no location, catalog does | Catalog managed location |
| Neither schema nor catalog has location | Metastore default managed location |

## 7. External Locations — Required for External Tables

Before creating an external table, an admin must register an **External Location** in Unity Catalog:

### External Location Components
| Component | Description |
|---|---|
| **External Location** | A registered cloud storage path in Unity Catalog |
| **Storage Credential** | Cloud credentials (service principal, IAM role) to access the storage |
| **URL** | The cloud storage path prefix |

### Creating an External Location
```sql
-- Create a storage credential (admin)
CREATE STORAGE CREDENTIAL my_credential
WITH AZURE_MANAGED_IDENTITY (
  CONNECTOR = '/subscriptions/.../connectors/my_connector'
);

-- Create external location (admin)
CREATE EXTERNAL LOCATION raw_data_location
URL 'abfss://raw@myaccount.dfs.core.windows.net/'
WITH (STORAGE CREDENTIAL my_credential);

-- Grant access to users
GRANT READ FILES, WRITE FILES
ON EXTERNAL LOCATION raw_data_location
TO `data-engineers`;
```

## 8. Converting External → Managed (ALTER TABLE SET MANAGED)

### Recommended Method (2026): `ALTER TABLE SET MANAGED`
```sql
ALTER TABLE catalog.schema.raw_events SET MANAGED;
```

### What `SET MANAGED` Does
1. Moves data files to the Unity Catalog managed storage location
2. Updates the Delta log to reflect the new storage path
3. Changes the table type from `EXTERNAL` to `MANAGED`
4. Provides a brief write window for migration

### Benefits vs. CTAS / DEEP CLONE
| Feature | SET MANAGED | CTAS | DEEP CLONE |
|---|---|---|---|
| Preserves history | Yes | No | Yes |
| Preserves permissions | Yes | No | Partial |
| Minimal downtime | Yes (minutes) | Table recreation | Long for large tables |
| Single command | Yes | Multi-step | Multi-step |
| Recommended for migration | **Primary** | Discouraged | Fallback |

### Post-Conversion Cleanup
After `SET MANAGED`, the original files in the external storage location still exist:
```sql
-- Step 1: Convert
ALTER TABLE catalog.schema.raw_events SET MANAGED;

-- Step 2: Verify the table is now managed
DESCRIBE TABLE EXTENDED catalog.schema.raw_events;
-- Look for: Type = MANAGED

-- Step 3: Manually delete original files from external location
-- (via cloud storage console or dbutils.fs.rm)
-- to avoid double storage costs
```

### Reverting with UNSET MANAGED
If the conversion needs to be reversed (within the retention window, before VACUUM runs):
```sql
ALTER TABLE catalog.schema.raw_events UNSET MANAGED;
-- Reverts to external table pointing to the original location
-- Only possible if VACUUM has not deleted the original files
```

## 9. Converting Managed → External (UNSET MANAGED)

`UNSET MANAGED` effectively marks a table as external again, pointing it back to the original storage location. This is primarily used to:
- Revert a recent `SET MANAGED` conversion
- Allow another system to take ownership of data files

```sql
ALTER TABLE catalog.schema.raw_events UNSET MANAGED;
```

> Note: The original external location must still exist with valid data.

## 10. Alternative Conversion Methods — CTAS and DEEP CLONE

### CTAS (Create Table As Select)
```sql
-- Create a new managed table from an external table
CREATE TABLE catalog.schema.orders_managed
AS SELECT * FROM catalog.schema.orders_external;

-- Limitations:
-- Does NOT preserve Delta history
-- Does NOT preserve table permissions
-- Table name changes (unless you DROP old and RENAME)
-- Must manually set up clustering, properties, etc.
```

### DEEP CLONE
```sql
-- Creates a full copy including Delta history
CREATE TABLE catalog.schema.orders_managed
DEEP CLONE catalog.schema.orders_external;

-- Best used for:
-- Cross-metastore copies
-- Creating sandbox copies
-- One-time data migration when SET MANAGED is not available

-- Does NOT automatically remove old location — manual cleanup required
```

### Method Comparison
| Scenario | Recommended Method |
|---|---|
| Convert external → managed (in-place, keep history) | `ALTER TABLE SET MANAGED` |
| Copy table to different catalog/schema | `DEEP CLONE` |
| Create fresh managed copy without history | `CTAS` |
| Create managed table from scratch | `CREATE TABLE ... USING DELTA` |

## 11. Metadata Sync — MSCK REPAIR TABLE

When an external table's files are modified by a **non-Databricks client** (e.g., direct S3 upload, external ETL tool), the Unity Catalog metadata may become stale.

### Sync Metadata
```sql
-- Bring UC metadata in sync with actual files
MSCK REPAIR TABLE catalog.schema.external_table SYNC METADATA;
```

### When to Use MSCK REPAIR
- Files were added/removed from the external location by a tool outside Databricks
- Partition metadata is out of date for Hive-partitioned external tables
- UC catalog shows different files than what exists in cloud storage

### SYNC Command (Hive Metastore to UC Migration)
```sql
-- Upgrade a table from Hive Metastore to Unity Catalog
SYNC TABLE hive_metastore.schema.table
TO catalog.schema.table;
```

## 12. When to Use Each Table Type

### Use Managed Tables When:
- Building new tables for production workloads
- You want automatic lifecycle management (DROP cleans up files)
- You want Predictive Optimization (auto OPTIMIZE/VACUUM/ANALYZE)
- You want UNDROP capability
- Simplest setup — no External Location registration needed

### Use External Tables When:
- Registering **existing data** that already lives in cloud storage
- Data must be **shared with non-Databricks tools** (e.g., EMR, Athena) that read directly from cloud storage
- Compliance requirements dictate that data must remain at a specific storage path
- Data files are **owned by another system** and Databricks should not manage their lifecycle
- You need to prevent accidental data deletion (DROP TABLE leaves files intact)

## 13. Comparison Summary Table

| Feature | Managed Table | External Table |
|---|---|---|
| **CREATE syntax** | No `LOCATION` clause | `LOCATION 'path'` required |
| **Storage control** | Unity Catalog assigns path | User specifies path |
| **DROP effect on data** | Data files deleted | Data files preserved |
| **DROP effect on metadata** | Removed | Removed |
| **UNDROP support** | Yes (within 7 days) | No |
| **External Location required** | No | Yes |
| **Predictive Optimization** | Yes | No |
| **Automatic VACUUM** | Yes (via PO) | No |
| **MSCK REPAIR** | Not needed | Yes (if files changed externally) |
| **ALTER SET LOCATION** | Not applicable | Yes — changes file pointer |
| **Convert to other type** | UNSET MANAGED → External | SET MANAGED → Managed |
| **File ownership** | Unity Catalog | You |
| **Recommended default** | Yes | Only when necessary |

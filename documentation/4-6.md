## 1. Core Concepts Overview

### What Is Serverless Compute for Jobs?
Serverless compute for workflows lets you run a Lakeflow Job **without configuring and deploying infrastructure**. Databricks efficiently manages compute resources — including optimizing and scaling — for your workloads, with **autoscaling and Photon automatically enabled**.

Serverless compute for workflows continuously optimizes infrastructure (instance types, memory, processing engines) to match the specific processing requirements of each workload, and Databricks **automatically upgrades the Databricks Runtime version** used under the hood, balancing new enhancements against job stability.

Because **cluster creation permission is not required**, all workspace users can use serverless compute to run their workflows — this removes a common access-control bottleneck associated with classic jobs/all-purpose compute.

### How It's Created and Run
You can create and run jobs that use serverless compute via:
- The Lakeflow Jobs **UI**
- The **Jobs REST API**
- **Databricks Asset Bundles** (Declarative Automation Bundles)
- The **Databricks SDK for Python**

---

## 2. Requirements

- The Databricks workspace must have **Unity Catalog enabled**.
- Because serverless compute for workflows uses **standard access mode**, your workloads must support that access mode.
- The workspace must be in a **supported region** for serverless compute.

## 3. Recommended Compute by Task Type

| Task Type | Recommended Compute | Supported Compute |
|---|---|---|
| Notebooks | Serverless jobs | Serverless jobs, classic jobs, classic all-purpose |
| Python script | Serverless jobs | Serverless jobs, classic jobs, classic all-purpose |
| Python wheel | Serverless jobs | Serverless jobs, classic jobs, classic all-purpose |
| SQL | Serverless SQL warehouse | Serverless SQL warehouse, pro SQL warehouse |
| Lakeflow Spark Declarative Pipelines | Serverless pipeline | Serverless pipeline, classic pipeline |
| dbt | Serverless SQL warehouse | Serverless SQL warehouse, pro SQL warehouse |
| dbt CLI commands | Serverless jobs | Serverless jobs, classic jobs, classic all-purpose |
| JAR | Classic jobs | Classic jobs, classic all-purpose |
| Spark Submit | Classic jobs | Classic jobs (serverless not supported) |

> [!NOTE]
> Serverless compute is directly supported for **notebook, Python script, dbt, Python wheel, and JAR** task types. By default, serverless compute is selected automatically when you create a job with one of these task types. Using serverless compute for **JAR tasks is in Public Preview**, even though the table above lists classic jobs as the JAR recommendation for stability.

### Choosing Compute Per Task
Databricks recommends using serverless compute for **all** job tasks where supported. You can still mix compute types across tasks in the same job — this may be required when a task type isn't supported by serverless (e.g., Spark Submit).

| Compute Type | Where It's Configured |
|---|---|
| Classic jobs compute | Directly in the Lakeflow Jobs UI/job definition, same settings as all-purpose compute |
| Serverless compute for jobs | Default for supported task types; Databricks manages all compute settings |
| SQL warehouses (serverless/pro) | Configured separately by admins/users with cluster creation privileges; tasks point at an existing warehouse |
| Lakeflow Spark Declarative Pipelines compute | Configured during pipeline setup; Databricks manages compute for serverless pipelines |
| All-purpose compute | Optional for tasks, but **not recommended for production jobs** |

## 4. Creating and Switching Jobs to Serverless Compute

### Create a New Job on Serverless
1. Add a supported task type (notebook, Python script, Python wheel, dbt, or JAR) — serverless compute is selected by default.
2. No cluster configuration step is required.

> [!NOTE]
> Because serverless compute ensures sufficient resources are provisioned, you might see **increased startup times** for jobs that require large amounts of memory or include many tasks.

### Switch an Existing Job to Serverless
In the **Job details** side panel, either:
- Click **Swap** under **Compute** → click **New** → enter/update settings → click **Update**, or
- Use the **Compute** drop-down menu on a task and select **Serverless**.

### Scheduling a Notebook Directly
You can also create and run a job that uses serverless compute directly from the **notebook UI**, without going through the Jobs UI first.

## 5. Performance Modes

The **Performance optimized** setting (Job details page) controls how fast a job's serverless tasks run. Both modes use the same SKU; they differ in DBU consumption and startup latency.

| Mode | Behavior | Best For |
|---|---|---|
| **Standard performance** (Performance optimized disabled) | Uses less compute to reduce cost; startup latency of roughly **4–6 minutes**, depending on compute availability and scheduling | Workloads that can tolerate slightly higher startup latency |
| **Performance optimized** (enabled) | Starts and runs faster, consuming more DBUs | Time-sensitive workloads |

> [!NOTE]
> A job must have at least one serverless task to configure **Performance optimized** in the UI, and the setting only affects the serverless tasks within that job. **Standard performance mode is not supported** for one-time runs created via the `runs/submit` endpoint.

## 6. Serverless Usage Policies (Billing Attribution)

> [!NOTE]
> This feature is in **Public Preview**.

**Serverless usage policies** let your organization apply custom tags to serverless usage for granular billing attribution. If your workspace uses serverless usage policies, you select the job's policy via the **Usage policy** setting in the job details UI. If you're assigned to only one policy, it's auto-selected for new jobs.

- Existing jobs are **not automatically tagged** once you're assigned a policy — you must manually update existing jobs to attach a policy.

## 7. Configuration, Environments & High Memory

### Spark Configuration Parameters
Serverless compute only allows setting a **specific, restricted list** of Spark configuration parameters (see the Supported Spark configuration parameters reference). These can only be set at the **session level** — e.g., inside a notebook that's added as a task in the job, rather than at the compute/cluster level (compute-scoped configuration isn't supported on serverless).

### Environments and Dependencies
Library/dependency configuration for serverless compute is managed through the **serverless environment** settings rather than cluster-level libraries. Notebook-scoped libraries are the supported mechanism for adding dependencies.

### High Memory for Notebook Tasks
> [!NOTE]
> This feature is in **Public Preview**.

You can configure a notebook task to use a higher memory size via the **Memory** setting in the notebook's **Environment** side panel. High memory is **only available for notebook task types**.

## 8. Auto-Optimization & Retries

Serverless compute for workflows includes **auto-optimization**, which automatically optimizes the compute used to run jobs and **retries failed tasks**. It is enabled by default, and Databricks recommends leaving it enabled to ensure critical workloads run successfully at least once.

If your workload **must execute at most once** (e.g., a non-idempotent job), you can disable auto-optimization per task:
1. Next to **Retries**, click **Add** (or the edit icon if a retry policy already exists).
2. In the **Retry Policy** dialog, uncheck **Enable serverless auto-optimization (may include additional retries)**.
3. Click **Confirm**, then save/create the task.

## 9. Sharing Compute Across Tasks (Classic Comparison)

> [!NOTE]
> Compute sharing across tasks in the way described below applies to **classic jobs compute**. Serverless compute is provisioned per task automatically by Databricks; it does not require manual cluster sharing configuration, though Databricks still manages efficient resource reuse behind the scenes.

For classic compute, configuring tasks to share the same jobs compute resource can reduce latency from repeated cluster start-up:

| | Single Task | Shared Across Tasks |
|---|---|---|
| **Start** | When the task run begins | When the first task configured to use the resource begins |
| **Terminate** | After the task runs | After the final task configured to use the resource runs |
| **Idle compute** | Not applicable | Remains on and idle while non-using tasks run |

- A shared job cluster is scoped to a **single job run** and cannot be reused by other jobs or other runs of the same job.
- Libraries **cannot** be declared at the shared job cluster level — declare dependent libraries in each task's settings instead.
- When multiple tasks share a jobs compute resource, they run on the **same driver JVM** — class state and singletons persist across tasks for the run's duration. Watch for:
  - **Scala singleton/companion object state** persisting and potentially being overwritten by concurrent tasks.
  - **Libraries loaded by one task remaining available** to subsequent tasks.
  - For task-level isolation, use a separate compute resource per task, add explicit `depends_on` ordering, or refactor code to avoid shared mutable state.

## 10. Limitations

> [!IMPORTANT]
> Top-level serverless-for-jobs limitations:
> - **No support for Continuous scheduling** with serverless compute (i.e., a job-level **Continuous** trigger is not supported on serverless).
> - **No support for default or time-based interval Structured Streaming triggers** on serverless.

### Streaming Limitations
Serverless compute only supports these Structured Streaming triggers:
- `Trigger.AvailableNow()` — **recommended** for serverless compute.
- `Trigger.Once()` — supported but deprecated, not recommended.

**Not supported**: `Trigger.Continuous(interval)` and `Trigger.ProcessingTime(interval)` (including the implicit default of `Trigger.ProcessingTime("0 seconds")` when no trigger is specified — you must explicitly set a supported trigger). Attempting an unsupported trigger raises `INFINITE_STREAMING_TRIGGER_NOT_SUPPORTED`.

For continuous streaming-style workloads on serverless, use **continuous pipeline mode** in a Lakeflow Spark Declarative Pipeline, or use `Trigger.AvailableNow()` inside a job configured with a **Run jobs continuously** trigger.

### Job-Specific Limitations
- **Task logs are not isolated per task run** — logs may contain output from multiple tasks.
- **Task libraries are not supported for notebook tasks** — use notebook-scoped libraries instead.
- By default, **serverless jobs have no query execution timeout**; set one via the `spark.databricks.execution.timeout` Spark property if needed.

### Other Notable Serverless Limitations (apply broadly, including jobs)
- **R is not supported**; only **Spark Connect APIs** are available (Spark RDD APIs are not).
- **Unity Catalog is required** to connect to external data sources; DBFS access is limited (use UC volumes or workspace files).
- **Compute policies, compute-scoped init scripts, compute-scoped libraries, instance pools, compute event logs, and environment variables are not supported.**
- **Most Spark compute configurations are not configurable** — only a documented allow-list of Spark properties can be set.
- **Dataframe/SQL caching APIs are not supported** (`df.cache()`, `CACHE TABLE`, etc. raise an exception).
- **JAR libraries are not supported in notebooks** (JAR *tasks* in jobs are supported separately, currently in Public Preview).
- **Databricks Container Services is not supported.**
- **Cross-workspace access** is allowed only within the same region, and only if the destination workspace has no IP ACL or front-end PrivateLink configured.

## 11. REST API & DAB — Syntax & Examples

### REST API: Notebook Task on Serverless Compute
Omitting any cluster-related fields (`new_cluster`, `existing_cluster_id`) on a supported task type results in the task running on serverless compute by default.
```json
{
  "name": "serverless-etl-job",
  "tasks": [
    {
      "task_key": "ingest_notebook",
      "notebook_task": {
        "notebook_path": "/Workspace/Users/user@company.com/ingest_notebook"
      }
    }
  ]
}
```

### REST API: Notebook Task with Performance Optimized Mode and Environment Spec
```json
{
  "task_key": "transform",
  "notebook_task": {
    "notebook_path": "/Workspace/Users/user@company.com/transform"
  },
  "environment_key": "default",
  "performance_target": "PERFORMANCE_OPTIMIZED"
}
```

### DAB Example — Serverless Job with Environment Spec
```yaml
resources:
  jobs:
    serverless_job_environment:
      name: serverless_job_environment
      tasks:
        - task_key: task
          spark_python_task:
            python_file: ../src/main.py
          # References an environment spec defined below
          environment_key: default
      environments:
        - environment_key: default
          spec:
            environment_version: '2'
            dependencies:
              - my-library
```

### DAB Example — Mixed Compute: Serverless Notebook + Classic JAR Task
```yaml
resources:
  jobs:
    mixed_compute_job:
      name: mixed_compute_job
      tasks:
        - task_key: ingest
          notebook_task:
            notebook_path: ../src/ingest.ipynb
          # No cluster fields -> runs on serverless by default

        - task_key: legacy_jar_step
          depends_on:
            - task_key: ingest
          new_cluster:
            spark_version: 15.4.x-scala2.12
            node_type_id: i3.xlarge
            num_workers: 2
          libraries:
            - jar: ../dist/legacy_step.jar
          spark_jar_task:
            main_class_name: com.example.LegacyStep
```

### Deploy & Run
```bash
databricks bundle validate -t prod
databricks bundle deploy -t prod
databricks bundle run -t prod serverless_job_environment
```

## 12. Monitoring Cost and Query Details

### Cost Monitoring
Monitor the cost of jobs using serverless compute via the **billable usage system table** (`system.access.billing` / billable usage system table reference), which includes user and workload attributes for serverless costs. Current pricing is published on the Databricks pricing page.

### Billing on the Invoice
- Classic compute appears as **"Jobs Compute"**.
- Classic compute with Photon appears as **"Jobs Compute (Photon)"**.
- Serverless compute on AWS/GCP appears as **"Jobs Serverless Compute"**.
- Serverless compute on Azure appears as **"Automated Serverless Compute"**.
- Serverless compute is billed with a **one-minute minimum**; idle resources are automatically wound down after a period of inactivity.
- You are not charged for compute start-up time — Databricks maintains a ready fleet, so billing starts once the job actually begins executing.

### Viewing Query Details for Job Runs
1. **Jobs & Pipelines** → open the job → open the specific run.
2. Click **Timeline** to view the run split into individual tasks.
3. Expand a task to see its query statements and runtimes.
4. Click a statement to open its **query profile** for execution details and metrics.
5. From the **Compute** section of the **Task run** panel, click **Query history** to see prefiltered query history for that task run.

## 13. Best Practices

**Default Choice:**
- Databricks recommends using **serverless compute for all job tasks** where the task type supports it — it removes infrastructure management and cluster-creation-permission bottlenecks.
- Reserve classic jobs/all-purpose compute for task types serverless doesn't support (JAR for production stability, Spark Submit) or workloads hitting a documented serverless limitation (e.g., Scala/R requirements, custom init scripts, instance pools).

**Performance vs. Cost:**
- Use **Performance optimized** mode for latency-sensitive jobs (e.g., feeding a near-real-time dashboard via a table update trigger); use **standard performance** mode for cost-sensitive batch jobs that can tolerate a few minutes of extra startup latency.

**Streaming Workloads:**
- Use `Trigger.AvailableNow()` for batch-style streaming ingestion on serverless — it's the only fully recommended trigger.
- For workloads that genuinely need to run continuously, prefer a continuous-mode Lakeflow Spark Declarative Pipeline (serverless-compatible) over trying to force `Trigger.ProcessingTime` or `Trigger.Continuous` directly, which are unsupported on serverless.
- Remember serverless jobs do **not** support the job-level **Continuous** trigger type at all — use the pipeline's continuous mode instead, or fall back to classic compute if you specifically need a continuously-triggering job.

**Reliability:**
- Leave **auto-optimization** enabled for idempotent jobs to benefit from automatic retries; explicitly disable it only for non-idempotent, at-most-once workloads.
- Be cautious sharing classic compute across tasks with Scala companion objects or other mutable singleton state — isolate tasks onto separate compute resources if correctness depends on it.

**Cost Governance:**
- Use **serverless usage policies** (Public Preview) to tag and attribute serverless spend per team/project, and remember to retroactively tag pre-existing jobs since tagging isn't automatic.
- Query the **billable usage system table** regularly to track serverless DBU consumption trends per job.

**Compatibility Checks Before Migration:**
- Audit notebooks/scripts for R or Scala UDF usage, DBFS mounts with instance profiles, Hive SerDe tables, dataframe caching APIs, and compute-scoped libraries before migrating a classic job to serverless — these are the most common blockers surfaced by the serverless compute limitations list.

## 1. Spark UI Overview — Access & Navigation

### How to Access the Spark UI in Databricks
1. From the **Lakeflow Jobs UI** → click a running or completed task → click **Spark UI** link in the task run detail panel
2. From a **running cluster** → Compute → click the cluster → **Spark UI** tab
3. From a **notebook** → top right corner when a cluster is attached → cluster name → **Spark UI**

### Spark UI Main Tabs
| Tab | What It Shows |
|---|---|
| **Jobs** | All Spark jobs (actions) triggered in this session/run |
| **Stages** | All stages across all jobs — where most diagnosis happens |
| **Tasks** | Individual task-level breakdown within a stage |
| **Storage** | Cached/persisted RDDs and DataFrames |
| **Environment** | Spark configurations and JVM properties |
| **Executors** | Per-executor memory, GC time, task counts |
| **SQL / DataFrame** | Logical and physical query plans with execution metrics |

## 2. Jobs Tab — Spark Job Hierarchy

### Key Concepts
- A **Spark job** is triggered by an **action** (e.g., `count()`, `write()`, `collect()`, `show()`)
- A job is divided into **stages** separated by **shuffle boundaries**
- Stages are divided into **tasks** — one task per partition

### Hierarchy
```
Spark Application
└── Job  (1 per action)
    └── Stage  (1+ per job, separated by shuffles)
        └── Task  (1 per partition)
```

### Jobs Tab Columns
| Column | Description |
|---|---|
| Job ID | Unique identifier |
| Description | Linked to the line of code that triggered the action |
| Submitted | Timestamp |
| Duration | Wall-clock time |
| Stages | Count of total / succeeded / failed stages |
| Tasks | Count of total / succeeded / skipped tasks |

## 3. Stages Tab — The Core Diagnostic View

The **Stages tab** is the primary destination for performance diagnosis.

### Stages Tab Columns
| Column | Description |
|---|---|
| Stage ID | Unique identifier for the stage |
| Description | Code context (join, groupBy, write, etc.) |
| Duration | Total time for the stage |
| Tasks | Total / succeeded / failed / killed |
| Input | Data read from storage (files, Delta) |
| Output | Data written to storage |
| Shuffle Read | Data read from shuffle files (bytes) |
| Shuffle Write | Data written to shuffle files (bytes) |
| Spill (Memory) | Data deserialized in memory that was spilled |
| Spill (Disk) | Serialized data written to disk due to memory pressure |

### Finding Bottleneck Stages
Sort by **Duration** (descending) to find the slowest stage. Then click the stage to see task-level detail.

### Summary Metrics Table
| Metric | Min | 25th % | Median | 75th % | Max |
|---|---|---|---|---|---|
| Task Duration | | | | | |

| Shuffle Write Size | | | | | |

| Spill (Memory) | | | | | |
| Spill (Disk) | | | | | |

### How to Read Summary Metrics

| Pattern | Interpretation |
|---|---|
| **Max Duration >> 75th %** | Data skew — one or few tasks processing much more data |
| **Shuffle Read: Max >> Median** | Shuffle skew — uneven shuffle partition sizes |
| **Spill (Disk) > 0** | Executor memory pressure — data written to disk |
| **GC Time > 10% of Duration** | Excessive garbage collection — likely memory pressure |
| **Min << Median (very fast tasks)** | Some tasks might be processing empty or tiny partitions |

## 5. Task-Level Metrics — Inside a Stage

Within a stage's task table, each row represents one task (one partition):

### Task Table Key Columns
| Column | Description |
|---|---|
| Task ID | Unique per stage |
| Status | Success, Failed, Killed |
| Duration | Wall-clock time for this individual task |
| GC Time | Time this task's JVM spent in garbage collection |
| Shuffle Read Size | Bytes this task read from shuffle files |
| Shuffle Write | Bytes this task wrote to shuffle files |
| Spill (Memory) | Memory spilled for this task |
| Spill (Disk) | Disk bytes spilled for this task |
| Errors | Error message if task failed |

### How Partitions Map to Tasks
- Default shuffle partitions: **200** (`spark.sql.shuffle.partitions`)
- In a 200-partition shuffle, 200 tasks run in the downstream stage
- If cluster has 20 cores, tasks run in **waves** of 20 at a time (10 waves)
- **Uneven partition sizes** → some tasks in the last wave run much longer = apparent skew

## 6. Data Skew — Concepts & Detection

### What is Data Skew?
Data skew occurs when **data is unevenly distributed across partitions** — some partitions have far more rows or bytes than others. The tasks processing large partitions become **straggler tasks** that delay the entire stage.

### Root Causes
| Cause | Example |
|---|---|
| Hot key in join | `customer_id = 'UNKNOWN'` appears in 40% of rows |
| Null values | `NULL` hashed to the same partition |
| Low cardinality group-by | `GROUP BY country` where 90% of data is one country |
| Timestamp rounding | All events fall in the same hour bucket |

### Detecting Skew in the Spark UI

**Step 1 — Stages tab**: Find the slowest stage (sort by duration)

**Step 2 — Summary Metrics**: Compare Max vs. Median Duration:
```
Healthy stage:  Min=1s   Median=2s   Max=3s      (small variance)
Skewed stage:   Min=0.1s Median=1s   Max=45min   (SKEW — Max >> Median)
```

**Step 3 — Task table**: Look at Shuffle Read Size per task:
```
Healthy: most tasks read 50–100 MB each
Skewed:  most tasks read 50 MB, but 1–2 tasks read 5 GB each
```

**Step 4 — Stage progress bar**: Stage stuck at 99% for minutes while the rest of the tasks completed = 1 or 2 straggler tasks holding the stage

### Programmatic Skew Detection
```python
from pyspark.sql.functions import spark_partition_id, count

# Check row count per partition
df.groupBy(spark_partition_id().alias("partition_id")) \
  .agg(count("*").alias("row_count")) \
  .orderBy("row_count", ascending=False) \
  .show(20)

# A healthy result: all partitions have similar row counts
# A skewed result: partition 0 has 10M rows, all others have ~50K
```

## 7. Shuffle — Concepts & Metrics

### What is a Shuffle?
A **shuffle** redistributes data across executor nodes based on a key (e.g., join key, group-by key). It is one of the most expensive operations in Spark because it:
1. Writes intermediate data to disk (**shuffle write**)
2. Transfers data across the network between executors
3. Reads data from other executors (**shuffle read**)

### When Shuffles Occur
| Operation | Shuffle? |
|---|---|
| `join()` (non-broadcast) | Yes — both sides hashed by key |
| `groupBy()` + aggregation | Yes — data grouped by key |
| `orderBy()` / `sort()` | Yes — data sorted globally |
| `repartition(n)` | Yes — explicit repartition |
| `coalesce(n)` | No — only reduces partitions locally |
| `filter()`, `select()` | No — narrow transformations |
| `map()`, `flatMap()` | No — narrow transformations |
| broadcast join | No — eliminates shuffle on small side |

### Shuffle Metrics in Stages Tab
| Metric | Description |
|---|---|
| **Shuffle Write** | Total bytes all tasks wrote as shuffle output (before network transfer) |
| **Shuffle Read** | Total bytes all tasks read from remote executors' shuffle files |
| **Shuffle Read Blocked Time** | Time tasks spent waiting for shuffle data from other executors |

### Healthy vs. Problematic Shuffle
| Condition | Indicator | Action |
|---|---|---|
| Large Shuffle Write (GB+) | Potentially expensive | Check if broadcast join can eliminate it |
| Large Shuffle Read >> Write | Data being repeated/replicated | Check for skew on the read side |
| High Shuffle Read Blocked Time | Network bottleneck or GC pauses | Check executor GC, network |
| Max Shuffle Read >> Median | Shuffle skew | Tune partitions or enable AQE |

### Shuffle Partition Tuning
```python
# Default: 200 partitions (often too few for large datasets)
spark.conf.set("spark.sql.shuffle.partitions", "1000")

# Target: 100-200 MB per partition after shuffle
# If shuffle write = 100 GB and you want 200 MB partitions:
# 100 GB / 200 MB = 500 partitions

# AQE (preferred): automatically adjusts partition count at runtime
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")
```

## 8. Disk Spill & Memory Spill — Concepts & Detection

### What is Spill?
Spill occurs when a task needs more memory for its computation than is available in the executor's **execution memory pool**. Spark writes the excess data to disk temporarily, then reads it back — causing significant performance degradation.

```
Execution Memory Full
        │
        ▼
Spark serializes data → writes to local disk (Spill Disk)
        │
        ▼
Later reads data back from disk for computation
        ▼ (I/O much slower than RAM — 10-100x penalty)
```

### Spill (Memory) vs. Spill (Disk) — The Difference

| Metric | What It Represents |
|---|---|
| **Spill (Memory)** | Size of the data **before serialization** (in memory, uncompressed) |
| **Spill (Disk)** | Size of the data **after serialization** written to disk (compressed) |

> Spill (Memory) is typically **larger** than Spill (Disk) because serialization compresses the data.
> Both metrics refer to the **same spilled data**, just measured differently.

### Finding Spill in Spark UI
1. **Stages tab** → `Spill (Memory)` and `Spill (Disk)` columns — non-zero = spill occurred
2. **Stage detail → Summary Metrics** → same columns broken down by task (min/median/max)
3. **Task table** → individual task-level spill amounts

### Operations Most Likely to Cause Spill
| Operation | Why It Can Spill |
|---|---|
| Large `join()` | Builds a hash table in memory; overflows if too large |
| `groupBy()` + aggregation | Holds partial aggregates; spills if too many keys |
| `orderBy()` / `sort()` | Sorts all data; spills when data > sort memory |
| Over-caching | `cache()` on data that doesn't fit fills storage memory → pushes execution memory down |

### Spill Diagnosis Checklist

| Symptom | Location | Root Cause |
|---|---|---|
| `Spill (Disk) > 0` in Stages tab | Stages → Spill column | Memory pressure from large partitions |
| Max Spill >> Median Spill | Stage Summary Metrics | Skewed partitions causing one task to spill heavily |
| `ExecutorLostFailure` in error | Executor logs | OOM — executor killed by OS/YARN for exceeding memory |
| `java.lang.OutOfMemoryError: GC overhead` | Task error | JVM GC unable to free enough memory |

## 9. Straggler Tasks — Patterns & Diagnosis

A **straggler task** is a task that takes significantly longer than its peers in the same stage.

### Visual Pattern in Spark UI
```
Stage progress: [===================99%] ← stuck here for minutes

Task Duration Distribution:
Task 0:   2s  ██
Task 1:   3s  ███
Task 2:   2s  ██
...
Task 47:  28min ████████████████████████████████████ ← straggler
```

### Common Causes of Stragglers
| Cause | Spark UI Evidence |
|---|---|
| Data skew | Max Duration >> Median, Max Shuffle Read >> Median |
| Disk spill on one task | Max Spill (Disk) >> others in same stage |
| GC pressure on one executor | Max GC Time >> others, executor has high GC% |
| Network slowness to one node | High Shuffle Read Blocked Time for specific task |
| Speculative execution triggered | Two tasks with same index — one slower, one faster |

### Speculative Execution
Databricks enables **speculative execution** by default:
- If a task takes significantly longer than the median, a duplicate **speculative task** is launched on a different executor
- Whichever finishes first wins; the other is killed
- Visible in Spark UI: two task rows with the same partition index

## 10. Adaptive Query Execution (AQE)

AQE is Databricks' automatic optimization framework that adjusts the query plan **at runtime** based on actual data statistics.

### AQE Key Features
| Feature | What It Does | Config |
|---|---|---|
| **Coalesce Shuffle Partitions** | Merges small shuffle partitions to reduce task count | `spark.sql.adaptive.coalescePartitions.enabled` |
| **Handle Skew Join** | Detects skewed partitions in joins and splits them | `spark.sql.adaptive.skewJoin.enabled` |
| **Switch Join Strategy** | Converts sort-merge join to broadcast join if one side is small | Part of `spark.sql.adaptive.enabled` |

### Default Status in Databricks
As of Databricks Runtime 7.3+, AQE is **enabled by default**:
```python
# Verify AQE is enabled (should be true by default in Databricks)
spark.conf.get("spark.sql.adaptive.enabled")   # → "true"
```

### AQE and Skew Join
```python
# Explicitly enable skew join handling (enabled by default in Databricks)
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")

# Threshold for what is considered a skewed partition:
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionFactor", "5")
# A partition is skewed if its size > 5x the median partition size
```

### When AQE Cannot Fix Skew
- Skew in non-join operations (e.g., `groupBy` aggregation)
- Extreme skew where one partition is 100x+ the median
- Custom code that bypasses the Spark SQL optimizer

## 11. Remediation Strategies

### Strategy 1: Enable/Verify AQE (First Resort)
```python
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")
```

### Strategy 2: Broadcast Join (Eliminate Shuffle)
```python
from pyspark.sql.functions import broadcast

# Join where small_df fits in driver/executor memory
result = large_df.join(broadcast(small_df), "customer_id")

# Or via config (auto-broadcast tables smaller than threshold)
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "50m")  # 50 MB
```

### Strategy 3: Salting (Manual Skew Fix for Joins)
```python
from pyspark.sql.functions import rand, floor, concat_ws, col, lit

N = 10  # salt buckets

# Add salt to the skewed table
large_df_salted = large_df.withColumn(
    "salt", floor(rand() * N).cast("string")
).withColumn("salted_key", concat_ws("_", col("join_key"), col("salt")))

# Explode the small table to match all salt values
small_df_exploded = small_df.crossJoin(
    spark.range(N).withColumnRenamed("id", "salt")
).withColumn("salted_key", concat_ws("_", col("join_key"), col("salt").cast("string")))

result = large_df_salted.join(small_df_exploded, "salted_key")
```

### Strategy 4: Tune Shuffle Partitions
```python
# Rule of thumb: target 100-200 MB per partition
# If shuffle write is 200 GB → set 1000-2000 partitions
spark.conf.set("spark.sql.shuffle.partitions", "1000")

# With AQE enabled, use a high upper bound and let AQE coalesce
spark.conf.set("spark.sql.adaptive.coalescePartitions.initialPartitionNum", "1000")
```

### Strategy 5: Repartition Explicitly
```python
# Repartition on a high-cardinality column to distribute evenly
df.repartition(400, "customer_id").groupBy("customer_id").agg(...)

# For writes: repartition before write to control output file count
df.repartition(16).write.mode("overwrite").parquet("/output/")
```

### Strategy 6: Fix Disk Spill — Increase Executor Memory
```python
# Option A: Use memory-optimized instance types (e.g., r-series on AWS)

# Option B: Tune memory fraction (more execution memory, less storage memory)
spark.conf.set("spark.memory.fraction", "0.8")          # default 0.6
spark.conf.set("spark.memory.storageFraction", "0.3")   # default 0.5

# Option C: Increase shuffle partitions to reduce per-partition data size
spark.conf.set("spark.sql.shuffle.partitions", "2000")
```

### Strategy 7: Avoid Unnecessary Data in Memory
```python
# Apply filters early (push predicates down)
df.filter(col("date") >= "2024-01-01") \
  .select("customer_id", "amount") \
  .join(...)   # Less data to shuffle = less spill

# Avoid collecting large DataFrames to driver
# BAD:
data = df.collect()  # OOM risk for large DataFrames

# GOOD:
df.write.mode("overwrite").parquet("/tmp/output/")
```

## 12. Executor Tab — Memory & GC Metrics

The **Executors tab** shows per-executor resource usage:

### Key Executor Columns
| Column | Description | Warning Sign |
|---|---|---|
| **Task Time** | Total CPU time spent on tasks | Low value relative to wall time = idle executors |
| **GC Time** | Time in garbage collection | > 10% of task time = memory pressure |
| **Peak JVM Memory Used** | Highest JVM heap usage | Near or at `spark.executor.memory` = at risk of OOM |
| **Shuffle Read** | Total data read by this executor | Much higher than peers = skew landing on one executor |
| **Shuffle Write** | Total data written | |
| **Storage Memory Used** | Cached RDD/DataFrame size | High = less execution memory available |
| **Failed Tasks** | Tasks that failed on this executor | Non-zero = unstable executor |

### GC Time Threshold
- **< 5%** of task time: Healthy
- **5–10%**: Monitor, may need tuning
- **> 10%**: Significant GC overhead — increase executor memory or reduce data per executor

## 13. SQL / DataFrame Query Plan Tab

The **SQL tab** shows each SQL query or DataFrame action with:

### Query Plan Visualization
- **Exchange** nodes = shuffle operations (look for their size estimates)
- **Sort** nodes = potential spill sources
- **BroadcastHashJoin** = broadcast join (good — no shuffle)
- **SortMergeJoin** = shuffle-based join (watch for skew)
- **HashAggregate** = in-memory aggregation (can spill)

### AQE Adaptive Plan
When AQE is enabled, the SQL tab shows both:
- **Initial plan** (before execution)
- **Adaptive plan** (after AQE modifications at runtime)

Check if AQE converted a `SortMergeJoin` to `BroadcastHashJoin` — this means AQE eliminated a shuffle automatically.

## 14. Diagnostic Decision Tree

```
Job is slow or failing
        │
        ▼
Open Spark UI → Stages tab
Sort by Duration (descending)
        │
        ▼
Click slowest stage → Summary Metrics
        │
        ├── Spill (Disk) > 0?
        │       YES → Memory pressure / spill
        │       ├── Check per-task spill: Max >> Median? → Skew causing spill
        │       │       Fix: AQE skew join, salting, repartition
        │       └── All tasks spilling? → Not enough executor memory
        │               Fix: more partitions, larger instances, tune memory.fraction
        │
        ├── Max Duration >> 75th percentile?
        │       YES → Data skew
        │       ├── Shuffle Read Max >> Median? → Shuffle skew
        │       │       Fix: AQE, salting, broadcast join
        │       └── No shuffle difference → Compute skew (code logic)
        │               Fix: repartition before expensive operation
        │
        ├── High GC Time?
        │       YES → Memory pressure on JVM heap
        │       Fix: increase executor memory, reduce cache(), tune memory.fraction
        │
        └── All tasks slow uniformly?
                YES → Cluster under-provisioned or data volume too large
                Fix: more executors/cores, larger instance type
```

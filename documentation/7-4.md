## 1. What is ABAC?

**Attribute-Based Access Control (ABAC)** is a Unity Catalog security model where access decisions are driven by **attributes** (governed tags) rather than by explicit per-object grants.

### ABAC Key Idea
```
Traditional approach (per-table):
  ALTER TABLE table_A SET MASK ...
  ALTER TABLE table_B SET MASK ...
  ALTER TABLE table_C SET MASK ...   ← must repeat for every table

ABAC approach (tag-based):
  1. Tag table_A, table_B, table_C with: sensitivity = 'PII'
  2. Define one policy at catalog/schema level:
     "Apply ssn_mask to any column tagged sensitivity=PII"
  → All three tables are automatically protected
  → New tables with the tag are automatically protected too
```

### Why ABAC?
- **Scale**: One policy protects hundreds of tables
- **Automation**: New tables inheriting a tag are automatically protected
- **Separation of duties**: Governance teams manage tags and policies; data teams manage data
- **Consistency**: No "governance gaps" from forgetting to apply a per-table mask

---

## 2. ABAC vs. Table-Level Policies

| Dimension | Table-Level (`ALTER TABLE SET MASK`) | ABAC Policy (`CREATE POLICY`) |
|---|---|---|
| Scope | One table | Catalog / Schema / Table |
| Trigger | Explicit ALTER command | Governed tag on object |
| Scalability | Low — manual per table | High — automatic via tags |
| New table coverage | Manual — must re-apply | Automatic — inherits from schema/catalog |
| DBR Requirement | DBR 15.4 LTS | DBR 16.4+ or Serverless |
| Best For | One-off, unique cases | Enterprise-wide PII/PHI compliance |

## 3. Governed Tags — The Foundation of ABAC

**Governed tags** are account-level metadata attributes with:
- **Controlled keys**: Only authorized users can create tag keys
- **Optional allowed values**: Keys can have a fixed list of valid values
- **Strict assignment**: Only authorized users can apply governed tags to objects

### How Governed Tags Differ from Regular Tags
| Feature | Regular Tags | Governed Tags |
|---|---|---|
| Who creates keys | Any user with permissions | Account admins only |
| Value validation | Free-form | Optional restricted values |
| Used in ABAC policies | No | Yes — required for ABAC |
| Audit trail | Standard | Full audit trail |

### Creating a Governed Tag (Admin)
```sql
-- Create a governed tag key with restricted values
CREATE TAG sensitivity ALLOWED VALUES ('PII', 'PHI', 'INTERNAL', 'PUBLIC');

-- Create a governed tag with no value restriction
CREATE TAG data_owner;
```

### Applying Tags to Objects
```sql
-- Tag a table
ALTER TABLE catalog.schema.customers SET TAGS ('sensitivity' = 'PII');

-- Tag a specific column
ALTER TABLE catalog.schema.customers
ALTER COLUMN ssn SET TAGS ('sensitivity' = 'PII');

-- Tag a schema (inherits to tables for ABAC evaluation)
ALTER SCHEMA catalog.pii_schema SET TAGS ('sensitivity' = 'PII');
```

---

## 4. Governed vs. Ungoverned Tags

ABAC policies **require governed tags** — ungoverned (free-form) tags cannot trigger ABAC policies.

```
Ungoverned tag: any user can create, no value validation
→ Cannot be used in ABAC policies

Governed tag: admin-controlled key, optional value list
→ Can be used in ABAC policies
```

## 5. ABAC Policy Scope — Catalog, Schema, Table

An ABAC policy can be attached to:
- **Catalog** — applies to all schemas and tables within
- **Schema** — applies to all tables within the schema
- **Table** — applies only to that specific table

### Policy Attachment Levels
```
ABAC Policy attached at CATALOG level:
    └── Applies to all SCHEMAS in the catalog
        └── Applies to all TABLES, VIEWS, STREAMING TABLES in each schema

ABAC Policy attached at SCHEMA level:
    └── Applies to all TABLES, VIEWS, STREAMING TABLES in the schema
    (Does NOT affect sibling schemas)

ABAC Policy attached at TABLE level:
    └── Applies only to that specific table
```

## 6. Policy Inheritance — How Policies Flow Down

ABAC policies inherit **downward** through the hierarchy, but only for objects that bear the matching governed tag.

### Key Inheritance Rule
```
Policy at CATALOG level:
  → Evaluates each TABLE in the catalog
  → If the TABLE has the matching tag → policy applies
  → If the TABLE does not have the tag → policy does not apply

Example:
  CREATE POLICY mask_pii ON CATALOG prod_catalog
  COLUMN MASK ssn_mask
  WHEN COLUMN TAG('sensitivity') = 'PII';

  → prod_catalog.hr.employees.ssn  (tagged sensitivity=PII) → masked
  → prod_catalog.hr.employees.name (no PII tag)             → not masked
  → prod_catalog.finance.revenue   (no PII tag on table)    → not masked
```

### IMPORTANT: Tag Inheritance to Columns
- Tags applied to a **table** are inherited for ABAC evaluation at the table level
- Tags applied to a **schema/catalog** are inherited for ABAC evaluation at the table level
- Tags do **NOT** automatically inherit to the **column level**
- For column-level masking based on column tags → columns must be tagged **explicitly**

```sql
-- Table-level tag does NOT automatically tag the columns:
ALTER TABLE catalog.schema.users SET TAGS ('sensitivity' = 'PII');
-- This tags the TABLE, not the individual columns

-- To trigger a COLUMN-level policy based on column tags:
ALTER TABLE catalog.schema.users
ALTER COLUMN ssn SET TAGS ('sensitivity' = 'PII');  -- explicit column tag
```

## 7. Creating an ABAC Column Mask Policy

### Pattern
```sql
-- Step 1: Create the UDF that defines the masking logic
CREATE OR REPLACE FUNCTION catalog.governance.ssn_mask(ssn STRING)
RETURNS STRING
RETURN CASE
  WHEN is_account_group_member('pii-authorized') THEN ssn
  ELSE '***-**-****'
END;

-- Step 2: Create the ABAC policy at catalog or schema level
CREATE POLICY mask_ssn_policy ON CATALOG prod_catalog
  COLUMN MASK catalog.governance.ssn_mask
  TO `account users`
  WHEN COLUMN TAG('sensitivity') = 'PII';
-- Result: Any column in prod_catalog tagged sensitivity=PII gets this mask
```

### Column Mask Policy Scope Options
```sql
-- Policy at schema level (narrower scope)
CREATE POLICY mask_pii_schema ON SCHEMA prod_catalog.hr
  COLUMN MASK catalog.governance.ssn_mask
  TO `account users`
  WHEN COLUMN TAG('sensitivity') = 'PII';

-- Policy at table level (most specific)
CREATE POLICY mask_pii_table ON TABLE prod_catalog.hr.employees
  COLUMN MASK catalog.governance.ssn_mask
  TO `account users`
  WHEN COLUMN TAG('sensitivity') = 'PII';
```

## 8. Creating an ABAC Row Filter Policy

### Pattern
```sql
-- Step 1: Create the UDF (returns BOOLEAN)
CREATE OR REPLACE FUNCTION catalog.governance.region_filter(region STRING)
RETURNS BOOLEAN
RETURN is_account_group_member('admin')
    OR is_account_group_member(CONCAT(region, '-team'));

-- Step 2: Create the ABAC row filter policy at schema level
CREATE POLICY restrict_by_region ON SCHEMA prod_catalog.sales
  ROW FILTER catalog.governance.region_filter
  WHEN TABLE TAG('access') = 'restricted';
-- Result: Any table in prod_catalog.sales tagged access=restricted
-- gets the row filter applied automatically
```

## 9. Tagging Objects for Policy Activation

Once a policy is defined, tagging objects activates the policy on them.

### Tagging Tables
```sql
-- Activate the mask_pii policy on a new table
ALTER TABLE prod_catalog.hr.contractors
SET TAGS ('sensitivity' = 'PII');
-- → mask_pii_policy automatically applies to all PII-tagged columns in contractors
```

### Tagging Columns
```sql
-- Activate column-level masking on a specific column
ALTER TABLE prod_catalog.hr.employees
ALTER COLUMN national_id SET TAGS ('sensitivity' = 'PII');
-- → The column mask policy now applies to national_id
```

### Removing a Tag (Deactivates the Policy)
```sql
-- Remove the tag → policy is no longer applied to this table
ALTER TABLE prod_catalog.hr.contractors
UNSET TAGS ('sensitivity');
```

## 10. Implementation Workflow — End to End

```
1. DEFINE governed tag keys (Admin)
   → CREATE TAG sensitivity ALLOWED VALUES ('PII', 'PHI', ...);

2. CREATE UDF for masking/filtering logic
   → CREATE FUNCTION catalog.gov.ssn_mask(ssn STRING) RETURNS STRING ...;

3. CREATE POLICY at desired scope
   → CREATE POLICY mask_pii ON CATALOG prod_catalog
      COLUMN MASK catalog.gov.ssn_mask
      WHEN COLUMN TAG('sensitivity') = 'PII';

4. TAG objects (columns/tables/schemas)
   → ALTER TABLE catalog.schema.employees
      ALTER COLUMN ssn SET TAGS ('sensitivity' = 'PII');

5. VERIFY enforcement
   → Query as a non-privileged user to confirm masking
   → Use DESCRIBE TABLE EXTENDED to see applied policies
```

## 11. Required Privileges for ABAC

### To Create an ABAC Policy
| Requirement | Detail |
|---|---|
| On the securable object (catalog/schema/table) | Must be the **owner** OR have **MANAGE** privilege |
| On the UDF function | **EXECUTE** privilege |
| On parent containers | **USE CATALOG**, **USE SCHEMA** |

### To Apply a Tag to an Object
| Action | Required Privilege |
|---|---|
| Tag a table | **APPLY TAG** on the table (or ownership) |
| Tag a column | **APPLY TAG** on the table (or ownership) |
| Tag a schema | **APPLY TAG** on the schema (or ownership) |

### To Query a Policy-Protected Object
- Users only need **SELECT** on the table
- Policies run transparently — users don't need EXECUTE on the UDF or any knowledge of the policy

## 12. Compute Requirements

| Policy Type | Minimum Compute |
|---|---|
| Table-level `SET MASK` / `SET ROW FILTER` | DBR 15.4 LTS |
| ABAC policies | **DBR 16.4+** or **Serverless compute** |

> **Legacy workload warning**: Clusters running DBR < 16.4 **cannot access tables protected by ABAC policies**. They must be upgraded or excluded from the policy scope.

## 14. Tag Inheritance Rules

### What Inherits (for ABAC Evaluation)
| Tag Applied To | Inherits To (for ABAC) |
|---|---|
| Metastore | Catalogs, schemas, tables |
| Catalog | Schemas, tables within |
| Schema | Tables within |
| Table | Not columns (must be tagged explicitly) |
| Column | Only that column |

### Critical Limitation: Column Tags Don't Auto-Inherit
```
Schema tagged sensitivity=PII:
  → Tables inherit the tag (for table-level policies)
  → Columns do NOT inherit the tag
  → Column-level policies require explicit column tags

To protect specific columns:
  ALTER TABLE ... ALTER COLUMN ssn SET TAGS ('sensitivity' = 'PII');
  ALTER TABLE ... ALTER COLUMN dob SET TAGS ('sensitivity' = 'PII');
  -- Must be done individually per column
```

## 15. Limitations

### Object Compatibility
| Object Type | ABAC Support |
|---|---|
| Delta tables | Yes |
| Materialized views | Yes |
| Streaming tables | Yes |
| Standard views | **Cannot apply policy directly** (base table policies apply when queried through view) |

### Feature Compatibility
| Feature | Compatible with ABAC? |
|---|---|
| Time travel | Generally NOT supported |
| DEEP/SHALLOW CLONE | NOT supported |
| Delta Sharing | Special considerations — share owner may be exempt |
| Older DBR (< 16.4) | Cannot access ABAC-protected tables |

### Tag Constraints
- One tag value per key per object (cannot assign multiple values to the same key)
- Tags must be governed tags (not free-form) for ABAC policies
- Column tags don't automatically inherit from table/schema tags

### Separation of Duties Advantage
- Governance teams can create and manage tags and policies without needing `SELECT` on the underlying data
- Only requires `MANAGE` privilege on the securable object and `EXECUTE` on the UDF

## 16. ABAC vs. Table-Level — Comparison Summary

| Feature | Table-Level Policies | ABAC Policies |
|---|---|---|
| Mechanism | `ALTER TABLE SET MASK / SET ROW FILTER` | `CREATE POLICY ... WHEN TAG ...` |
| Trigger | Explicit per-table command | Governed tag on object |
| Scope | Single table | Catalog, Schema, or Table |
| New table auto-protection | No — manual | Yes — if tag is applied |
| DBR requirement | DBR 15.4 LTS | DBR 16.4+ or Serverless |
| Tag requirement | Not needed | Governed tags required |
| Scalability | Low | High |
| Recommended for | Specific one-off requirements | Enterprise-wide compliance |
| Per-table ALTER required | Yes | No |
| View support | Cannot apply to views | Cannot apply directly; base table policies respected |

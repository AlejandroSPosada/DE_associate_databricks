## 1. Core Concept & Naming

### What are Declarative Automation Bundles?
- **DABs** = Infrastructure as Code (IaC) for Databricks
- Define your entire project — jobs, pipelines, clusters, permissions — as YAML files
- Deploy consistently and repeatably across Dev / Staging / Production environments
- The single recommended standard for production CI/CD on Databricks (as of 2026)

### Naming Evolution
| Old Name | New Name (2026) |
|---|---|
| Databricks Asset Bundles | **Declarative Automation Bundles** |
| `databricks bundle` CLI | Same — `databricks bundle` |
| `mode: development` | Same |

### Deployment Engine (2026)
- Bundles created with **CLI 1.3.0+** use the **direct deployment engine** (default)
- Replaces the older Terraform-based deployment backend

## 2. Project Structure

Standard bundle project layout:

```
my-project/
├── databricks.yml           # REQUIRED root configuration file
├── resources/               # YAML files for each resource type
│   ├── jobs/
│   │   ├── etl_job.yml
│   │   └── reporting_job.yml
│   └── pipelines/
│       └── dlt_pipeline.yml
├── src/                     # Source code: notebooks, Python, SQL
│   ├── notebooks/
│   │   └── ingest.ipynb
│   └── etl/
│       ├── __init__.py
│       └── transform.py
├── tests/                   # Unit and integration tests
│   └── test_transform.py
├── artifacts/               # Build outputs (wheels, JARs) — NOT committed to Git
└── .github/
    └── workflows/
        └── deploy.yml       # CI/CD pipeline definition
```

> **Key rule**: `databricks.yml` must be at the project root. All paths in YAML are relative to this file.

## 3. Root Configuration — `databricks.yml`

The `databricks.yml` is the **mandatory entry point** for every bundle.

```yaml
# ─────────────────────────────────────────────────────────
# MINIMAL VALID databricks.yml
# ─────────────────────────────────────────────────────────
bundle:
  name: my-etl-project      # Required. Bundle identifier.
```

```yaml
# ─────────────────────────────────────────────────────────
# FULL STRUCTURE — All major sections shown
# ─────────────────────────────────────────────────────────
bundle:
  name: my-etl-project

workspace:
  host: https://<default-workspace-url>
  root_path: /Workspace/Users/${workspace.current_user.userName}/.bundle/${bundle.name}/${bundle.target}

include:
  - resources/jobs/*.yml
  - resources/pipelines/*.yml

variables:
  catalog_name:
    description: "Unity Catalog catalog name"
    default: "dev_catalog"
  node_type:
    description: "Worker node type"
    default: "i3.xlarge"

artifacts:
  my_wheel:
    type: whl
    path: ./src
    build: "pip install -e ."

resources:
  jobs:
    my_etl_job:
      name: "ETL Job - ${bundle.target}"
      tasks:
        - task_key: ingest
          notebook_task:
            notebook_path: ./src/notebooks/ingest.ipynb

targets:
  dev:
    default: true
    mode: development
    workspace:
      host: https://<dev-workspace>
    variables:
      catalog_name: "dev_catalog"

  prod:
    mode: production
    workspace:
      host: https://<prod-workspace>
    variables:
      catalog_name: "prod_catalog"
    run_as:
      service_principal_name: "prod-sp@example.com"
```

## 4. Top-Level Configuration Keys

| Key | Required | Description |
|---|---|---|
| `bundle.name` | YES | Unique identifier for the bundle |
| `targets` | Recommended | Defines deployment environments (dev, staging, prod) |
| `resources` | Recommended | Declares Databricks objects (jobs, pipelines, etc.) |
| `variables` | Optional | Defines reusable, overridable values |
| `workspace` | Optional | Default workspace connection settings |
| `include` | Optional | Glob patterns to include additional YAML files |
| `artifacts` | Optional | Defines build steps for Python wheels or JARs |
| `permissions` | Optional | Access control applied to all deployed resources |
| `run_as` | Optional | Identity used to execute bundle resources |

## 5. Targets — Dev / Test / Prod

Targets define **named deployment environments**. Each target can override any top-level setting.

```yaml
targets:
  dev:
    default: true              # Used when -t flag is omitted
    mode: development
    workspace:
      host: https://dev-adb.azuredatabricks.net
    variables:
      catalog_name: dev_catalog
      node_type: Standard_DS3_v2

  staging:
    mode: production
    workspace:
      host: https://staging-adb.azuredatabricks.net
    variables:
      catalog_name: staging_catalog
      node_type: Standard_DS4_v2

  prod:
    mode: production
    workspace:
      host: https://prod-adb.azuredatabricks.net
    variables:
      catalog_name: prod_catalog
      node_type: Standard_DS5_v2
    run_as:
      service_principal_name: "prod-service-principal"
    permissions:
      - level: CAN_MANAGE
        group_name: data-engineers
      - level: CAN_VIEW
        group_name: data-analysts
```

### Target Isolation Rules
| Setting | Dev target | Prod target |
|---|---|---|
| Resources prefix | `[dev <username>]` auto-added | No prefix |
| Schedules/triggers | Auto-paused | Active (honor YAML config) |
| Concurrent runs | Enabled (unlimited) | As defined in YAML |
| Deployment lock | Disabled | Enabled |
| Cluster override | Allowed (`--cluster-id`) | NOT allowed |
| DLT pipeline mode | `development: true` | `development: false` |

### Deploying to a Target
```bash
# Deploy to default target (dev)
databricks bundle deploy

# Deploy to a specific target
databricks bundle deploy -t staging
databricks bundle deploy -t prod

# Force deploy even if Git branch doesn't match
databricks bundle deploy -t prod --force
```

## 6. Deployment Modes — `development` vs `production`

Modes are **optional** convenience settings that apply a group of behaviors at once.

### `mode: development`
```yaml
targets:
  dev:
    mode: development
```

**Automatic behaviors when deploying in development mode:**
- Prepends all resource names with `[dev <short_username>]`
- Tags deployed jobs and pipelines with a `dev` Databricks tag
- Sets all DLT pipelines to `development: true`
- **Pauses** all job schedules and triggers (prevents accidental production runs)
- Enables **concurrent runs** on all jobs (faster iteration)
- **Disables the deployment lock** (no conflicts from simultaneous deploys)
- Allows `--cluster-id <id>` to override cluster definitions at deploy time

### `mode: production`
```yaml
targets:
  prod:
    mode: production
    git:
      branch: main            # Optional: validates deployer is on this branch
```

**Automatic validations when deploying in production mode:**
- Validates all DLT pipelines are `development: false`
- Validates the current Git branch matches the `git.branch` setting (if specified)
- **Does NOT allow** cluster override with `--cluster-id` or `--compute-id`
- Requires `run_as` and `permissions` to be explicitly set (when not using a service principal)
- Validates workspace paths are not overridden to user-specific paths

### No mode (default)
```yaml
targets:
  test:
    workspace:
      host: https://test-adb.azuredatabricks.net
    # No mode: — resources deployed as-is, no automatic prefix or pausing
```

## 7. Custom Presets

Presets let you **customize** or **override** the default behaviors of a mode.

```yaml
targets:
  dev:
    mode: development
    presets:
      name_prefix: "testing_"          # Override auto prefix — use 'testing_' instead
      pipelines_development: true       # Set development:true for all pipelines
      trigger_pause_status: PAUSED      # Pause all schedules/triggers
      jobs_max_concurrent_runs: 10      # Max concurrent runs for all jobs
      tags:
        department: finance             # Apply custom tag to all resources
        env: dev
```

**Preset precedence rules:**
1. Individual resource settings (highest priority)
2. Presets
3. Mode defaults (lowest priority)

> Example: If a job has `max_concurrent_runs: 5` and preset sets `jobs_max_concurrent_runs: 20`, the job's value of **5** wins.

## 8. Variables — Definition & Reference

### Defining Variables

Variables are declared in the top-level `variables` mapping:

```yaml
variables:
  # Simple string variable with a default
  catalog_name:
    description: "Unity Catalog catalog"
    default: "dev_catalog"
    type: string              # Optional: string (default), integer, boolean

  # Variable with NO default (must be set before deploy)
  db_password:
    description: "Database password — must be set via env var or --var"

  # Complex variable (type: complex for objects)
  cluster_config:
    description: "Cluster configuration object"
    type: complex
    default:
      spark_version: "15.4.x-scala2.12"
      node_type_id: "i3.xlarge"
      num_workers: 2
```

### Referencing Variables

Use `${var.<variable_name>}` anywhere in your YAML:

```yaml
resources:
  jobs:
    my_job:
      name: "ETL Job [${var.catalog_name}]"
      tasks:
        - task_key: transform
          notebook_task:
            notebook_path: ./src/transform.py
            base_parameters:
              catalog: ${var.catalog_name}
          new_cluster:
            spark_version: ${var.cluster_config.spark_version}
            node_type_id: ${var.cluster_config.node_type_id}
            num_workers: ${var.cluster_config.num_workers}
```

### Overriding Variables per Target

```yaml
variables:
  catalog_name:
    default: "dev_catalog"
  num_workers:
    default: 2

targets:
  dev:
    variables:
      catalog_name: dev_catalog
      num_workers: 2

  staging:
    variables:
      catalog_name: staging_catalog
      num_workers: 4

  prod:
    variables:
      catalog_name: prod_catalog
      num_workers: 8
```

> **Important**: Bundle variables are **deployment-time variables**. They are resolved when you run `databricks bundle deploy`, not at job run time. To pass values at run time, use **job parameters** instead.

## 9. Substitutions (Built-in Variables)

Built-in substitutions are automatically resolved at deployment time. Use the `${...}` syntax:

### Core Substitutions

| Substitution | Resolves To |
|---|---|
| `${bundle.name}` | Bundle name from `bundle.name` |
| `${bundle.target}` | Current target name (e.g., `dev`, `prod`) |
| `${workspace.host}` | Workspace URL for the current target |
| `${workspace.current_user.userName}` | Deploying user's email (e.g., `alice@company.com`) |
| `${workspace.current_user.short_name}` | Short username (e.g., `alice`) |
| `${workspace.current_user.domain_friendly_name}` | Domain-friendly name |
| `${workspace.file_path}` | Workspace file path for the bundle |
| `${workspace.root_path}` | Root path of the bundle in workspace |
| `${var.<name>}` | Value of a custom variable |
| `${resources.jobs.<job-name>.id}` | Deployed job ID |
| `${resources.pipelines.<pipeline-name>.name}` | Deployed pipeline name |
| `${resources.models.<model-name>.name}` | Deployed model name |

### Usage Examples

```yaml
bundle:
  name: hello-bundle

workspace:
  # Creates a unique path per user and per target
  root_path: /Workspace/Users/${workspace.current_user.userName}/.bundle/${bundle.name}/${bundle.target}

resources:
  pipelines:
    my_pipeline:
      name: my_pipeline
      # Use bundle.target to create target-specific schema names
      schema: pipeline_bundle_${bundle.target}
      libraries:
        - notebook:
            path: ../src/my_pipeline.ipynb
      configuration:
        # Pass workspace file path to pipeline as a config property
        bundle.sourcePath: ${workspace.file_path}/src
```

> **Note**: Use `${bundle.target}` — NOT the deprecated `${bundle.environment}`.

## 10. Setting Variable Values — 4 Methods

Variables are resolved in this **priority order** (highest to lowest):

### Method 1: `--var` Flag (Highest Priority)
```bash
# Single variable
databricks bundle deploy -t prod --var="catalog_name=prod_catalog"

# Multiple variables (comma-separated)
databricks bundle validate --var="catalog_name=prod_catalog,num_workers=8"

# Multiple --var flags
databricks bundle deploy --var="catalog_name=prod_catalog" --var="num_workers=8"
```

### Method 2: Environment Variables (`BUNDLE_VAR_` prefix)
```bash
# Linux / macOS
export BUNDLE_VAR_catalog_name=prod_catalog
export BUNDLE_VAR_num_workers=8
databricks bundle deploy -t prod

# Windows
set BUNDLE_VAR_catalog_name=prod_catalog && databricks bundle deploy -t prod

# Inline (Linux/macOS)
BUNDLE_VAR_catalog_name=prod_catalog databricks bundle deploy -t prod
```

### Method 3: `variable-overrides.json` File
```
.databricks/
└── bundle/
    ├── dev/
    │   └── variable-overrides.json
    └── prod/
        └── variable-overrides.json
```

`.databricks/bundle/prod/variable-overrides.json`:
```json
{
  "catalog_name": "prod_catalog",
  "num_workers": "8"
}
```

### Method 4: `targets.variables` in YAML (Lowest Priority)
```yaml
targets:
  prod:
    variables:
      catalog_name: prod_catalog
      num_workers: 8
```

## 11. Resources — Jobs, Pipelines, Clusters

Bundle resources map directly to Databricks REST API payloads. Supported resource types:

| Resource Type | YAML Key | Description |
|---|---|---|
| Jobs | `resources.jobs` | Lakeflow Jobs definitions |
| Pipelines | `resources.pipelines` | DLT / Lakeflow Declarative Pipelines |
| Clusters | `resources.clusters` | All-purpose cluster definitions |
| Models | `resources.models` | MLflow model registrations |
| Experiments | `resources.experiments` | MLflow experiments |
| Model serving | `resources.model_serving_endpoints` | Serving endpoints |
| Dashboards | `resources.dashboards` | Lakeview dashboards |
| Quality monitors | `resources.quality_monitors` | Data quality monitors |

### Job Resource — Full Example

```yaml
resources:
  jobs:
    etl_job:
      name: "ETL Pipeline [${bundle.target}]"

      # Schedule
      schedule:
        quartz_cron_expression: "0 0 6 * * ?"
        timezone_id: "America/Chicago"
        pause_status: UNPAUSED         # Override mode's auto-pause

      # Email notifications
      email_notifications:
        on_failure:
          - alerts@company.com

      # Tasks
      tasks:
        - task_key: ingest
          description: "Ingest raw data from source"
          notebook_task:
            notebook_path: ./src/notebooks/ingest.ipynb
            base_parameters:
              catalog: ${var.catalog_name}
              env: ${bundle.target}
          job_cluster_key: main_cluster
          timeout_seconds: 3600
          retry_on_timeout: false
          max_retries: 2

        - task_key: transform
          description: "Transform to silver layer"
          depends_on:
            - task_key: ingest
          notebook_task:
            notebook_path: ./src/notebooks/transform.ipynb
          job_cluster_key: main_cluster

        - task_key: quality_check
          depends_on:
            - task_key: transform
          python_wheel_task:
            package_name: my_package
            entry_point: run_quality_checks
          job_cluster_key: main_cluster

      # Cluster definitions shared across tasks
      job_clusters:
        - job_cluster_key: main_cluster
          new_cluster:
            spark_version: "15.4.x-scala2.12"
            node_type_id: ${var.node_type}
            num_workers: ${var.num_workers}
            spark_conf:
              spark.sql.shuffle.partitions: "200"
            custom_tags:
              project: my-etl
              env: ${bundle.target}
```

### DLT Pipeline Resource

```yaml
resources:
  pipelines:
    silver_pipeline:
      name: "Silver Layer Pipeline [${bundle.target}]"
      development: false           # Set true for dev (auto-set by mode: development)

      catalog: ${var.catalog_name}
      schema: silver

      libraries:
        - notebook:
            path: ./src/pipelines/silver_notebook.ipynb
        - notebook:
            path: ./src/pipelines/silver_helpers.py

      clusters:
        - label: default
          num_workers: 4
          node_type_id: ${var.node_type}

      configuration:
        bundle.sourcePath: ${workspace.file_path}/src
        pipelines.clusterShutdown.delay: "60s"

      continuous: false
      photon: false
      channel: CURRENT
```

## 12. Artifacts — Python Wheels & JARs

The `artifacts` section defines how custom Python libraries are built and deployed:

```yaml
artifacts:
  my_package:
    type: whl                       # whl or jar
    path: ./src/my_package          # Directory with setup.py or pyproject.toml
    build: "pip wheel . --no-deps"  # Optional: custom build command

  my_jar:
    type: jar
    path: ./lib/my_library.jar      # Pre-built JAR path
```

Using a wheel artifact in a job task:

```yaml
resources:
  jobs:
    etl_job:
      tasks:
        - task_key: run_wheel
          python_wheel_task:
            package_name: my_package    # Matches artifact name
            entry_point: main           # Entry point in the package
            parameters:
              - "--env"
              - ${bundle.target}
          libraries:
            - whl: ./src/my_package/dist/my_package-0.1.0-py3-none-any.whl
```

> When you run `databricks bundle deploy`, Databricks automatically builds the wheel and uploads it to the workspace before deploying the job.

## 13. Permissions & `run_as`

### `run_as` — Execution Identity

Controls **who runs** deployed jobs and pipelines (decoupled from the deployer's identity):

```yaml
# Global run_as (applies to all resources)
run_as:
  service_principal_name: "prod-sp@company.com"  # Recommended for prod
  # OR:
  user_name: "alice@company.com"                 # For user identity

# Per-target run_as (override global)
targets:
  dev:
    run_as:
      user_name: "${workspace.current_user.userName}"  # Developer's own identity
  prod:
    run_as:
      service_principal_name: "prod-sp@company.com"
```

### `permissions` — Access Control

```yaml
# Global permissions (applied to ALL deployed resources)
permissions:
  - level: CAN_MANAGE
    group_name: data-engineers
  - level: CAN_RUN
    group_name: data-analysts
  - level: CAN_VIEW
    service_principal_name: "monitoring-sp@company.com"

# Per-resource permissions
resources:
  jobs:
    sensitive_job:
      name: "Sensitive ETL"
      permissions:
        - level: CAN_MANAGE
          user_name: alice@company.com
        - level: CAN_RUN
          group_name: job-runners
```

**Available permission levels**: `CAN_VIEW`, `CAN_RUN`, `CAN_MANAGE`

## 14. Modularization with `include`

Break large configurations into separate files for maintainability:

```yaml
# databricks.yml — root file
bundle:
  name: large-project

include:
  - resources/jobs/*.yml
  - resources/pipelines/*.yml
  - resources/monitors/*.yml
  - config/variables.yml
```

`resources/jobs/etl_job.yml`:
```yaml
resources:
  jobs:
    etl_job:
      name: "ETL Job"
      tasks:
        - task_key: ingest
          notebook_task:
            notebook_path: ../../src/notebooks/ingest.ipynb
```

`config/variables.yml`:
```yaml
variables:
  catalog_name:
    default: dev_catalog
  node_type:
    default: i3.xlarge
```

> All included files are merged at bundle processing time. Resource keys must be unique across all included files.

## 15. Bundle CLI Commands

### Project Initialization
```bash
# Scaffold a new bundle from a template
databricks bundle init

# Available built-in templates:
# - default-python     (Python jobs + notebooks)
# - dlt                (Delta Live Tables pipeline)
# - ml                 (Machine learning project)
# - default-sql        (SQL-based workflows)
```

### Validation
```bash
# Validate syntax and schema (no deployment)
databricks bundle validate

# Validate for a specific target
databricks bundle validate -t prod

# Validate with variable overrides
databricks bundle validate -t prod --var="catalog_name=prod_catalog"

# Generate JSON schema for IDE autocomplete
databricks bundle schema > bundle_config_schema.json
```

### Deployment
```bash
# Deploy to default target
databricks bundle deploy

# Deploy to specific target
databricks bundle deploy -t staging
databricks bundle deploy -t prod

# Deploy with variable overrides
databricks bundle deploy -t prod --var="num_workers=16"

# Force deploy (skips Git branch check in production mode)
databricks bundle deploy -t prod --force

# Deploy to dev with a specific cluster ID
databricks bundle deploy -t dev --cluster-id 1234-567890-abcde123
```

### Running Resources
```bash
# Run a job in the bundle
databricks bundle run etl_job

# Run for a specific target
databricks bundle run -t prod etl_job

# Run a pipeline
databricks bundle run silver_pipeline

# Pass job parameters at runtime
databricks bundle run etl_job --python-params '["--env", "prod"]'
```

### Resource Generation & Management
```bash
# Generate YAML from existing workspace resources
databricks bundle generate

# Destroy all deployed resources (use with caution!)
databricks bundle destroy
databricks bundle destroy -t dev

# Summary of deployed resources
databricks bundle summary
```

```yaml
# ═══════════════════════════════════════════════════════════════

bundle:

permissions:
  - level: CAN_MANAGE
    group_name: data-engineers
  - level: CAN_RUN
    group_name: analysts

# ─── Targets ────────────────────────────────────────────────────
targets:

  dev:
    default: true
    mode: development
    workspace:
      host: https://adb-111111111.1.azuredatabricks.net
    variables:
      catalog_name: dev_catalog
      num_workers: 2
      node_type: Standard_DS3_v2

  staging:
    mode: production
    workspace:
      host: https://adb-222222222.2.azuredatabricks.net
    variables:
      catalog_name: staging_catalog
      num_workers: 4
      node_type: Standard_DS4_v2

  prod:
    mode: production
    workspace:
      host: https://adb-333333333.3.azuredatabricks.net

      branch: main                    # Validates we're on main before prod deploy

      node_type: Standard_DS5_v2
    run_as:
      service_principal_name: "prod-service-principal@company.com"
```

```yaml
        - task_key: transform

            notebook_path: ../../src/notebooks/transform.ipynb

            spark_version: "15.4.x-scala2.12"
```

## 17. Best Practices

### Version Control
- Commit `databricks.yml`, all resource YAML, source code, and test files to Git
- **Do NOT commit** artifacts (`.whl`, `.jar`) — build them in CI/CD
- **Do NOT commit** credentials — use Databricks Secrets

### Configuration Design
- Use **`mode: development`** for dev and test targets — benefits from auto-pause and prefix
- Use **`mode: production`** for staging and prod — enforces safety validations
- Use **variables** to keep configuration DRY (Don't Repeat Yourself)
- Use **substitutions** like `${bundle.target}` to dynamically differentiate resource names
- Use **`${workspace.current_user.userName}`** in `root_path` to prevent dev collisions between team members

### Deployment Safety
- Always run `databricks bundle validate -t <target>` before deploying
- Specify `git.branch: main` in prod target to prevent accidental off-branch deploys
- Use **Service Principals** (not personal tokens) for production CI/CD pipelines
- Use **OIDC** (Workload Identity Federation) for token-less CI/CD authentication

### Modularization
- Use `include` to split large bundles into per-resource YAML files
- Group by resource type: `resources/jobs/`, `resources/pipelines/`, etc.
- Keep variable declarations in a dedicated `config/variables.yml` for clarity

### CI/CD Workflow Pattern
```
On PR open:        databricks bundle validate -t staging
On merge to main:  databricks bundle deploy -t staging
                   (run integration tests)
On approval:       databricks bundle deploy -t prod
```

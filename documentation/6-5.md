## 1. Cluster Diagnostic Resources — Where to Look

| Resource | Location in UI | Best For |
|---|---|---|
| **Event Log** | Cluster detail → Event Log tab | Cluster lifecycle events, startup issues, init script status |
| **Driver Logs** | Cluster detail → Driver Logs tab | Application errors, library install output, driver OOM |
| **Spark UI** | Cluster detail → Spark UI (or via task detail in Jobs) | Stage/task failures, executor OOM, skew, spill |
| **Executor Logs** | Spark UI → Executors tab → Log links | Executor-specific errors, GC issues |
| **Cluster Log Delivery** | Configured cloud storage (S3/ADLS/GCS) | Long-term log retention, init script granular output |
| **Init Script Logs** | Cloud storage: `<log_delivery_path>/init_scripts/` | Exact failure point in init scripts |

### First-Responder Triage Order
1. **Event Log** — is the cluster failing before Spark even starts?
2. **Driver Logs (stderr)** — is there a Python/Java exception at startup?
3. **Spark UI Stages** — is the failure happening during task execution?
4. **Executor Logs** — is a specific executor failing with OOM/GC?

## 2. Cluster Event Log — Reading Key Events

Navigate to: **Compute → [Cluster Name] → Event Log tab**

### Key Event Types

| Event | Meaning |
|---|---|
| `CREATING` | Cluster provisioning started |
| `RUNNING` | Cluster successfully started and ready |
| `INIT_SCRIPTS_STARTED` | Init scripts beginning execution |
| `INIT_SCRIPTS_FINISHED` | Init scripts completed (check status: success or failure) |
| `DRIVER_HEALTHY` | Driver JVM started successfully |
| `DRIVER_UNAVAILABLE` | Driver process has died |
| `RESIZING` | Autoscaling triggered — node count changing |
| `UPSIZE_COMPLETED` / `DOWNSIZE_COMPLETED` | Autoscaling finished |
| `NODES_LOST` | One or more worker nodes were lost (spot preemption or hardware failure) |
| `TERMINATING` | Cluster is shutting down |
| `TERMINATED` | Cluster fully stopped |

### Diagnosing Startup Failures from Event Log
```
Healthy startup sequence:
CREATING → INIT_SCRIPTS_STARTED → INIT_SCRIPTS_FINISHED (success)
         → DRIVER_HEALTHY → RUNNING

Failed startup sequences:
CREATING → INIT_SCRIPTS_STARTED → INIT_SCRIPTS_FINISHED (failure)
         → DRIVER_UNAVAILABLE → [cluster retries or terminates]

CREATING → DRIVER_UNAVAILABLE (immediately)
         → [init scripts never started — infrastructure issue]
```

## 3. Driver Logs — stdout, stderr, log4j

Navigate to: **Compute → [Cluster Name] → Driver Logs tab**

### Three Log Files

| Log File | Contents | Best For |
|---|---|---|
| **stdout** | Standard output from Spark driver, print statements, notebook output | General execution flow |
| **stderr** | Error messages, stack traces, Java exceptions | Startup errors, OOM errors, library errors |
| **log4j** | Spark internal logging (INFO, WARN, ERROR, DEBUG level) | Spark configuration issues, metastore connectivity, SQL planner |

### Common Error Signatures in Driver Logs

| Error | Location | Meaning |
|---|---|---|
| `ImportError: No module named 'xyz'` | stderr | Python library not installed or wrong environment |
| `ModuleNotFoundError` | stderr | Same as ImportError |
| `pip._internal.exceptions.DistributionNotFound` | stderr | Library install failed |
| `ERROR: Cannot install xyz because these package versions conflict` | stderr | Library version conflict |
| `java.lang.OutOfMemoryError: Java heap space` | stderr | Driver JVM heap exhausted |
| `java.lang.OutOfMemoryError: GC overhead limit exceeded` | stderr | Driver JVM spending >98% of time in GC |
| `SparkContext has been shut down` | stderr/log4j | Driver restarted or lost |
| `Connection refused` | log4j | Metastore connectivity issue |

## 4. Cluster Startup Failures — Diagnostic Workflow

### Step 1: Check Event Log
Look for `INIT_SCRIPTS_FINISHED` with failure status. If no init scripts are configured and the cluster still fails, proceed to check infrastructure.

### Step 2: Isolate Init Scripts & Libraries
```
Test: Remove ALL init scripts and cluster-scoped libraries → restart cluster
├── Cluster starts → Problem is in init scripts or cluster libraries
│       ├── Re-add init scripts one by one
│       └── Re-add libraries one by one until failure reappears
└── Cluster still fails → Infrastructure issue (DBR version, Spark config, metastore)
```

### Step 3: Read Driver Logs (stderr)
Look for the first error message — this is usually the root cause.

### Step 4: Check for DBR Version Compatibility
If failure started after a Databricks Runtime upgrade, check if:
- Init script uses deprecated Spark configs
- Library versions are incompatible with the new Python/Java versions in the DBR

### Step 5: Check Init Script Logs in Cloud Storage
If log delivery is configured:
```
<storage_path>/cluster_id=<id>/init_scripts/<script_name>.log
```
Contains stdout/stderr from each init script execution with timestamps.

## 5. Init Script Failures

Init scripts run **before** Spark starts on each node (driver and workers). A failed init script prevents the cluster from starting.

### How Init Scripts Fail the Cluster
- Any command in the init script that returns a **non-zero exit code** causes the cluster startup to fail
- Shell scripts exit with the last command's exit code by default
- A `pip install` failure, a missing file, a permissions error — all cause non-zero exit codes

### Diagnosing Init Script Failures

**In Event Log:**
```
INIT_SCRIPTS_STARTED
INIT_SCRIPTS_FINISHED  ← status = FAILED (timestamp)
```

**In Init Script Log (cloud storage):**
```bash
[2026-01-15 09:23:45] Installing custom-lib==2.3.1
[2026-01-15 09:23:47] ERROR: Could not find a version that satisfies the requirement custom-lib==2.3.1
[2026-01-15 09:23:47] Exit code: 1
```

### Common Init Script Issues

| Issue | Cause | Fix |
|---|---|---|
| Non-zero exit code | `pip install` fails, command not found | Validate script locally; use `set -e` carefully; check package existence |
| Script hangs | Waiting for network resource, infinite loop | Add timeouts; use `--timeout` in pip |
| Hidden characters (^M) | Script edited on Windows (CRLF line endings) | Run `dos2unix` on the script file |
| Permission errors | Script tries to write to read-only path | Change target directory; use `/tmp/` |
| Wrong environment check | Script runs on all nodes but should only run on driver | Check `DB_IS_DRIVER` environment variable |

### Init Script Best Practice: Check Node Type
```bash
#!/bin/bash
# Only install on driver node
if [[ $DB_IS_DRIVER = "TRUE" ]]; then
  pip install --no-cache-dir custom-package==1.2.3
fi
```

### Validating Init Script Exit Code
```bash
#!/bin/bash
set -e  # Exit immediately if any command fails
pip install --no-cache-dir my-package==1.0.0
echo "Installation successful"
```

## 6. Library Conflicts — Types & Causes

### Type 1: Version Conflict
Two installed libraries require incompatible versions of the same dependency:
```
ERROR: Cannot install library-a 2.0 and library-b 1.5 because they both
require numpy, but library-a needs numpy>=1.24 and library-b requires numpy<1.20
```

### Type 2: Databricks Runtime Core Library Override
DBR includes optimized versions of key libraries (`numpy`, `pandas`, `pyarrow`, `protobuf`). Installing a different version can break the Spark-Python bridge:
```python
# This can break cluster-wide Python if numpy 2.x is incompatible with DBR's internal use
pip install numpy==2.0.0  # May break if DBR depends on numpy 1.x behavior
```

### Type 3: `ImportError` at Runtime (Not at Install)
Library installs successfully but fails to import because:
- Conflicting shared library (`.so` file) versions
- Wrong architecture (ARM vs x86)
- Missing system dependency

### Type 4: Cluster-Scoped vs. Notebook-Scoped Conflict
```python
# Notebook-scoped install (affects only this notebook's Python process)
%pip install requests==2.28.0

# If the cluster already has requests==2.31.0 as a cluster library,
# the notebook-scoped version takes precedence for this notebook only
```

## 7. Library Installation Methods — Precedence & Scope

### Installation Methods (from lowest to highest precedence)

| Method | Scope | Precedence | Best For |
|---|---|---|---|
| **DBR built-in** | All clusters with this DBR | Lowest | Core Spark/ML libraries |
| **Cluster-scoped library** (UI/API) | This cluster only | 2nd | Shared libraries needed by all notebooks on cluster |
| **Init script `pip install`** | This cluster, all nodes | 3rd | System-level or node-wide dependencies |
| **Notebook-scoped `%pip install`** | This notebook session only | Highest | Per-notebook experimental installs |

### Notebook-Scoped Libraries (`%pip install`)
```python
# Installs only for this notebook — does not affect other notebooks on the cluster
%pip install great-expectations==0.18.0

# Restart the Python kernel to pick up the new library
# (happens automatically after %pip install)
```

### Cluster-Scoped Libraries (UI)
- Go to Compute → [Cluster] → Libraries tab → Install
- Supports: PyPI, Maven, CRAN, DBFS/Unity Catalog Volumes (`.whl`, `.jar`, `.egg`)
- Installed on ALL nodes (driver + all workers)
- Applied at cluster start — requires cluster restart to change

### Preferred Storage for Library Files
- Store `.whl` and `.jar` files in **Unity Catalog Volumes** (recommended)
- Not in DBFS `/tmp/` (not persisted across restarts in all configurations)

## 8. Out of Memory (OOM) — Driver OOM

The **driver node** runs the Spark driver JVM, hosts notebook sessions, and collects results from executors.

### Common Causes of Driver OOM

| Operation | Why It Causes Driver OOM |
|---|---|
| `df.collect()` | Pulls ALL data from all executors into driver memory |
| `df.toPandas()` | Same as collect — converts to in-memory Pandas DataFrame on driver |
| Large `broadcast()` | Broadcast data is serialized and kept in driver memory |
| Excessive notebook output | Cell output stored in driver memory |
| `pd.read_csv()` on large file | Reads entire file into driver Python process |

### Diagnosing Driver OOM
```
In Driver Logs (stderr):
java.lang.OutOfMemoryError: Java heap space
    at org.apache.spark.rdd.RDD.collect(RDD.scala:1065)
    ...

OR:
java.lang.OutOfMemoryError: GC overhead limit exceeded
```

### Fixes for Driver OOM

```python
# BAD: Pulls entire 100M row DataFrame to driver
data = df.collect()
pdf = df.toPandas()

# GOOD: Write to storage instead of collecting
df.write.mode("overwrite").parquet("/mnt/output/")

# GOOD: Use take() for small samples
sample = df.take(100)   # Only 100 rows to driver

# GOOD: Use limit() + toPandas() for manageable subsets
sample_pdf = df.limit(1000).toPandas()  # Only 1000 rows

# GOOD: Use Pandas API on Spark (distributed, not collected to driver)
import pyspark.pandas as ps
ps_df = ps.from_pandas(pdf)  # Stays distributed
```

### Driver Memory Configuration
```
spark.driver.memory = 4g   (default — increase for driver-heavy workloads)
spark.driver.maxResultSize = 1g  (limit on result data sent to driver)
```

## 9. Out of Memory (OOM) — Executor OOM

**Executor nodes** run Spark tasks. Each executor has a fixed JVM heap for processing partitions.

### Common Causes of Executor OOM

| Cause | Symptom in Spark UI |
|---|---|
| Data skew — one partition too large | Max task duration >> Median; Max Spill >> Median |
| Too few shuffle partitions | Large Shuffle Read per task (1 GB+ per task) |
| Too large broadcast table | `java.lang.OutOfMemoryError` on many executors simultaneously |
| Excessive caching without unpersist | Storage memory fills → execution memory squeezed |
| UDFs with large state | GC overhead limit exceeded in UDF-heavy stages |

### Diagnosing Executor OOM
```
In Spark UI → Stages → Tasks:
Status: FAILED
Error: java.lang.OutOfMemoryError: Java heap space
       java.lang.OutOfMemoryError: GC overhead limit exceeded
       ExecutorLostFailure (executor 3 exited caused by: SIGKILL)

In Executors Tab:
Peak JVM Memory Used: near or at executor memory limit
GC Time: > 10% of task time
Failed Tasks: non-zero
```

### Fixes for Executor OOM
```python
# Fix 1: Increase shuffle partitions (reduce data per task)
spark.conf.set("spark.sql.shuffle.partitions", "1000")

# Fix 2: Enable AQE to handle skew automatically
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")

# Fix 3: Reduce broadcast threshold (avoid broadcasting too much data)
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "10m")  # 10 MB max

# Fix 4: Unpersist cached DataFrames when done
df_cached = df.cache()
df_cached.count()  # Trigger caching
# ... use df_cached ...
df_cached.unpersist()  # Free executor storage memory
```

### Executor Memory Configuration
```
spark.executor.memory = 8g        (JVM heap for each executor)
spark.executor.cores = 4          (cores per executor = 4 concurrent tasks)
# Per-task memory = executor.memory / executor.cores
# Example: 8g / 4 cores = 2g per task
```

## 10. Memory Overhead Exceeded

**Off-heap memory** is memory used outside the JVM heap — for Python processes (PySpark), native libraries, shuffle metadata, and OS overhead.

### `memoryOverhead` — What It Covers
```
Total executor memory = spark.executor.memory + spark.executor.memoryOverhead
                       = JVM heap             + off-heap (default: max(384MB, 10% of heap))
```

### When memoryOverhead Is Exceeded
```
Error: Container killed by YARN for exceeding memory limits
       13.5 GB of 13.2 GB physical memory used. Consider boosting
       spark.yarn.executor.memoryOverhead
```
Or on Kubernetes/Databricks:
```
ExecutorLostFailure: Container exited because of exceeded memory limits
```

### Common Causes of Overhead Exceeded

| Cause | Reason |
|---|---|
| PySpark UDFs | Python subprocess runs in off-heap memory |
| PySpark `pandas_udf` | Arrow IPC between JVM and Python uses off-heap |
| Large shuffle in Python | PySpark serialization uses off-heap |
| Native libraries (TensorFlow, scikit-learn) | JNI native code runs outside JVM heap |
| Many small Python objects | Python memory is off-heap from JVM perspective |

### Fix: Increase Memory Overhead
```python
# In cluster Spark config or job config:
spark.executor.memoryOverhead = 2g   # Explicitly set (default ~10% of executor.memory)

# Alternative: use memoryOverheadFactor
spark.executor.memoryOverheadFactor = 0.2  # 20% of executor.memory
```

## 11. ExecutorLostFailure — Infrastructure vs. Memory

`ExecutorLostFailure` can have two very different root causes that require different fixes:

### Cause 1: Memory-Based Executor Loss
```
ExecutorLostFailure: executor 5 exited caused by: Container killed by ...
                     because it exceeded memory limits
```
**Fix**: Increase `spark.executor.memory` or `spark.executor.memoryOverhead`

### Cause 2: Infrastructure-Based Executor Loss
```
ExecutorLostFailure: executor 3 exited caused by: Spot instance preemption
                     / EC2 instance termination / Node failure
```
**Fix**: Switch to on-demand instances, or make job fault-tolerant

### How to Distinguish Them — Event Log Check
| Evidence | Root Cause |
|---|---|
| Event Log shows `NODES_LOST` with "spot preemption" | Infrastructure (spot instance) |
| Spark UI shows OOM error in failed task | Memory (executor heap) |
| Event Log shows `Container killed … exceeded memory limits` | Memory overhead |
| Multiple executors lost simultaneously | Infrastructure (zone outage, preemption wave) |
| Single executor lost repeatedly on same data | Data skew causing OOM |

## 12. Cluster Autoscaling & Spot Instance Loss

### Spot Instance Behavior
- **Spot/preemptible instances** are cheaper but can be reclaimed by the cloud provider with little notice
- When a spot instance is reclaimed, Spark's tasks on that executor are lost and must be recomputed
- Databricks automatically reschedules lost tasks — this is transparent for most operations

### Diagnosing Spot Loss vs. Memory OOM
**Event Log indicators:**
- `NODES_LOST` event with message containing "spot" or "preempted" → spot loss
- `RESIZING` without `NODES_LOST` → normal autoscaling
- `DRIVER_UNAVAILABLE` → driver node lost (more severe — job fails entirely)

### Autoscaling Events
```
Event Log:
RESIZING: Cluster scaled from 4 to 8 workers (autoscale up)
UPSIZE_COMPLETED: Now running with 8 workers
...
RESIZING: Cluster scaled from 8 to 4 workers (autoscale down after inactivity)
DOWNSIZE_COMPLETED: Now running with 4 workers
```

## 13. DBR Compatibility Issues

### When DBR Upgrades Break Clusters

After a Databricks Runtime version upgrade, clusters may fail due to:

| Issue | Example |
|---|---|
| Deprecated Spark config | `spark.sql.hive.convertMetastoreParquet` removed in DBR 14+ |
| Library version bump | DBR 14 ships Python 3.11 — some packages not yet compatible |
| Removed class | Java/Scala class removed from Spark, init script fails |
| Changed behavior | Default shuffle partitions changed, causing OOM |

### Checking DBR Compatibility
1. Check Databricks [Release Notes](https://docs.databricks.com/release-notes/runtime/releases.html) for breaking changes
2. Look for `WARN` or `ERROR` in log4j logs for deprecated config warnings
3. Compare DBR Python version: DBR 13 = Python 3.10, DBR 14 = Python 3.11, DBR 15+ = Python 3.12+

### Finding DBR Version in Cluster Config
```bash
# Via environment variable in init scripts or notebooks:
echo $DATABRICKS_RUNTIME_VERSION
# Output: 15.4.x-scala2.12

# Via spark.version:
spark.version   # e.g., "3.5.0"
```

## 14. Diagnostic Decision Tree

```
CLUSTER FAILS TO START
        │
        ▼
Check Event Log
        │
        ├── INIT_SCRIPTS_FINISHED (FAILED)?
        │       YES → Check init script logs in cloud storage
        │       ├── Non-zero exit code → Fix the failing command
        │       ├── Package not found → Fix version/package name
        │       └── Network timeout → Fix proxy / add --timeout
        │
        ├── DRIVER_UNAVAILABLE immediately (no init scripts)?
        │       YES → Infrastructure issue
        │       ├── Check Spark config for deprecated settings
        │       ├── Check DBR compatibility
        │       └── Contact Databricks support if persistent
        │
        └── Cluster starts but NOTEBOOK fails?
                YES → Library or environment issue
                ├── ImportError → Library not installed (cluster library vs. notebook scope)
                └── Version conflict → Use %pip install or fix cluster library

CLUSTER RUNNING BUT JOB FAILS
        │
        ▼
Check Spark UI → Stages → Failed Tasks
        │
        ├── java.lang.OutOfMemoryError on driver?
        │       YES → Driver OOM
        │       ├── Look for collect(), toPandas() in code
        │       └── Increase spark.driver.memory or spark.driver.maxResultSize
        │
        ├── java.lang.OutOfMemoryError on executor?
        │       YES → Executor OOM
        │       ├── Skew? (Max >> Median task duration) → AQE, salting
        │       ├── Large partitions? → Increase shuffle partitions
        │       └── Increase spark.executor.memory or memoryOverhead
        │
        └── ExecutorLostFailure?
                ├── Check Event Log for NODES_LOST + spot preemption → Infrastructure
                └── Check Spark UI for OOM error → Memory
```

## 15. Prevention Best Practices

| Issue | Prevention |
|---|---|
| Init script failures | Test on dev cluster first; use `set -e`; avoid overriding DBR core libs |
| Library conflicts | Use notebook-scoped `%pip install` for isolation; pin exact versions |
| Driver OOM | Avoid `collect()`/`toPandas()` on large DataFrames; use `limit()` |
| Executor OOM | Enable AQE; tune shuffle partitions; use memory-optimized instances for heavy workloads |
| Memory overhead | Increase `memoryOverhead` for PySpark UDF-heavy jobs |
| Spot instance loss | Use on-demand for driver; use spot only for workers; ensure tasks are idempotent |
| DBR upgrade breaks | Test on dev cluster before upgrading production; read release notes |
| Core library conflict | Don't override DBR's `numpy`, `pandas`, `pyarrow` unless absolutely necessary |

## 1. Core Concepts Overview

### What is Control Flow in Lakeflow Jobs?
Lakeflow Jobs supports custom control flow logic using a visual authoring UI and code-based configuration tools. Jobs can go beyond a simple linear sequence of tasks to include:

| Mechanism | Task Type / Field | Use Case |
|---|---|---|
| **Retries** | `max_retries`, `min_retry_interval_millis` per task | Re-run a failed task automatically (transient errors, schema evolution) |
| **Dependency conditions** | `run_if` on any task | Run a downstream task only when upstream tasks have a specific outcome |
| **If/else branching** | `condition_task` | Fork the DAG based on a boolean expression evaluated at runtime |
| **Looping** | `for_each_task` | Run a nested task repeatedly, once per item in a list |
| **Disabled tasks** | task disable toggle | Skip a task without removing it from the DAG or losing its history |

### DAG Execution Model
- Tasks are visualized as a **Directed Acyclic Graph (DAG)**; tasks with no unresolved dependencies run **in parallel** by default.
- Databricks runs upstream tasks before downstream tasks, running as many of them in parallel as possible.
- A task is marked `Excluded` (skipped and treated as successful) when its `Run if` condition evaluates to unmet and the condition type handles failures.
- A task is marked `Upstream failed` when its `Run if` condition is unmet and the condition type handles success-based gates.

## 2. Retries & Timeouts

### When to Use Retries
Retries specify how many times a particular task should be re-run if the task fails with an error message. Errors are often transient and resolved through restart. Some features on Databricks, such as schema evolution with Structured Streaming, assume that you run jobs with retries to reset the environment and allow a workflow to proceed.

### Default Retry Behavior by Job Mode
| Mode | Default Behavior |
|---|---|
| **Standard (scheduled / manual)** | No retries by default. Configure explicitly per task. |
| **Serverless** | Auto-optimizes retries by default. Can be disabled per task with `disable_auto_optimization: true`. |
| **Continuous** | Cannot use manual retry policies. Automatically retries the entire job using an exponential backoff algorithm. |

### Continuous Mode — Exponential Backoff
You cannot use retry policies in a continuous job. Instead, continuous jobs automatically retry the entire job on failure using an exponential backoff algorithm. You can additionally configure retries at the task level by setting the **Task retry mode** to **On failure**. When Task retry mode is set to On failure, failed tasks are retried with an exponentially increasing delay until the maximum number of allowed retries is reached (three for a single task job). After the maximum retries are reached, the run is canceled, and a new run is triggered.

For each subsequent job run failure, the retry period is increased up to a maximum retry period set by the system. After reaching the maximum retry period, the job continues to be retried using the maximum retry period. There is no limit on the number of retries for a continuous job. If the job run completes successfully and starts a new run, or if the run exceeds a threshold without failure, the job is considered healthy, and the backoff sequence resets.

### Retry Configuration Fields (REST API / DAB)
| Field | Type | Description |
|---|---|---|
| `max_retries` | integer | Maximum number of retry attempts on failure. `0` = no retries, `-1` = unlimited (legacy). |
| `min_retry_interval_millis` | integer | Minimum time (ms) between the start of the failed run and the next retry attempt. |
| `retry_on_timeout` | boolean | If `true`, the task is also retried when it times out. Default: `false`. |
| `timeout_seconds` | integer | Maximum seconds a task run is allowed. `0` = no timeout. When both timeout and retries are set, the timeout applies to **each retry**. |
| `disable_auto_optimization` | boolean | Serverless only. Set `true` to opt out of the serverless auto-retry behavior. |

> [!IMPORTANT]
> If you configure both **Timeout** and **Retries**, the timeout applies to **each individual retry**, not the total wall-clock time.

### UI Configuration (Retries)
1. Open the job → **Tasks** tab → select the task
2. Scroll to **Advanced task settings** → click **+ Add** next to **Retries**
3. Set:
   - **Max retries** (number of attempts after initial failure)
   - **Retry interval** (milliseconds between retries, measured from the start of the failed run)
   - **Retry on timeout** (checkbox)
4. Click **Save task**

### REST API Snippet — Task with Retries
```json
{
  "task_key": "ingest_data",
  "notebook_task": {
    "notebook_path": "/Workspace/Shared/ingest"
  },
  "max_retries": 3,
  "min_retry_interval_millis": 300000,
  "retry_on_timeout": false,
  "timeout_seconds": 3600
}
```

### DAB YAML Snippet — Retries
```yaml
tasks:
  - task_key: ingest_data
    notebook_task:
      notebook_path: ../src/ingest.ipynb
    max_retries: 3
    min_retry_interval_millis: 300000
    retry_on_timeout: false
    timeout_seconds: 3600
```

### Serverless — Disable Auto-Optimization
To opt a specific task out of serverless automatic retries (e.g. when re-running causes data duplication):
```json
{
  "task_key": "transform_silver",
  "notebook_task": { "notebook_path": "/Workspace/Shared/transform" },
  "disable_auto_optimization": true
}
```

## 3. Run If — Dependency-Based Branching

### What It Does
The **Run if dependencies** field allows you to add control flow logic to tasks based on other tasks' success, failure, or completion. Dependencies are visually represented in the job DAG as lines between tasks.

This is different from the If/else task (Section 4): **Run if** reacts to the *outcome* of upstream tasks (pass/fail/done), while **If/else** evaluates a *value* expression.

### `Run if` Condition Options
| Condition | Behavior when unmet | Use Case |
|---|---|---|
| **All succeeded** *(default)* | Task marked `Upstream failed` | Normal happy-path chaining |
| **At least one succeeded** | Task marked `Upstream failed` | Fan-in: continue if any branch succeeded |
| **None failed** | Task marked `Upstream failed` | At least one ran; none errored (allows skipped upstreams) |
| **All done** | Never skipped — always runs | Cleanup / notification tasks that must always run |
| **At least one failed** | Task marked `Excluded` | Error-handling / alerting tasks |
| **All failed** | Task marked `Excluded` | Run only when every upstream branch failed |

> [!NOTE]
> Skipped (`Excluded`) upstream tasks are treated as **successful** when evaluating `Run if` conditions.
> If you cancel a task run, the cancellation propagates through downstream tasks, but tasks with a `Run if` condition that handles failure are still run (e.g. cleanup tasks).

### UI Configuration (Run If)
1. Select a task in the DAG → **Depends on** field: add upstream tasks
2. Select one of the conditional options in the **Run if dependencies** field
3. Click **Save task**

### Classic Pattern: Always-Run Cleanup Task
```
ingest → transform → [Run if: All succeeded] → load_to_prod
                 ↘ [Run if: All done]     → cleanup_temp_tables
```
The `cleanup_temp_tables` task uses `All done` so it runs whether `transform` succeeded or failed, while `load_to_prod` only runs on success.

### REST API Snippet — Run If
```json
{
  "task_key": "send_failure_alert",
  "depends_on": [
    { "task_key": "ingest_data" },
    { "task_key": "transform_data" }
  ],
  "run_if": "AT_LEAST_ONE_FAILED",
  "notebook_task": {
    "notebook_path": "/Workspace/Shared/alert_on_failure"
  }
}
```

### `run_if` Enum Values (API)
```
ALL_SUCCESS | AT_LEAST_ONE_SUCCEEDED | NONE_FAILED | ALL_DONE | AT_LEAST_ONE_FAILED | ALL_FAILED
```

### DAB YAML — Full Example with Run If Branching
```yaml
resources:
  jobs:
    etl_with_cleanup:
      name: etl-with-cleanup
      tasks:
        - task_key: ingest
          notebook_task:
            notebook_path: ../src/ingest.ipynb

        - task_key: transform
          depends_on:
            - task_key: ingest
          run_if: ALL_SUCCESS
          notebook_task:
            notebook_path: ../src/transform.ipynb

        - task_key: load_to_prod
          depends_on:
            - task_key: transform
          run_if: ALL_SUCCESS
          notebook_task:
            notebook_path: ../src/load.ipynb

        - task_key: cleanup_temp
          depends_on:
            - task_key: transform
          run_if: ALL_DONE          # Runs regardless of transform success/failure
          sql_task:
            warehouse_id: 1a111111a1111aa1
            file:
              path: ../sql/cleanup.sql
              source: WORKSPACE

        - task_key: alert_on_failure
          depends_on:
            - task_key: ingest
            - task_key: transform
          run_if: AT_LEAST_ONE_FAILED
          notebook_task:
            notebook_path: ../src/send_alert.ipynb
```

## 4. If/Else Condition Task — Value-Based Branching

### What It Does
Use the `If/else condition` task to add boolean conditional logic to task graphs. These tasks consist of a boolean operator and a pair of operands, where the operands might reference the job or task state using configured or dynamic parameters or task values.

The condition task does **not** require a cluster to execute and does **not** support retries or notifications.

### Supported Operands and Type Behavior
| Operator | Behavior |
|---|---|
| `==` | String comparison. `12.0 == 12` → **false** |
| `!=` | String comparison. |
| `>`, `>=`, `<`, `<=` | Numeric comparison. `12.0 >= 12` → **true** |

> [!NOTE]
> Only numeric, string, and boolean values are allowed when referencing task values in an operand. Non-numeric values are serialized to strings. For example, a boolean task value becomes `"true"` or `"false"` as a string before evaluation.

### What Can Be Compared
The left operand can reference any of the following:
- A job parameter: `{{job.parameters.<name>}}`
- A task parameter variable
- A task value from an upstream task: `{{tasks.<task_name>.values.<value_name>}}`
- A dynamic value reference: `{{job.run_id}}`, `{{job.repair_count}}`, etc.

### UI Configuration (If/Else)
1. Add task → **Type** → select `If/else condition`
2. First **Condition** text box: enter left operand (e.g. `{{tasks.process_records.values.bad_records}}`)
3. Select boolean operator from drop-down
4. Second **Condition** text box: enter comparison value (e.g. `0`)
5. Click **Save task**
6. Select the `If/else condition` task in the graph → click **Add task**
   - New task defaults to `<task-name> (true)` — the branch for when the condition is **true**
   - Change to `<task-name> (false)` for the **false** branch

### Example: Branch on Record Quality
Suppose a task `process_records` sets a task value `bad_records`. Run a quality fix only if bad records exist:
```
process_records → check_bad_records (If/else: {{tasks.process_records.values.bad_records}} > 0)
                       ↓ (true)            ↓ (false)
               run_data_fix           load_to_prod
```

The notebook code in `process_records`:
```python
bad_count = df.filter(col("is_invalid") == True).count()
dbutils.jobs.taskValues.set(key="bad_records", value=bad_count)
```

### DAB YAML — If/Else Task
```yaml
resources:
  jobs:
    quality_gate_job:
      name: quality-gate-job
      tasks:
        - task_key: process_records
          notebook_task:
            notebook_path: ../src/process_records.ipynb

        - task_key: check_bad_records
          depends_on:
            - task_key: process_records
          condition_task:
            op: GREATER_THAN
            left: '{{tasks.process_records.values.bad_records}}'
            right: '0'

        # True branch: fix data quality issues
        - task_key: run_data_fix
          depends_on:
            - task_key: check_bad_records
              outcome: 'true'
          notebook_task:
            notebook_path: ../src/fix_quality.ipynb

        # False branch: proceed directly to load
        - task_key: load_to_prod
          depends_on:
            - task_key: check_bad_records
              outcome: 'false'
          notebook_task:
            notebook_path: ../src/load.ipynb
```

### REST API Snippet — Condition Task
```json
{
  "task_key": "check_bad_records",
  "depends_on": [{ "task_key": "process_records" }],
  "condition_task": {
    "op": "GREATER_THAN",
    "left": "{{tasks.process_records.values.bad_records}}",
    "right": "0"
  }
}
```

### `op` Enum Values (API)
```
EQUAL_TO | NOT_EQUAL | GREATER_THAN | GREATER_THAN_OR_EQUAL | LESS_THAN | LESS_THAN_OR_EQUAL
```

### Another Example: Retry Safeguard (Loop Limit)
Use `{{job.repair_count}}` to prevent infinite manual repairs:
```yaml
- task_key: check_repair_limit
  condition_task:
    op: LESS_THAN
    left: '{{job.repair_count}}'
    right: '5'

- task_key: retry_etl
  depends_on:
    - task_key: check_repair_limit
      outcome: 'true'
  notebook_task:
    notebook_path: ../src/etl.ipynb
```

## 5. For Each Task — Looping

### What It Does
Use the `For each` task to run a nested task in a loop, passing a different set of parameters to each iteration of the task.

Adding a `For each` task to a job requires defining two tasks: the `For each` task and a **nested task**. The nested task is the task to run for each iteration and is one of the standard Lakeflow Jobs task types. You cannot add another `For each` task as the nested task.

Nested tasks that do not have dependencies on each other can be run concurrently.

### UI Configuration (For Each)
1. Add task → **Type** → select `For each`
2. Enter **Task name**
3. In the **Inputs** text box: define the values as a JSON-formatted array of values to iterate over
4. Set **Concurrency** (default: `1`): number of iterations that can run in parallel
5. Click **Add a task to loop over**
6. Configure the nested task type and parameters
7. Use `{{input}}` to reference the current array element, or `{{input.<key>}}` to reference individual fields from objects

### Parameter Types for the For Each Task
| Source | Inputs Syntax | Max Size |
|---|---|---|
| Literal JSON array | `["us-east-1", "eu-west-1", "ap-south-1"]` | 5,000 chars in Inputs box |
| Task value from upstream | `{{tasks.<task_name>.values.<value_name>}}` | 48 KB (the referenced value) |
| Job parameter | `{{job.parameters.<name>}}` | 10 KB |

> [!NOTE]
> Parameters are limited to 5,000 characters in the Inputs text box directly, or 48 KB if you use task value references (where the value can be much larger). If your parameters require more than 48 KB, you can pass a lookup to a larger config file.

### `{{input}}` Reference Syntax
| Inputs type | Access syntax in nested task |
|---|---|
| Array of strings | `{{input}}` → the current string value |
| Array of objects | `{{input.key_name}}` → value of a specific key |
| Array of numbers | `{{input}}` → the current number |

### Example 1: Loop Over Regions (Static Array)
```yaml
resources:
  jobs:
    regional_etl:
      name: regional-etl
      tasks:
        - task_key: process_all_regions
          for_each_task:
            inputs: '["us-east-1", "eu-west-1", "ap-south-1"]'
            concurrency: 3           # Run all 3 in parallel
            task:
              task_key: process_region
              notebook_task:
                notebook_path: ../src/process_region.ipynb
                base_parameters:
                  region: '{{input}}'
```

### Example 2: Loop Over Objects (Dynamic List from Upstream Task)
```yaml
resources:
  jobs:
    dynamic_table_processing:
      name: dynamic-table-processing
      tasks:
        # Step 1: generate the list of tables
        - task_key: generate_table_list
          notebook_task:
            notebook_path: ../src/get_tables.ipynb

        # Step 2: loop over the list
        - task_key: process_each_table
          depends_on:
            - task_key: generate_table_list
          for_each_task:
            # References a task value set in generate_table_list
            inputs: '{{tasks.generate_table_list.values.tables}}'
            concurrency: 5
            task:
              task_key: transform_table
              notebook_task:
                notebook_path: ../src/transform.ipynb
                base_parameters:
                  table_name: '{{input.name}}'
                  schema:     '{{input.schema}}'
```

The upstream notebook (`get_tables.ipynb`) sets the task value:
```python
tables = [
    {"name": "orders",    "schema": "sales"},
    {"name": "customers", "schema": "crm"},
    {"name": "products",  "schema": "catalog"}
]
dbutils.jobs.taskValues.set(key="tables", value=tables)
```

### Example 3: Metadata-Driven SQL Loop (Control Table Pattern)
```yaml
tasks:
  - task_key: read_control_table
    sql_task:
      warehouse_id: 1a111111a1111aa1
      query:
        query_id: 11111111-1111-1111-1111-111111111111  # Returns list as task value

  - task_key: process_each_entity
    depends_on:
      - task_key: read_control_table
    for_each_task:
      inputs: '{{tasks.read_control_table.values.entities}}'
      concurrency: 2
      task:
        task_key: run_entity_pipeline
        notebook_task:
          notebook_path: ../src/entity_pipeline.ipynb
          base_parameters:
            entity_id:   '{{input.id}}'
            entity_type: '{{input.type}}'
```

### Monitoring For Each Runs
Running a job with a `For each` task is identical to running any other job. The task run history for a `For each` task is presented as a **table of task iterations**, showing the status and duration of each individual loop iteration.

### REST API Snippet — For Each Task
```json
{
  "task_key": "process_all_regions",
  "for_each_task": {
    "inputs": "[\"us-east-1\", \"eu-west-1\", \"ap-south-1\"]",
    "concurrency": 3,
    "task": {
      "task_key": "process_region",
      "notebook_task": {
        "notebook_path": "/Workspace/Shared/process_region",
        "base_parameters": {
          "region": "{{input}}"
        }
      }
    }
  }
}
```

## 6. Disabled Tasks

### What It Does
Disable a task to skip it at run time without removing it from the job. The task keeps its configuration and run history, so you can re-enable it later without rebuilding the task.

Common scenarios:
- Temporarily excluding a task while debugging an upstream issue without losing the task's settings.
- Pausing a broken task so the rest of the job continues to run on schedule.
- Keeping the job's DAG and run history intact while deciding whether to remove a task.

### Downstream Task Behavior When a Task Is Disabled
Lakeflow Jobs evaluates downstream tasks against their `Run if` conditions to determine whether they also run. An `If/else condition` task fails if the upstream task that provides its condition value is disabled. Similarly, a `For each` task fails if the upstream task that provides its input values is disabled.

### UI — Enable / Disable
- **Disable**: Select the task in the DAG → click the circle-off icon
- **Re-enable**: Select the task → click the check-circle icon
- **Skip for a single run only** (without modifying the job): use **Run now with different settings** instead

## 8. Python SDK — Control Flow Examples

> Install: `pip install databricks-sdk`

```python
from databricks.sdk import WorkspaceClient
from databricks.sdk.service import jobs

w = WorkspaceClient()

# --- Job with retries, Run if, If/else, and For each ---
created = w.jobs.create(
    name="control-flow-example",
    tasks=[
        # Task 1: ingest with retries
        jobs.Task(
            task_key="ingest",
            notebook_task=jobs.NotebookTask(
                notebook_path="/Workspace/Shared/ingest"
            ),
            max_retries=3,
            min_retry_interval_millis=60000,
            retry_on_timeout=False,
            timeout_seconds=1800,
        ),

        # Task 2: If/else — check row count from task value
        jobs.Task(
            task_key="check_row_count",
            depends_on=[jobs.TaskDependency(task_key="ingest")],
            run_if=jobs.RunIf.ALL_SUCCESS,
            condition_task=jobs.ConditionTask(
                op=jobs.ConditionTaskOp.GREATER_THAN,
                left="{{tasks.ingest.values.row_count}}",
                right="0",
            ),
        ),

        # Task 3a: True branch — loop over regions
        jobs.Task(
            task_key="process_regions",
            depends_on=[
                jobs.TaskDependency(task_key="check_row_count", outcome="true")
            ],
            for_each_task=jobs.ForEachTask(
                inputs='["us-east-1", "eu-west-1"]',
                concurrency=2,
                task=jobs.ForEachNestedTask(
                    task_key="process_region",
                    notebook_task=jobs.NotebookTask(
                        notebook_path="/Workspace/Shared/process_region",
                        base_parameters={"region": "{{input}}"},
                    ),
                ),
            ),
        ),

        # Task 3b: False branch — log no data
        jobs.Task(
            task_key="skip_no_data",
            depends_on=[
                jobs.TaskDependency(task_key="check_row_count", outcome="false")
            ],
            notebook_task=jobs.NotebookTask(
                notebook_path="/Workspace/Shared/log_no_data"
            ),
        ),

        # Task 4: Cleanup — always runs (All done)
        jobs.Task(
            task_key="cleanup",
            depends_on=[
                jobs.TaskDependency(task_key="process_regions"),
                jobs.TaskDependency(task_key="skip_no_data"),
            ],
            run_if=jobs.RunIf.ALL_DONE,
            sql_task=jobs.SqlTask(
                warehouse_id="1a111111a1111aa1",
                file=jobs.SqlTaskFile(
                    path="/Workspace/Shared/cleanup.sql",
                    source=jobs.Source.WORKSPACE,
                ),
            ),
        ),

        # Task 5: Alert — runs if ingest failed
        jobs.Task(
            task_key="alert_on_failure",
            depends_on=[jobs.TaskDependency(task_key="ingest")],
            run_if=jobs.RunIf.AT_LEAST_ONE_FAILED,
            notebook_task=jobs.NotebookTask(
                notebook_path="/Workspace/Shared/alert"
            ),
        ),
    ],
)
print(f"Created job: {created.job_id}")
```

            - task_key: ingest

            op: GREATER_THAN

## 10. Task Values — Passing State Between Tasks

Task values are the primary mechanism for feeding runtime state (e.g. row counts, table names, region lists) into If/else and For each conditions.

### Setting a Task Value (Python / Notebook)
```python
# Set an integer count
bad_count = df.filter(col("is_invalid")).count()
dbutils.jobs.taskValues.set(key="bad_records", value=bad_count)

# Set a list for a For each task
tables = [{"name": "orders", "schema": "sales"}, {"name": "customers", "schema": "crm"}]
dbutils.jobs.taskValues.set(key="tables", value=tables)
```

### Reading a Task Value in a Downstream Task
```python
# In the downstream task:
bad_records = dbutils.jobs.taskValues.get(
    taskKey="process_records",
    key="bad_records",
    default=0
)
```

### Referencing a Task Value in Configuration (Dynamic Value Reference)
```
{{tasks.<upstream_task_key>.values.<value_name>}}
```

### Examples by Use Case
| Use Case | Expression |
|---|---|
| If/else on row count | `{{tasks.ingest.values.row_count}}` |
| For each on a table list | `{{tasks.generate_list.values.tables}}` |
| Repair count guard | `{{job.repair_count}}` |
| Current job run ID | `{{job.run_id}}` |
| Job parameter | `{{job.parameters.env}}` |

## 11. Control Flow Decision Guide & Best Practices

### Which Mechanism to Use?
| Scenario | Use |
|---|---|
| Retry on transient failure (network, rate limit) | **Retries** (`max_retries` + `min_retry_interval_millis`) |
| Run cleanup/notification regardless of outcome | **Run if: All done** |
| Run error-handling task only when something failed | **Run if: At least one failed** |
| Skip downstream if upstream was skipped or failed | **Run if: All succeeded** (default) |
| Fork DAG based on a computed value at runtime | **If/else condition task** |
| Run the same task logic against a dynamic list | **For each task** |
| Temporarily skip a task without deleting it | **Disabled task** |

### Retries Best Practices
- Match `min_retry_interval_millis` to the expected recovery time of the transient error (e.g. 5 min = `300000` ms for quota exhaustion).
- Set `retry_on_timeout: false` unless you have evidence the operation eventually succeeds given more time.
- Use `disable_auto_optimization: true` on serverless tasks that are not idempotent (to prevent double-writes on auto-retry).
- Do not combine `max_retries` with very short `timeout_seconds`; remember the timeout applies per retry attempt.

### Branching Best Practices
- Prefer **Run if** over **If/else** when the decision is purely about whether an upstream task succeeded or failed — it is simpler and requires no separate task node.
- Use **If/else** when the decision depends on a *value* produced at runtime (record counts, flags, thresholds).
- After a job run with an **If/else condition** task, view the result and the expression evaluation details in the job run detail UI.
- An `If/else condition` task **fails** if the upstream task that provides its condition value is disabled — account for this when disabling tasks.

### For Each Best Practices
- Set **concurrency > 1** when iterations are independent and compute resources allow parallel execution.
- Use **task value references** (`{{tasks.<name>.values.<key>}}`) instead of static literal arrays for lists generated at runtime — they support up to 48 KB vs. 5,000 chars for inline literals.
- Downstream tasks can depend on the `For each` task (the outer wrapper), not on the nested task directly.
- Nested tasks inherit retry settings; configure `max_retries` on the nested task definition for per-iteration retries.

### General Patterns
```
Pattern: ETL with cleanup and alerting
ingest (retries=3) → transform → [All succeeded] → load
                                ↘ [All done]      → cleanup
                   ↘ [At least one failed]         → alert

Pattern: Quality gate with If/else
ingest → check_quality (If/else: row_count > 0)
              ↓ true                 ↓ false
   process_regions (For each)   log_no_data

Pattern: Idempotent loop
generate_list → for_each [concurrency=5]
                      ↓ nested: process_item (retries=2)
```

## 1. Core Concepts Overview

### What is a Trigger?
A **trigger** is the mechanism that initiates a Lakeflow Jobs run. Triggers can be time-based (a schedule), event-based (a file arriving, a table changing), continuous (re-run on completion/failure), or manual (Run now / API / external orchestrator).

### Trigger Types at a Glance
| Trigger Type | Behavior | Configured Via |
|---|---|---|
| **Scheduled** | Runs the job on a time-based schedule (simple interval or Quartz cron) | `schedule` (legacy) or `trigger.periodic` / `trigger.cron` (Jobs API) |
| **Table update** | Runs the job when one or more monitored Unity Catalog tables/views are updated | `trigger.table_update` |
| **File arrival** | Runs the job when new files land in a Unity Catalog external location or volume | `trigger.file_arrival` |
| **Continuous** | Keeps a job always running by starting a new run whenever the previous run completes or fails | `continuous` |
| **None (manual)** | Runs are started only via **Run now**, the REST API, or an external orchestrator (e.g., Airflow) | No trigger configured |

> [!NOTE]
> Some workspaces also expose a **Model update** trigger type (fires when a Unity Catalog model version is created or updated) in the **Schedules & Triggers** UI. This guide focuses on the three most common automated trigger types: **scheduled, file arrival, table update**.

### Where Triggers Are Configured
In the workspace UI: open the job → **Job details** pane → **Schedules & Triggers** section → **Add trigger** → choose the trigger type (**Scheduled**, **Table update**, **File arrival**, **Model update**, or **Continuous**) → configure type-specific options → **Save**.

### Concurrency Behavior
By default, only a single run of a job can be active at a time; additional trigger fires are skipped while a run is active unless you raise the **max concurrent runs** setting in Advanced settings.

## 2. Scheduled Trigger

### What It Does
Runs a job on a time-based schedule. This is the most common trigger type for predictable, periodic workloads (e.g., nightly ETL).

### Two Schedule Modes
- **Simple**: Specify an interval and a unit of time (minutes/hours/days/weeks) to run the job periodically. You cannot pick the exact time of the first run — the scheduler chooses it automatically.
- **Advanced**: Specify the period, starting time, and time zone, with the option to view/edit the underlying **Quartz Cron Syntax** directly.

### UI Configuration
1. Workspace sidebar → **Jobs & Pipelines** → open your job
2. **Job details** pane → **Add trigger**
3. **Trigger type** → **Scheduled**
4. **Schedule type** → **Simple** or **Advanced**
5. For Advanced, optionally check **Show Cron Syntax** to edit the raw Quartz expression
6. Click **Save**

### Important Behavior Notes
- Databricks enforces a **minimum interval of 10 seconds** between subsequent scheduled runs of a job, regardless of the cron expression.
- Choose a time zone that observes daylight saving time, or **UTC** — DST-observing zones can cause hourly jobs to be skipped or delayed by an hour around DST transitions; use UTC to run at an absolute hourly cadence.
- The job scheduler is **not intended for low-latency** workloads — network or cloud issues can delay runs by several minutes; delayed runs fire as soon as the service is available again.
- You can also schedule a job directly from the **notebook UI** when scheduling a single notebook.

### REST API Body Snippet (legacy `schedule` field)
```json
{
  "name": "nightly-etl-job",
  "schedule": {
    "quartz_cron_expression": "0 0 6 * * ?",
    "timezone_id": "UTC",
    "pause_status": "UNPAUSED"
  },
  "tasks": [
    { "task_key": "ingest", "notebook_task": { "notebook_path": "/Workspace/Users/user@company.com/ingest" } }
  ]
}
```

### REST API Body Snippet (`trigger.periodic`)
```json
{
  "name": "hourly-pipeline-refresh",
  "trigger": {
    "pause_status": "UNPAUSED",
    "periodic": {
      "interval": 1,
      "unit": "HOURS"
    }
  }
}
```

> [!NOTE]
> A job can define either the legacy `schedule` field (cron-based, simplest for periodic schedules) or the newer unified `trigger` object, which also supports `periodic`, `file_arrival`, and `table_update` sub-configurations. New jobs that need event-based triggers must use `trigger`.

## 3. File Arrival Trigger

### What It Does
Triggers a job run when new files arrive in a Unity Catalog **external location** or **volume** (e.g., Amazon S3, Azure Storage, Google Cloud Storage). Useful when a fixed schedule is inefficient because data arrives irregularly.

### How It Works
- File arrival triggers make a **best-effort check for new files every minute**, subject to underlying cloud storage performance.
- They incur **no additional Databricks cost** beyond the cloud provider's cost of listing files.
- A trigger can monitor the **root** or a **subpath** of an external location or volume, and it **recursively checks all subdirectories** under the configured path.
- For optimal performance, enable **file events** on the external location — this lets Databricks track ingestion metadata from cloud-provider change notifications instead of repeatedly listing the bucket, improving both file arrival trigger and Auto Loader efficiency.

### Prerequisites
- Workspace must have **Unity Catalog** enabled.
- The storage location must be a Unity Catalog **volume** or **external location**.
- You need **READ** permission on the storage location and **CAN MANAGE** permission on the job.

### UI Configuration
1. **Jobs & Pipelines** → open your job
2. **Job details** pane → **Add trigger**
3. **Trigger type** → **File arrival**
4. **Storage location** → URL of the root or subpath of a UC external location or volume
5. (Optional) **Advanced** options:
   - **Minimum time between triggers in seconds** — cooldown; caps run frequency by waiting this long after a run completes before starting another, even if files keep arriving.
   - **Wait after last change in seconds** — debounce; waits this long after the most recent file arrival before starting a run, and each new arrival resets the timer (minimum allowed value: 60 seconds). Useful for batched file drops.
6. Click **Test connection** to validate, then **Save**

### Processing Arrived Files (Auto Loader)
```python
# Configuration
file_location = "[REPLACE]"        # same URL configured on the file arrival trigger
checkpoint_location = "[REPLACE]"  # separate location for the Auto Loader checkpoint
sink_table = "[REPLACE]"

streamingQuery = (
    spark.readStream.format("cloudFiles")
    .option("cloudFiles.format", "json")
    .option("cloudFiles.schemaLocation", checkpoint_location)
    .option("cloudFiles.useManagedFileEvents", "true")
    .load(file_location)
    .writeStream
    .option("checkpointLocation", checkpoint_location)
    .trigger(availableNow=True)
    .toTable(sink_table)
)
```

### Limitations
- Only **new** files trigger a run — overwriting an existing file with the same name does **not**.
- The monitored path **cannot** contain external tables or managed catalog/schema locations, and **cannot** use wildcards (`*`, `?`).
- Without file events enabled: max **50 jobs** per workspace can use a file arrival trigger on a given location, and the location is capped at **10,000 files** (limit applies to the configured subpath, not necessarily the whole root).
- With file events enabled: no hard file-count limit, but triggers can time out under very high extraneous file churn; isolate the target path via a dedicated UC volume if this occurs.
- On S3/GCS, a file arrival trigger pointed at a non-existent or deleted path **does not error** — it just finds no files until the path exists again, since S3/GCS don't distinguish empty/non-existent/deleted directories.

### REST API Body Snippet (`trigger.file_arrival`)
```json
{
  "name": "process-new-uploads",
  "trigger": {
    "pause_status": "UNPAUSED",
    "file_arrival": {
      "url": "/Volumes/mycatalog/myschema/myvolume/incoming/",
      "min_time_between_triggers_seconds": 60,
      "wait_after_last_change_seconds": 300
    }
  },
  "tasks": [
    { "task_key": "load_files", "notebook_task": { "notebook_path": "/Workspace/Users/user@company.com/load_files" } }
  ]
}
```

## 4. Table Update Trigger

### What It Does
Triggers a job run when one or more **source tables are updated**, removing the need for a continuously running cluster or knowledge of the upstream process that updates the table.

> [!NOTE]
> Table update triggers became **generally available** in Lakeflow Jobs; triggering on OpenSharing tables/views and on system tables is currently **Beta**.

### How It Works
- Monitors **Unity Catalog Delta and Iceberg managed tables**, UC **external tables backed by Delta Lake**, **materialized views**, **streaming tables**, and UC **views/metric views** that depend on supported tables.
- Can also monitor **OpenSharing** tables/views/materialized views/streaming tables/metric views, and **system tables** (Beta; requires enabling the relevant workspace- and/or account-level beta).
- When monitoring multiple tables, you choose whether the trigger fires when **All tables are updated** or when **Any table is updated**.
- Incurs **no additional cost** beyond cloud provider costs for listing tables/reading storage updates.
- For best performance, **enable file events** on the external location backing the monitored tables — this also improves Auto Loader and file arrival trigger efficiency. Tables in metastore root-level storage must first be moved to an external location before file events can be enabled.

### UI Configuration
1. **Jobs & Pipelines** → open your job
2. **Job details** pane → **Add trigger**
3. **Trigger type** → **Table update**
4. **Tables** → add the tables to monitor
5. If multiple tables selected, set **Trigger when**: **All tables are updated** or **Any table is updated**
6. (Optional) **Advanced** options:
   - **Minimum time between triggers in seconds** — minimum wait after a previous run completes before triggering another, even if monitored tables update sooner.
   - **Wait after last change in seconds** — wait period after a table update before starting a run; further updates during this window reset the timer.
   - If **both** are set: the trigger waits for the minimum-time window first, then applies the wait-after-last-change debounce on top of that.
7. Click **Test trigger** to validate, then **Save**

### Limitations
- Up to **10** managed/Delta tables per trigger (views' dependent tables count toward this limit too).
- Without file events, a maximum of **1,000 jobs** can have a table update trigger per workspace.
- Up to **1,000** table update triggers on OpenSharing objects or system tables per workspace.
- For **views**: only views depending on tables also supported by table update triggers work; views using `read_files`, views depending on non-UC or federated tables are **not** supported (trigger creation succeeds, but no runs fire for unsupported dependencies). A view's update is detected whenever any dependent table updates — even changes filtered out by the view definition can trigger a run. A monitored view also has a separate limit of **10 dependent views**.

### REST API Body Snippet (`trigger.table_update`)
```json
{
  "name": "refresh-on-new-data",
  "trigger": {
    "pause_status": "UNPAUSED",
    "table_update": {
      "table_names": [
        "main.sales.orders",
        "main.sales.returns"
      ],
      "condition": "ANY_UPDATED",
      "min_time_between_triggers_seconds": 120,
      "wait_after_last_change_seconds": 60
    }
  },
  "tasks": [
    { "task_key": "refresh_dashboard", "dashboard_task": { "dashboard_id": "33333333-3333-3333-3333-333333333333" } }
  ]
}
```

> [!NOTE]
> Configure email or system destination notifications on job failure to be alerted if a file arrival or table update trigger **fails to evaluate** (separate from a job run failing).

## 5. Continuous Trigger

### What It Does
Keeps a job **always running** by automatically starting a new run as soon as the previous run completes or fails — useful for always-on streaming-style workloads orchestrated as a job.

### Behavior Notes
- If a run is **active** when a continuous trigger is **resumed** from a paused state, the scheduler waits for that run to finish before starting a new one.
- Pair with streaming tasks or pipeline tasks that are designed to run indefinitely; Databricks recommends **not** wrapping an already-continuous Lakeflow Spark Declarative Pipeline in a job-level continuous trigger — keep pipeline tasks in **triggered mode** when orchestrated by a job.

### REST API Body Snippet (`trigger.continuous` style via `continuous` field)
```json
{
  "name": "always-on-streaming-job",
  "continuous": {
    "pause_status": "UNPAUSED"
  },
  "tasks": [
    { "task_key": "stream_task", "notebook_task": { "notebook_path": "/Workspace/Users/user@company.com/streaming_etl" } }
  ]
}
```

## 6. Manual / API-Driven Runs

When no trigger is configured (**None**), runs only start when:
- A user clicks **Run now** in the UI
- A program calls the **Run now** REST endpoint
- An external orchestrator (e.g., Apache Airflow, Azure Data Factory) calls the Jobs API as part of a larger pipeline

### REST API Example
```json
POST /api/2.2/jobs/run-now
{
  "job_id": 123456789,
  "notebook_params": {
    "year_param": "2026"
  }
}
```

## 7. REST API — Syntax & Examples

### Unified `trigger` Object Structure
The Jobs API exposes a single `trigger` object on `jobs/create`, `jobs/update`, and `jobs/reset` that holds exactly one of the event-based or periodic sub-configurations, plus a shared `pause_status`:

```json
{
  "trigger": {
    "pause_status": "UNPAUSED",
    "periodic": { "interval": 1, "unit": "HOURS" },
    "file_arrival": { "url": "...", "min_time_between_triggers_seconds": 60, "wait_after_last_change_seconds": 300 },
    "table_update": { "table_names": ["..."], "condition": "ANY_UPDATED" }
  }
}
```
> In practice only one of `periodic`, `file_arrival`, or `table_update` is set per trigger — they are shown together here only to illustrate the schema shape.

### Add a Table Update Trigger via API to an Existing Job
```json
POST /api/2.2/jobs/update
{
  "job_id": 123456789,
  "new_settings": {
    "trigger": {
      "pause_status": "UNPAUSED",
      "table_update": {
        "table_names": ["main.sales.orders"],
        "condition": "ANY_UPDATED",
        "wait_after_last_change_seconds": 120
      }
    }
  }
}
```

### Pause a Trigger via API
```json
POST /api/2.2/jobs/update
{
  "job_id": 123456789,
  "new_settings": {
    "trigger": {
      "pause_status": "PAUSED"
    }
  }
}
```

## 8. Databricks Asset Bundles (DABs) — Trigger Definitions

> [!IMPORTANT]
> Databricks Asset Bundles (also referred to as **Declarative Automation Bundles**) is the recommended infrastructure-as-code approach for defining job triggers consistently across dev/staging/prod targets.

### Scheduled — Legacy `schedule` Block
```yaml
resources:
  jobs:
    my-notebook-job:
      name: my-notebook-job
      schedule:
        quartz_cron_expression: '0 0 8 * * ?'   # daily at 8am
        timezone_id: UTC
        pause_status: UNPAUSED
      tasks:
        - task_key: my-notebook-task
          notebook_task:
            notebook_path: ./my-notebook.ipynb
```

### Scheduled — Periodic `trigger` Block
```yaml
resources:
  jobs:
    refresh_pipeline_hourly:
      name: refresh_pipeline_hourly
      trigger:
        periodic:
          interval: 1
          unit: HOURS
      email_notifications:
        on_failure:
          - someone@example.com
      tasks:
        - task_key: refresh_pipeline
          pipeline_task:
            pipeline_id: ${resources.pipelines.my_pipeline.id}
```

### File Arrival Trigger
```yaml
resources:
  jobs:
    process_new_files:
      name: process_new_files
      trigger:
        pause_status: UNPAUSED
        file_arrival:
          url: /Volumes/mycatalog/myschema/myvolume/incoming/
          min_time_between_triggers_seconds: 60
          wait_after_last_change_seconds: 300
      tasks:
        - task_key: load_files
          notebook_task:
            notebook_path: ../src/load_files.ipynb
```

### Table Update Trigger
```yaml
resources:
  jobs:
    refresh_on_orders_update:
      name: refresh_on_orders_update
      trigger:
        pause_status: UNPAUSED
        table_update:
          table_names:
            - main.sales.orders
            - main.sales.returns
          condition: ANY_UPDATED
          min_time_between_triggers_seconds: 120
          wait_after_last_change_seconds: 60
      tasks:
        - task_key: refresh_dashboard
          dashboard_task:
            dashboard_id: 33333333-3333-3333-3333-333333333333
```

### Deploy & Validate
```bash
databricks bundle validate -t prod
databricks bundle deploy -t prod
databricks bundle run -t prod refresh_on_orders_update
```

## 9. Pause and Resume Triggers

- **Pause** and **Resume** buttons appear in the **Job details** pane under **Schedules & Triggers** only for jobs with a configured trigger.
- **Pause**: any currently active run continues to completion, but the trigger stops starting new runs.
- **Resume**: the trigger resumes with the same previously configured behavior. For a **continuous** trigger, if a run is still active when resumed, the scheduler waits for it to finish before starting a new one.
- The same toggle is available inline when creating/editing a trigger via the **Trigger Status** control (**Active** vs **Paused**), and via the API's `pause_status` field (`UNPAUSED` / `PAUSED`).

## 10. Dynamic Value References from Triggers

When a job uses a **table update trigger**, three dynamic value references become available as parameter values:

| Reference | Description |
|---|---|
| `{{job.trigger.table_update.updated_tables}}` | JSON list of tables updated since the last job run |
| `` {{job.trigger.table_update.`<catalog.schema.table>`.commit_timestamp.iso_datetime}} `` | Most recent commit timestamp that triggered the run |
| `` {{job.trigger.table_update.`<catalog.schema.table>`.version}} `` | Most recent commit version that triggered the run |

If only one table is monitored, the `<catalog.schema.table>` qualifier can be omitted, e.g. `{{job.trigger.table_update.commit_timestamp.iso_datetime}}`.

## 11. Best Practices

**Choosing a Trigger Type:**
- Use **Scheduled** for predictable, periodic workloads where exact timing matters more than reacting to data as it lands.
- Use **File arrival** when data lands irregularly in cloud storage and you want to avoid wasted compute from polling on a fixed cron.
- Use **Table update** when downstream consumers (dashboards, models, marts) should refresh as soon as upstream Unity Catalog tables change, without coupling to an upstream job's own schedule — especially useful in Data Mesh / decentralized ownership setups.
- Use **Continuous** only for jobs intended to run indefinitely; keep continuous pipelines out of job-level continuous triggers to avoid double-continuous wrapping.

**Rate Control (File Arrival & Table Update):**
- Use **Minimum time between triggers** to cap run frequency (a cooldown) when arrivals/updates are frequent.
- Use **Wait after last change** to debounce bursty arrivals/updates so a single run processes a full batch.
- When both are set, the minimum-time window is honored first, then the wait-after-last-change debounce applies on top.

**Performance:**
- Enable **file events** on the Unity Catalog external location backing your monitored tables/paths — it improves file arrival trigger, table update trigger, and Auto Loader performance, and removes most of the file-count limits that apply otherwise.
- Avoid pointing a file arrival trigger at a high-churn root path; scope it to a dedicated UC volume mapped to the exact subdirectory you care about to prevent trigger evaluation timeouts.

**Reliability:**
- Configure **failure notifications** so you're alerted if a file arrival or table update trigger fails to *evaluate* — this is distinct from the job run itself failing.
- Remember overwriting a file with the same name does **not** count as a new arrival for file arrival triggers — design ingestion to write new file names or rely on table update triggers instead.

**CI/CD & Governance:**
- Define triggers as part of **Databricks Asset Bundles (DABs)** for IaC-managed, environment-parameterized deployment (`dev` / `staging` / `prod`).
- Always run `databricks bundle validate` before `databricks bundle deploy`.
- Use the trigger's `pause_status` field (or the UI Pause/Resume controls) to safely halt automated runs during maintenance windows without deleting the trigger configuration.

**Monitoring:**
- Use `system.lakeflow` system tables (`jobs`, `job_tasks`, `job_run_timeline`, `task_run_timeline`) to analyze how often each trigger type fires and correlate with run duration/cost.

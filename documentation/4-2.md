## 1. Core Concepts Overview

### What is Lakeflow Jobs?
- **Lakeflow Jobs** is Databricks' fully managed workflow orchestration service, used to schedule and coordinate multiple tasks (ETL, ML, analytics, AI) as a single DAG-based workflow.
- A **job** is the primary resource for coordinating, scheduling, and running operations — it can range from a single notebook task to hundreds of tasks with conditional logic and dependencies.
- A **task** is a unit of work within a job (notebook, SQL, dashboard, pipeline, Python script, dbt, JAR, etc.).
- A **trigger** initiates a job run — time-based (cron schedule), event-based (file arrival), or manual (`Run now`).
- Tasks are visualized as a **Directed Acyclic Graph (DAG)**, support `depends_on` relationships, and support control-flow constructs like If/else conditions and For each loops.

### Naming Note
- The product was formerly called **Databricks Workflows / Jobs**; it is now branded **Lakeflow Jobs** as part of the unified **Lakeflow** data engineering suite (alongside Lakeflow Connect and Lakeflow Spark Declarative Pipelines, formerly Delta Live Tables).

### Task Types Covered Here
| Task Type | Field in API/DAB | Purpose |
|---|---|---|
| **Notebook** | `notebook_task` | Runs a Databricks notebook (`.py`, `.ipynb`, `.sql`, `.r`, `.scala`) with parameters |
| **SQL** | `sql_task` | Runs a SQL query, SQL file, refreshes a SQL alert, or refreshes a legacy SQL dashboard |
| **Dashboard** | `dashboard_task` | Refreshes an AI/BI Dashboard and optionally sends a snapshot to subscribers |
| **Pipeline** | `pipeline_task` | Triggers an update of a Lakeflow Spark Declarative Pipeline (ETL pipeline, materialized views, streaming tables) |

> [!NOTE]
> The full task-type list also includes Python script, Python wheel, dbt, JAR, Power BI, Run Job, For each, Condition (If/else), Clean Rooms notebook, and Database Table Sync pipeline tasks. This guide focuses on the four most common types: **notebook, SQL, dashboard, pipeline**.

### Compute for Tasks
- Each task has an associated compute resource. If **serverless compute** is enabled for jobs, Databricks manages compute automatically; otherwise you must assign **jobs compute** (recommended) or **all-purpose compute** per task.

## 2. Notebook Task

### What It Does
A notebook task runs a Databricks notebook and passes parameters into it. Each task type has dynamic configuration options in the workspace UI.

### UI Configuration
1. Workspace sidebar → **Jobs & Pipelines** → **Create** → **Job**
2. Click the Notebook tile to configure the first task. If the Notebook tile is not available, click Add another task type and search for Notebook.
3. Enter a **Task name**
4. Select a notebook for the **Path** field (Workspace path or Git-sourced)
5. Optionally configure **Parameters** (key-value pairs, consumed in the notebook via `dbutils.widgets.get`)
6. Select a **Compute** option if serverless isn't enabled
7. Click **Create task**

### Accessing Parameters Inside the Notebook
```python
# Set a default (for when not running in a job)
dbutils.widgets.text("year_param", "2012", "Year Parameter")

# Retrieve a job-level or task-level parameter
year_value = dbutils.widgets.get("year_param")

display(babynames.filter(babynames.Year == year_value))
```

### Passing Output to Downstream Tasks
```python
# Databricks notebook source
message = "Hello from the first task"
dbutils.jobs.taskValues.set(key="message", value=message)
print(f"Produced message: {message}")
```
A downstream task references it with a dynamic value reference: `{{tasks.output_value.values.message}}`.

### DAB Example
```yaml
resources:
  jobs:
    my-notebook-job:
      name: my-notebook-job
      tasks:
        - task_key: my-notebook-task
          notebook_task:
            notebook_path: ./my-notebook.ipynb
          parameters:
            - name: my_job_run_id
              default: '{{job.run_id}}'
```

### REST API Body Snippet
```json
{
  "task_key": "notebook_task",
  "notebook_task": {
    "notebook_path": "/Workspace/Users/user@company.com/etl_notebook",
    "base_parameters": {
      "year_param": "2026"
    }
  }
}
```

> [!NOTE]
> Notebook, dbt, Python script, and SQL File tasks support version-controlled source code from a remote Git repository (`git_source`). This can be overridden per task by setting `source: WORKSPACE`.

## 3. SQL Task (Query, File, Alert, Legacy Dashboard)

### What It Does
The task runs a SQL query or file, or it refreshes a SQL alert or a legacy SQL dashboard when the sql_task field is present. All `sql_task` variants require a **SQL warehouse** (serverless or pro).

### 3.1 SQL Query Task
Runs a saved Databricks SQL query against a SQL warehouse.
```yaml
resources:
  jobs:
    my-sql-query-job:
      name: my-sql-query-job
      tasks:
        - task_key: my-sql-query-task
          sql_task:
            warehouse_id: 1a111111a1111aa1
            query:
              query_id: 11111111-1111-1111-1111-111111111111
```

### 3.2 SQL File Task
Runs a `.sql` file stored in the workspace or a Git repo.
```yaml
resources:
  jobs:
    my-sql-file-job:
      name: my-sql-file-job
      tasks:
        - task_key: my-sql-file-task
          sql_task:
            file:
              path: /Users/someone@example.com/hello-world.sql
              source: WORKSPACE
            warehouse_id: 1a111111a1111aa1
```

### 3.3 SQL Alert Task
Refreshes a Databricks SQL alert and triggers notifications to subscribers when its condition is met.
```yaml
resources:
  jobs:
    my-sql-alert-job:
      name: my-sql-alert-job
      tasks:
        - task_key: my-sql-alert-task
          sql_task:
            warehouse_id: 1a111111a1111aa1
            alert:
              alert_id: 11111111-1111-1111-1111-111111111111
```

### 3.4 Legacy SQL Dashboard
The `sql_task` type can also refresh a **legacy** SQL dashboard (the predecessor to AI/BI Dashboards). For new dashboards, use the dedicated **Dashboard task** (`dashboard_task`) described in Section 4.

### REST API Body Snippet
```json
{
  "task_key": "sql_task_example",
  "sql_task": {
    "warehouse_id": "1a111111a1111aa1",
    "query": {
      "query_id": "11111111-1111-1111-1111-111111111111"
    }
  }
}
```

## 4. Dashboard Task (AI/BI Dashboards)

### What It Does
You use this task to refresh a dashboard and send a snapshot to subscribers. This is the modern task type for **AI/BI Dashboards** (the successor to legacy SQL dashboards).

### UI Configuration
1. Add a task → select **Dashboard** as the task type
2. Choose the **Dashboard** to refresh
3. (Optional) Set **dataset parameters** — note that dataset parameters can't be set directly in the task. To use a dataset parameter, first attach it to a dashboard filter widget on the dashboard canvas, then set that filter's value in the task configuration
4. If a dashboard filter's URL Identifier matches a job-level parameter key, the task automatically applies that parameter's value to the filter
5. (Optional) Configure duration thresholds, notifications, retries under **Advanced task settings**
6. Click **Save task**

### Permissions & Credentials Behavior
Dashboard permissions and caching behavior depend on how the dashboard is published: for dashboards published with shared data permissions, viewers access data and compute according to the dashboard publisher's credentials. For dashboards published without shared data permissions, the identity defined in the job's Run as setting is used to refresh the dashboard, defaulting to the job owner.

### DAB Example
```yaml
resources:
  jobs:
    my-dashboard-job:
      name: my-dashboard-job
      tasks:
        - task_key: my-dashboard-task
          dashboard_task:
            dashboard_id: 11111111-1111-1111-1111-111111111111
```

### REST API Body Snippet
```json
{
  "task_key": "refresh_sales_dashboard",
  "dashboard_task": {
    "dashboard_id": "11111111-1111-1111-1111-111111111111",
    "subscription": {
      "subscribers": [
        { "user_name": "user@company.com" }
      ]
    }
  }
}
```

## 5. Pipeline Task (Lakeflow Spark Declarative Pipelines)

### What It Does
A pipeline task runs a pipeline. You can specify existing Lakeflow Spark Declarative Pipelines, such as a materialized view or streaming table. Lakeflow Jobs provides a procedural approach to defining relationships between tasks. Lakeflow Spark Declarative Pipelines provides a declarative approach to defining relationships between datasets and transformations.

### Key Concepts
- A triggered pipeline is a pipeline that does not run continuously, but must be triggered to start. A pipeline task can be the triggering mechanism for a triggered pipeline. Continuous pipelines do not need to be triggered, so triggering them through a task would be redundant.
- The task triggers a pipeline update when the pipeline_task field is present. Only pipelines configured to use triggered mode are supported.
- A pipeline can only run one update at a time. Jobs that contain a pipeline task are subject to the following concurrency cap: a job with max_concurrent_runs > 1 that contains a pipeline task is capped to a single concurrent run.

### UI Configuration
1. Add a task → select **Pipeline** as the task type
2. Create and name a new task and select pipeline for the Type
3. Choose the target pipeline (you can get the pipeline's ID from the **Pipeline details** tab of its settings page)
4. (Optional) Pass parameters via dynamic value references; you can access job or task parameters in your pipeline task using dynamic value references, and override parameters by adding Key-Value pairs in the Parameters of your task configuration

### Special Variant: Database Table Sync Pipeline
The Database Table Sync pipeline task is a pipeline task that runs the pipeline maintaining a Lakebase synced table. Use it to refresh a synced table on a schedule, or when the source Unity Catalog table changes, so that operational applications read current data from Lakebase Postgres. In the task Type drop-down, Database Table Sync pipeline appears under Ingestion and Transformation.

### DAB Example
```yaml
resources:
  jobs:
    my-pipeline-job:
      name: my-pipeline-job
      tasks:
        - task_key: my-pipeline-task
          pipeline_task:
            pipeline_id: 11111111-1111-1111-1111-111111111111
```

### REST API Body Snippet
```json
{
  "task_key": "run_etl_pipeline",
  "pipeline_task": {
    "pipeline_id": "11111111-1111-1111-1111-111111111111",
    "full_refresh": false
  }
}
```

## 6. REST API — Syntax & Examples

> **Base URL**: `https://<databricks-workspace-url>/api/2.2/jobs`
> **Auth Header**: `Authorization: Bearer <token>` (or OAuth/service principal token)

### 6.1 Create a Job with Multiple Task Types
```http
POST /api/2.2/jobs/create
```
```json
{
  "name": "daily-sales-pipeline",
  "tasks": [
    {
      "task_key": "ingest_notebook",
      "notebook_task": {
        "notebook_path": "/Workspace/Shared/ingest_sales"
      }
    },
    {
      "task_key": "run_pipeline",
      "depends_on": [{ "task_key": "ingest_notebook" }],
      "pipeline_task": {
        "pipeline_id": "11111111-1111-1111-1111-111111111111"
      }
    },
    {
      "task_key": "validate_sql",
      "depends_on": [{ "task_key": "run_pipeline" }],
      "sql_task": {
        "warehouse_id": "1a111111a1111aa1",
        "query": { "query_id": "22222222-2222-2222-2222-222222222222" }
      }
    },
    {
      "task_key": "refresh_dashboard",
      "depends_on": [{ "task_key": "validate_sql" }],
      "dashboard_task": {
        "dashboard_id": "33333333-3333-3333-3333-333333333333"
      }
    }
  ],
  "schedule": {
    "quartz_cron_expression": "0 0 6 * * ?",
    "timezone_id": "UTC"
  }
}
```

### 6.2 Run a Job Now
```http
POST /api/2.2/jobs/run-now
```
```json
{
  "job_id": 123456789,
  "notebook_params": { "year_param": "2026" }
}
```

### 6.3 Submit a One-Time Run (no saved job)
```http
POST /api/2.2/jobs/runs/submit
```
An optional array of objects specifying the dependency graph of the task. All tasks specified in this field must complete successfully before executing this task. The key is task_key, and the value is the name assigned to the dependent task.

### 6.4 Get / List / Delete a Job
```http
GET    /api/2.2/jobs/get?job_id=123456789
GET    /api/2.2/jobs/list
DELETE /api/2.2/jobs/delete
```

### API Operations Summary
| Operation | Method | Endpoint |
|---|---|---|
| Create a job | `POST` | `/api/2.2/jobs/create` |
| Run a job now | `POST` | `/api/2.2/jobs/run-now` |
| Submit a one-time run | `POST` | `/api/2.2/jobs/runs/submit` |
| Get job info | `GET` | `/api/2.2/jobs/get` |
| List jobs | `GET` | `/api/2.2/jobs/list` |
| Update job | `POST` | `/api/2.2/jobs/update` |
| Reset job (full overwrite) | `POST` | `/api/2.2/jobs/reset` |
| Delete a job | `POST` | `/api/2.2/jobs/delete` |

## 7. Python SDK — Jobs API

> Install: `pip install databricks-sdk`

```python
from databricks.sdk import WorkspaceClient
from databricks.sdk.service import jobs

w = WorkspaceClient(
    host="https://<workspace-url>",
    token="<personal-access-token>"
)

# --- Create a job with a notebook task, a pipeline task, a SQL task, and a dashboard task ---
created_job = w.jobs.create(
    name="daily-sales-pipeline",
    tasks=[
        jobs.Task(
            task_key="ingest_notebook",
            notebook_task=jobs.NotebookTask(
                notebook_path="/Workspace/Shared/ingest_sales"
            ),
        ),
        jobs.Task(
            task_key="run_pipeline",
            depends_on=[jobs.TaskDependency(task_key="ingest_notebook")],
            pipeline_task=jobs.PipelineTask(
                pipeline_id="11111111-1111-1111-1111-111111111111"
            ),
        ),
        jobs.Task(
            task_key="validate_sql",
            depends_on=[jobs.TaskDependency(task_key="run_pipeline")],
            sql_task=jobs.SqlTask(
                warehouse_id="1a111111a1111aa1",
                query=jobs.SqlTaskQuery(query_id="22222222-2222-2222-2222-222222222222"),
            ),
        ),
        jobs.Task(
            task_key="refresh_dashboard",
            depends_on=[jobs.TaskDependency(task_key="validate_sql")],
            dashboard_task=jobs.DashboardTask(
                dashboard_id="33333333-3333-3333-3333-333333333333"
            ),
        ),
    ],
)
print(f"Created job ID: {created_job.job_id}")

# --- Run the job now ---
run = w.jobs.run_now(job_id=created_job.job_id)
print(f"Triggered run ID: {run.run_id}")

# --- List jobs ---
for job in w.jobs.list():
    print(f"ID: {job.job_id} | Name: {job.settings.name}")

# --- Get job run status ---
run_status = w.jobs.get_run(run_id=run.run_id)
print(f"Run state: {run_status.state.life_cycle_state}")

# --- Delete a job ---
w.jobs.delete(job_id=created_job.job_id)
```

## 8. Databricks CLI — Jobs Commands

```bash
# Configure authentication
databricks configure --token

# --- Create a job from a JSON spec ---
databricks jobs create --json @job_spec.json

# --- List jobs ---
databricks jobs list

# --- Get job details ---
databricks jobs get --job-id 123456789

# --- Run a job now ---
databricks jobs run-now --job-id 123456789

# --- Run a job with notebook parameters ---
databricks jobs run-now --job-id 123456789 \
  --notebook-params '{"year_param": "2026"}'

# --- Get run output ---
databricks jobs get-run --run-id 987654321

# --- Delete a job ---
databricks jobs delete --job-id 123456789
```

## 9. Databricks Asset Bundles (DABs) — Task Definitions

> [!IMPORTANT]
> Databricks Asset Bundles (now also referred to as **Declarative Automation Bundles**) is the recommended infrastructure-as-code standard for deploying jobs with all task types across dev/staging/prod targets.

### Full Example: One Job, Four Task Types
```yaml
# databricks.yml
bundle:
  name: daily-sales-pipeline

resources:
  jobs:
    daily_sales_pipeline:
      name: daily-sales-pipeline
      schedule:
        quartz_cron_expression: "0 0 6 * * ?"
        timezone_id: "UTC"
      tasks:
        - task_key: ingest_notebook
          notebook_task:
            notebook_path: ../src/ingest_sales.ipynb

        - task_key: run_pipeline
          depends_on:
            - task_key: ingest_notebook
          pipeline_task:
            pipeline_id: 11111111-1111-1111-1111-111111111111

        - task_key: validate_sql
          depends_on:
            - task_key: run_pipeline
          sql_task:
            warehouse_id: 1a111111a1111aa1
            query:
              query_id: 22222222-2222-2222-2222-222222222222

        - task_key: refresh_dashboard
          depends_on:
            - task_key: validate_sql
          dashboard_task:
            dashboard_id: 33333333-3333-3333-3333-333333333333

targets:
  dev:
    mode: development
    workspace:
      host: https://dev-workspace.cloud.databricks.com

  prod:
    mode: production
    workspace:
      host: https://prod-workspace.cloud.databricks.com
    run_as:
      service_principal_name: "prod-service-principal"
```

### Deploy & Run
```bash
databricks bundle validate -t prod
databricks bundle deploy -t prod
databricks bundle run -t prod daily_sales_pipeline
```

> [!NOTE]
> The job git_source field and task source field set to GIT are not recommended for bundles, because local relative paths may not point to the same content in the Git repository. Bundles expect that a deployed job has the same files as the local copy from where it was deployed. Use Workspace-synced source instead for bundle-managed tasks.

## 10. Passing Parameters Between Tasks

### Parameter Types
| Type | Defined At | Pushed Down Automatically? |
|---|---|---|
| **Job parameter** | Job level | Yes, to tasks supporting key-value params |
| **Task parameter** | Task level | No — must be referenced explicitly |
| **Task value** | Set at runtime inside a task | No — referenced via `{{tasks.<key>.values.<name>}}` |

### Setting a Job Parameter and Referencing It
When you are configuring a task in the Databricks UI, use the dynamic value reference syntax to retrieve job parameters or other dynamic values. To retrieve job parameters, use the syntax: {{job.parameters.<name>}}.

### Notebook → Notebook Task Value Passing (DAB)
```yaml
resources:
  jobs:
    pass_task_values_job:
      name: pass_task_values_job
      tasks:
        - task_key: output_value
          notebook_task:
            notebook_path: ../src/output_notebook.ipynb
        - task_key: input_value
          depends_on:
            - task_key: output_value
          notebook_task:
            notebook_path: ../src/input_notebook.ipynb
            base_parameters:
              received_message: '{{tasks.output_value.values.message}}'
```

### Accessing Parameters by Task Type
Notebook code running in a task can access the values of parameters with the dbutils library. Each task type passes task values to the configured code assets differently — for example, notebook tasks use the dbutils.widgets submodule, while Python scripts pass values as arguments to the script as if it were being called from the command line. For SQL tasks and pipeline tasks, parameters are referenced using **named parameter syntax** within the SQL/pipeline source code.

## 11. Best Practices

**Task Design:**
- Prefer **serverless compute** for jobs to avoid cluster startup latency and reduce operational overhead, unless you need fixed/legacy cluster configurations.
- Use **job parameters** for values shared across multiple tasks; use **task parameters** only for task-specific overrides.
- Keep pipeline tasks on **triggered mode**; continuous pipelines should not be wrapped in a job trigger.

**Dashboard Tasks:**
- Attach dataset parameters to a dashboard filter widget before referencing them in a job — they cannot be set directly on the task.
- Match dashboard filter **URL Identifiers** to job parameter keys to enable automatic parameter pass-through.
- Be mindful of the **Run as** identity when the dashboard is published without shared data permissions — it determines whose credentials are used to refresh data.

**SQL Tasks:**
- Use the **SQL query task** for saved, reusable Databricks SQL queries.
- Use the **SQL file task** for version-controlled `.sql` scripts in a Git repo or workspace.
- Use the **SQL alert task** to drive notification-based monitoring workflows rather than polling dashboards.

**CI/CD & Governance:**
- Define all jobs and tasks as **Databricks Asset Bundles (DABs)** for IaC-based, environment-parameterized deployment (`dev` / `staging` / `prod` targets).
- Use **Service Principals** (via `run_as`) for production job ownership rather than personal accounts.
- Always run `databricks bundle validate` before `databricks bundle deploy`.

**Monitoring:**
- Use `system.lakeflow` system tables (`jobs`, `job_tasks`, `job_run_timeline`, `task_run_timeline`) to query historical job/task performance and cost.
- Configure **duration thresholds** and **streaming backlog thresholds** at the task level for proactive alerting on long-running or backlogged tasks.

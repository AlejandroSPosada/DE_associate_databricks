## 1. File Layout Optimization — Why It Matters

Delta Lake stores data as **Parquet files** in cloud object storage. Without optimization, files can become:

| Problem | Cause | Effect on Queries |
|---|---|---|
| **Small files** | Streaming inserts, many small writes | Too many files → high metadata overhead → slow scans |
| **Disorganized layout** | Random writes, no clustering | Full table scans even for selective queries |
| **Stale statistics** | Data changes without ANALYZE | Optimizer makes poor join/filter plans |
| **Orphaned files** | Deletes/updates leave old versions | Storage waste, increased scan costs |

**Data skipping** is the key benefit of file layout optimization: when files are co-located by common query predicates, the Delta reader skips files that cannot contain relevant rows based on per-file min/max statistics.

## 2. Traditional Approaches — Partitioning & Z-Order (Legacy)

### Hive-Style Partitioning
- Organizes files into directory hierarchies: `.../date=2024-01-01/region=US/...`
- Works well for **very high cardinality date columns** that are always used as filters
- **Problems**: partition explosion with high-cardinality columns, inflexible once created, poor for ad-hoc queries on different columns

### Z-Order (ZORDER BY)
```sql
-- Z-Order syntax (legacy)
OPTIMIZE table_name
ZORDER BY (customer_id, order_date);
```
- Multilevel sort that co-locates related data across multiple columns
- **Problems**: full table rewrite required for every optimization, expensive on large tables, manual maintenance burden, incompatible with Liquid Clustering

> **As of 2026: Liquid Clustering is the recommended replacement for both approaches for new tables.**

## 3. Liquid Clustering — Core Concept

Liquid Clustering is Delta Lake's modern, **incremental** data layout strategy.

### Core Principles
1. **Cluster keys defined at the table level** (not per-query like Z-Order)
2. **Incremental maintenance** — `OPTIMIZE` only clusters new/unclustered data (no full table rewrites)
3. **Keys can be changed** without rewriting history
4. **Automatic option** (`CLUSTER BY AUTO`) lets Databricks choose keys based on query patterns
5. **Incompatible with PARTITIONED BY and ZORDER BY**

### How Liquid Clustering Works
```
New data arrives (unoptimized)
        │
        ▼
OPTIMIZE runs (manually or via Predictive Optimization)
        │
        ▼
Liquid Clustering organizes new/unclustered files
by clustering key column ranges
        │
        ▼
Per-file min/max statistics recorded in Delta log
        │
        ▼
Queries with filters on clustering columns → DATA SKIPPING
```

### Data Skipping with Liquid Clustering
When a query filters on `WHERE customer_id BETWEEN 1000 AND 2000`:
- Delta checks per-file min/max statistics for `customer_id`
- Files where `max(customer_id) < 1000` or `min(customer_id) > 2000` are **skipped entirely**
- Only files that could contain matching rows are read

## 4. Liquid Clustering Syntax

### Create a New Clustered Table
```sql
-- Create with explicit clustering keys
CREATE TABLE catalog.schema.orders (
  order_id    BIGINT,
  customer_id BIGINT,
  order_date  DATE,
  region      STRING,
  amount      DECIMAL(10,2)
) USING DELTA
CLUSTER BY (customer_id, order_date);

-- Create with automatic key selection (Unity Catalog managed tables only)
CREATE TABLE catalog.schema.events (
  event_id    BIGINT,
  user_id     BIGINT,
  event_type  STRING,
  event_ts    TIMESTAMP
) USING DELTA
CLUSTER BY AUTO;
```

### Create with CTAS (Create Table As Select)
```sql
CREATE TABLE catalog.schema.orders_clustered
CLUSTER BY (customer_id, order_date)
AS SELECT * FROM source_table;
```

### Add Clustering to an Existing Table
```sql
ALTER TABLE catalog.schema.orders
CLUSTER BY (customer_id, order_date);
-- Note: this only affects FUTURE writes.
-- Run OPTIMIZE FULL to re-cluster existing data.
```

### Remove Clustering from a Table
```sql
ALTER TABLE catalog.schema.orders
CLUSTER BY NONE;
```

### Check Clustering Configuration
```sql
DESCRIBE TABLE EXTENDED catalog.schema.orders;
-- Look for: "Clusteringcolumns" in the output
```

## 5. OPTIMIZE — Incremental Clustering Maintenance

### Running OPTIMIZE on a Liquid-Clustered Table
```sql
-- Standard incremental optimize (recommended for routine maintenance)
OPTIMIZE catalog.schema.orders;

-- Full optimize — re-clusters ALL data including historical files
-- Use after: changing clustering keys, initial setup, major data changes
OPTIMIZE FULL catalog.schema.orders;
```

### What `OPTIMIZE` Does for Liquid-Clustered Tables
1. Identifies files that are **unclustered or under-clustered**
2. Reads those files and rewrites them in clustered order
3. Updates Delta log with new file statistics
4. Skips already-clustered files (incremental = only processes what needs work)

### `OPTIMIZE` vs `OPTIMIZE FULL`
| Command | Files Processed | When to Use |
|---|---|---|
| `OPTIMIZE` | Only new/unclustered files | Routine maintenance, after new data arrives |
| `OPTIMIZE FULL` | All files in the table | After changing clustering keys, initial cluster setup on existing table |

### Why Incremental OPTIMIZE is Better than Z-Order
| Aspect | OPTIMIZE (Liquid) | OPTIMIZE ZORDER BY (Legacy) |
|---|---|---|
| Files processed | Only new/unclustered | All files in table (or partition) |
| Cost | Low (incremental) | High (full rewrite each time) |
| Frequency | Can run frequently (even after each load) | Expensive to run often |
| Keys per run | Always same keys | Must specify in every OPTIMIZE command |

## 6. Changing Clustering Keys — ALTER TABLE

### Syntax
```sql
-- Change clustering keys to different columns
ALTER TABLE catalog.schema.orders
CLUSTER BY (region, order_date);

-- After this command:
-- 1. Future writes are clustered by (region, order_date)
-- 2. Existing files are still clustered by the OLD keys (customer_id, order_date)
-- 3. Run OPTIMIZE FULL to re-cluster all historical data
```

### What Happens After ALTER TABLE CLUSTER BY
| Data | Clustered By |
|---|---|
| Newly written files | New keys (region, order_date) |
| Existing historical files | Old keys (customer_id, order_date) until OPTIMIZE FULL runs |

### Full Re-Clustering Workflow
```sql
-- Step 1: Change keys
ALTER TABLE catalog.schema.orders CLUSTER BY (region, order_date);

-- Step 2: Re-cluster all historical data
OPTIMIZE FULL catalog.schema.orders;
-- Note: this can be expensive for very large tables
```

### Streaming Tables / Materialized Views
For pipeline-managed objects (streaming tables, materialized views):
- Cannot change clustering keys with `ALTER TABLE`
- Must update the pipeline definition (`CREATE STREAMING TABLE ... CLUSTER BY ...`) and refresh

## 7. CLUSTER BY AUTO

`CLUSTER BY AUTO` lets Databricks automatically select and evolve clustering keys based on actual query patterns (requires Unity Catalog managed tables).

### Syntax
```sql
CREATE TABLE catalog.schema.events (...)
USING DELTA
CLUSTER BY AUTO;
```

### How CLUSTER BY AUTO Works
1. Databricks monitors query patterns against the table
2. Analyzes which columns appear most in `WHERE` clause filters
3. Automatically selects the best clustering keys
4. Keys can evolve as query patterns change — no manual intervention

### When to Use CLUSTER BY AUTO
- Query patterns are not well-known at table creation time
- Query patterns change frequently over time
- Teams want zero-maintenance data layout optimization
- Combined with Predictive Optimization for fully automated management

## 8. Liquid Clustering vs. Z-Order vs. Partitioning

| Feature | Liquid Clustering | Z-Order | Hive Partitioning |
|---|---|---|---|
| **Defined** | `CLUSTER BY` at table level | Per `OPTIMIZE` run | `PARTITIONED BY` at table level |
| **Maintenance** | Incremental `OPTIMIZE` | Full table/partition rewrite | Manual or OPTIMIZE |
| **Key changes** | `ALTER TABLE CLUSTER BY` (easy) | Must rerun `OPTIMIZE ZORDER BY` | Cannot change without recreating |
| **Best for** | Any new table, evolving patterns | Static known query patterns | Very high selectivity date filtering |
| **Cardinality** | Low-to-medium cardinality works best | Medium cardinality | Very low cardinality only |
| **Columns supported** | Up to ~4 columns recommended | Multiple columns | Only partition columns |
| **Compatible** | Not with PARTITIONED BY or ZORDER | Not with CLUSTER BY | Not with CLUSTER BY |
| **Recommendation** | **Use for all new tables** | Legacy only | Use sparingly (date only) |

### Compatibility Rules (Critical)
- A table can use **EITHER** `CLUSTER BY` **OR** `PARTITIONED BY` — not both
- A table with `CLUSTER BY` **cannot** use `OPTIMIZE ... ZORDER BY`
- Running `OPTIMIZE ZORDER BY` on a Liquid-clustered table raises an error

## 9. Choosing Clustering Keys — Best Practices

### What Makes a Good Clustering Key?
| Criteria | Good Key | Bad Key |
|---|---|---|
| **Query filter frequency** | Appears in most WHERE clauses | Rarely filtered |
| **Cardinality** | Low-to-medium (hundreds to thousands of distinct values) | Very high cardinality (UUID, random ID) |
| **Data skipping** | Can skip many files per query | Nearly every file must be read |

### Recommended Column Types for Clustering Keys
- **Date/timestamp columns** — very common filters (`WHERE event_date = '2024-01-01'`)
- **Geography/region columns** — `WHERE region = 'US'`
- **Status/category columns** — `WHERE status = 'active'`
- **User/customer segments** — groups of users with similar query patterns

### Columns to Avoid as Clustering Keys
- **UUIDs, transaction IDs** — too high cardinality, nearly every file must be read
- **Free-text columns** — high cardinality, no data skipping benefit
- **Boolean columns alone** — only 2 values, poor data skipping

### Number of Keys
- **Recommended: 1 to 4 columns**
- More keys = diminishing returns on data skipping
- Start with the most selective filter column, add a second if the table is frequently filtered on two dimensions

```sql
-- Good: date + region (two common filter dimensions)
CLUSTER BY (order_date, region)

-- Avoid: too many columns
CLUSTER BY (order_date, region, status, user_id, product_id)  -- Too many
```

## 10. Liquid Clustering Limitations & Compatibility

| Limitation | Detail |
|---|---|
| Cannot use with `PARTITIONED BY` | Mutually exclusive — choose one |
| Cannot use with `ZORDER BY` | Running `OPTIMIZE ZORDER BY` on a clustered table raises an error |
| Streaming tables / MViews | Cannot change keys with `ALTER TABLE`; must update pipeline definition |
| Key change effect | `ALTER TABLE CLUSTER BY` only affects future writes; use `OPTIMIZE FULL` for historical data |
| Maximum recommended keys | 4 columns for best performance |
| Unity Catalog dependency for AUTO | `CLUSTER BY AUTO` requires Unity Catalog managed tables |

## 11. Predictive Optimization — Overview

**Predictive Optimization (PO)** is an automated table maintenance service for Unity Catalog managed tables.

### What PO Does
PO continuously monitors table usage patterns and automatically triggers maintenance operations using **serverless compute**, eliminating the need for manual scheduling of:
- `OPTIMIZE` (file compaction + clustering)
- `VACUUM` (removing orphaned/old files)
- `ANALYZE` (updating table statistics)

### Key Characteristics
| Attribute | Value |
|---|---|
| **Availability** | Unity Catalog managed tables only |
| **Compute** | Serverless (Jobs SKU) |
| **Billing** | Serverless jobs SKU (not your cluster) |
| **Default** | Enabled by default for all new UC tables (as of 2026) |
| **Z-Order** | Does NOT perform ZORDER BY — Liquid Clustering only |

---

## 12. Predictive Optimization — Operations Automated

### 1. OPTIMIZE (File Compaction + Clustering)
- Compacts small files into larger ones (targets ~128 MB–1 GB file sizes)
- If Liquid Clustering is enabled, applies incremental clustering
- Does **not** perform Z-Ordering
- Triggered based on amount of new data written since last OPTIMIZE

### 2. VACUUM (Storage Cleanup)
- Removes files older than `delta.deletedFileRetentionDuration` (default: **7 days**)
- Removes orphaned files not referenced by any Delta log version
- Improved with log-based approach (faster and cheaper than traditional VACUUM)

### 3. ANALYZE (Statistics Collection)
- Runs `ANALYZE TABLE` to update column-level statistics
- Updated statistics improve query optimizer join and filter plan quality
- Triggered based on data change since last analysis

### What PO Does NOT Do
- Does **not** run `OPTIMIZE ZORDER BY`
- Does **not** manage external tables (only Unity Catalog managed tables)
- Does **not** apply to Hive metastore tables

## 13. Enabling Predictive Optimization

### Enable at Account Level (Admin)
```sql
-- Enable for entire account (admin only)
ALTER METASTORE SET ENABLE_PREDICTIVE_OPTIMIZATION = ON;
```

### Enable at Catalog Level
```sql
ALTER CATALOG my_catalog
ENABLE PREDICTIVE OPTIMIZATION;
```

### Enable at Schema Level
```sql
ALTER SCHEMA my_catalog.my_schema
ENABLE PREDICTIVE OPTIMIZATION;
```

### Enable at Table Level
```sql
ALTER TABLE my_catalog.my_schema.my_table
SET TBLPROPERTIES ('delta.enablePredictiveOptimization' = 'enable');
```

### Disable at Table Level (Override Inheritance)
```sql
ALTER TABLE my_catalog.my_schema.my_table
SET TBLPROPERTIES ('delta.enablePredictiveOptimization' = 'disable');
```

### Check If Enabled
```sql
DESCRIBE CATALOG EXTENDED my_catalog;
DESCRIBE SCHEMA EXTENDED my_catalog.my_schema;
DESCRIBE TABLE EXTENDED my_catalog.my_schema.my_table;
-- Look for: "Predictive optimization" row in the output
```

## 14. Predictive Optimization Inheritance Model

PO follows a top-down inheritance model:

```
Account (metastore level)
    │
    ▼
Catalog  ← Enable/Disable here affects all schemas below
    │
    ▼
Schema   ← Enable/Disable here affects all tables below
    │
    ▼
Table    ← Can override schema/catalog setting individually
```

### Inheritance Rules
| Level Enabled | Effect |
|---|---|
| Account ON | All catalogs/schemas/tables inherit ON |
| Catalog ON | All schemas and tables in that catalog inherit ON |
| Schema ON | All tables in that schema inherit ON |
| Table DISABLE | Overrides any inherited ON — this table is exempt |

### Example: Enable for All Tables Except One
```sql
-- Enable at catalog level
ALTER CATALOG prod_catalog ENABLE PREDICTIVE OPTIMIZATION;

-- Disable for a specific sensitive table that should not be auto-vacuumed
ALTER TABLE prod_catalog.finance.raw_transactions
SET TBLPROPERTIES ('delta.enablePredictiveOptimization' = 'disable');
```

## 15. Monitoring Predictive Optimization

PO runs are observable through system tables:

```sql
-- View history of PO operations on a table
SELECT *
FROM system.storage.predictive_optimization_operations_history
WHERE table_name = 'orders'
ORDER BY operation_start DESC
LIMIT 20;

-- Key columns:
-- operation_type: OPTIMIZE, VACUUM, ANALYZE
-- status: SUCCEEDED, FAILED, SKIPPED
-- bytes_compacted, files_compacted
-- operation_start, operation_end
```

### What the History Shows
| Column | Description |
|---|---|
| `operation_type` | `OPTIMIZE`, `VACUUM`, or `ANALYZE` |
| `status` | `SUCCEEDED`, `FAILED`, `SKIPPED` |
| `bytes_compacted` | Bytes reduced by file compaction |
| `files_compacted` | Number of small files merged |
| `operation_start` | When PO triggered the operation |
| `operation_end` | When the operation completed |

## 16. Liquid Clustering + Predictive Optimization Together

The most powerful combination for fully automated table management:

### Recommended Setup for New Tables
```sql
-- Create table with Liquid Clustering (auto key selection)
CREATE TABLE catalog.schema.events (
  event_id   BIGINT,
  user_id    BIGINT,
  event_type STRING,
  event_ts   TIMESTAMP
) USING DELTA
CLUSTER BY AUTO;

-- Enable Predictive Optimization at catalog level
ALTER CATALOG my_catalog ENABLE PREDICTIVE OPTIMIZATION;

-- PO will automatically:
-- 1. Run OPTIMIZE (with Liquid Clustering) when new data arrives
-- 2. Run VACUUM when old file versions expire
-- 3. Run ANALYZE to keep statistics fresh
-- Zero manual maintenance required
```

### Fully Automated vs. Semi-Automated vs. Manual
| Approach | Clustering | Maintenance | Effort |
|---|---|---|---|
| `CLUSTER BY AUTO` + PO enabled | Automatic (query-based) | Automatic | None |
| `CLUSTER BY (cols)` + PO enabled | Manual key selection | Automatic OPTIMIZE/VACUUM/ANALYZE | Key selection only |
| `CLUSTER BY (cols)` + manual OPTIMIZE | Manual key selection | Manual | Scheduling + monitoring |
| `ZORDER BY` (legacy) | Manual per OPTIMIZE run | Full table rewrite each time | High maintenance |

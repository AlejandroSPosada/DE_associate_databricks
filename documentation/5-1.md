## 1. Core Concepts Overview

### What is a "Git Folder" (formerly Databricks Repos)?
- A **Git Folder** is a native Databricks workspace directory that mirrors a remote Git repository.
- Supports: **GitHub, GitLab, Azure DevOps (Bitbucket, AWS CodeCommit, etc.**).
- Enables cloning, branching, committing, pulling, merging, and rebasing — directly from the Databricks UI or API.

### Naming Evolution
| Old Name | New Name |
|---|---|
| Databricks Repos | **Databricks Git Folders** |
| `/Repos/...` | `/Workspace/Users/<email>/...` |

### Two Layers of CI/CD
| Layer | Tool | Purpose |
|---|---|---|
| **Git operations** | Repos API / Git Folders UI | Sync code between workspace and remote repo |
| **Deployment** | **Databricks Asset Bundles (DABs)** | Deploy jobs, pipelines, infrastructure as code |

> [!IMPORTANT]
> **Databricks Asset Bundles (DABs)** is the current recommended standard for production CI/CD. The Repos API is used for workspace synchronization only.

## 2. Git Folders (Repos) — UI Operations

### Clone a Repository (UI)
1. Sidebar → **Workspace** → browse to target folder
2. Click **Create > Git folder**
3. Fill in:
   - **Git repository URL**: `https://github.com/org/project.git`
   - **Git provider**: GitHub / Azure DevOps / GitLab / etc.
   - **Git folder name**: workspace folder name
   - **Sparse checkout mode** (optional): clone only specific subdirectories
   - **Enable Git CLI support (Beta)**: allows full `git` commands from terminal
4. Click **Create Git folder**

### Clone from Web Terminal
```bash
cd /Workspace/Users/<your-email>/<project>
git clone <remote-url>
```

### Access the Git Dialog
- From a **notebook**: click the branch name button next to the notebook title
- From the **Git folders browser**: click **Git** next to the repo name

### Git Dialog Actions
| Button/Action | Description |
|---|---|
| Branch dropdown | Switch to a different branch |
| **Create Branch** | Create a new branch |
| **Pull** | Pull latest changes from remote |
| **Commit & Push** | Stage, commit, and push to remote |
| Kebab menu (⋮) | Access Merge, Rebase, Reset, History |

## 3. Repos REST API — Syntax & Examples

> **Base URL**: `https://<databricks-workspace-url>/api/2.0/repos`
> **Auth Header**: `Authorization: Bearer <personal-access-token>`

### 3.1 Create / Clone a Repo
```http
POST /api/2.0/repos
```
**Request Body:**
```json
{
  "url": "https://github.com/my-org/my-project.git",
  "provider": "gitHub",
  "path": "/Workspace/Users/user@company.com/my-project"
}
```
**Provider values:** `gitHub`, `azureDevOpsServices`, `gitLab`, `bitbucketCloud`, `bitbucketServer`, `gitHubEnterprise`, `awsCodeCommit`

**Optional — Sparse Checkout:**
```json
{
  "url": "https://github.com/my-org/my-project.git",
  "provider": "gitHub",
  "path": "/Workspace/Users/user@company.com/my-project",
  "sparse_checkout": {
    "patterns": ["src/", "notebooks/"]
  }
}
```

**Response:**
```json
{
  "id": 123456789,
  "path": "/Workspace/Users/user@company.com/my-project",
  "url": "https://github.com/my-org/my-project.git",
  "provider": "gitHub",
  "branch": "main",
  "head_commit_id": "abc1234"
}
```

---

### 3.2 Update a Repo (Switch Branch / Pull Latest)
```http
PATCH /api/2.0/repos/{repo_id}
```

**Switch to a specific branch:**
```json
{
  "branch": "feature/my-feature"
}
```

**Switch to a tag:**
```json
{
  "tag": "v1.0.0"
}
```

> [!NOTE]
> Updating to a branch pulls the **latest commit** on that branch from the remote. This is how you "sync" a workspace folder after a PR is merged.

---

### 3.3 Get Repo Info
```http
GET /api/2.0/repos/{repo_id}
```
**Response:**
```json
{
  "id": 123456789,
  "path": "/Workspace/Users/user@company.com/my-project",
  "url": "https://github.com/my-org/my-project.git",
  "provider": "gitHub",
  "branch": "main",
  "head_commit_id": "def5678"
}
```

---

### 3.4 List All Repos
```http
GET /api/2.0/repos
```
**Optional query params:**
- `path_prefix`: filter by path prefix (e.g., `/Workspace/Users/`)
- `next_page_token`: for pagination

---

### 3.5 Delete a Repo
```http
DELETE /api/2.0/repos/{repo_id}
```

---

### 3.6 Full cURL Example — Update Branch After PR Merge
```bash
# Store token and workspace URL as env vars
export DATABRICKS_HOST="https://adb-XXXXXXXXXXXXXXXX.XX.azuredatabricks.net"
export DATABRICKS_TOKEN="dapiXXXXXXXXXXXXXXXXXXXXXXXXXXXX"
export REPO_ID="123456789"

curl -X PATCH \
  "${DATABRICKS_HOST}/api/2.0/repos/${REPO_ID}" \
  -H "Authorization: Bearer ${DATABRICKS_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"branch": "main"}'
```

### API Operations Summary

| Operation | Method | Endpoint |
|---|---|---|
| Clone a repo | `POST` | `/api/2.0/repos` |
| Switch branch / pull latest | `PATCH` | `/api/2.0/repos/{repo_id}` |
| Get repo info | `GET` | `/api/2.0/repos/{repo_id}` |
| List all repos | `GET` | `/api/2.0/repos` |
| Delete a repo | `DELETE` | `/api/2.0/repos/{repo_id}` |

## 4. Python SDK — Repos API

> Install: `pip install databricks-sdk`

```python
from databricks.sdk import WorkspaceClient
from databricks.sdk.service.workspace import CreateRepo

w = WorkspaceClient(
    host="https://<workspace-url>",
    token="<personal-access-token>"
    # Or use env vars: DATABRICKS_HOST, DATABRICKS_TOKEN
)

# --- Clone a repo ---
repo = w.repos.create(
    url="https://github.com/my-org/my-project.git",
    provider="gitHub",
    path="/Workspace/Users/user@company.com/my-project"
)
print(f"Created repo ID: {repo.id}")

# --- Switch branch (sync after PR merge) ---
w.repos.update(
    repo_id=repo.id,
    branch="main"
)
print(f"Switched to main branch")

# --- List all repos ---
for r in w.repos.list():
    print(f"ID: {r.id} | Branch: {r.branch} | Path: {r.path}")

# --- Get specific repo ---
repo_info = w.repos.get(repo_id=repo.id)
print(f"Head commit: {repo_info.head_commit_id}")

# --- Delete a repo ---
w.repos.delete(repo_id=repo.id)
```

## 5. Databricks CLI — Repos Commands

> Install: `pip install databricks-cli` or use `databricks/setup-cli` GitHub Action

```bash
# Configure authentication
databricks configure --token

# --- List repos ---
databricks repos list

# --- Create a repo ---
databricks repos create https://github.com/my-org/my-project.git \
  --path /Workspace/Users/user@company.com/my-project \
  --provider gitHub

# --- Update (sync) a repo to a branch ---
databricks repos update <repo_id> --branch main

# --- Delete a repo ---
databricks repos delete <repo_id>
```

## 6. Branches — Concepts & Operations

### Key Concepts
| Concept | Description |
|---|---|
| **Default branch** | Usually `main` or `master`. Protected in production. |
| **Feature branch** | Short-lived branch for individual development work. |
| **Release branch** | Long-lived branch representing a specific version. |
| **Trunk-based development** | Databricks-recommended strategy: short-lived branches, frequent merges to `main`. |

### Branch Operations in Git Dialog (UI)
```
1. Open Git Dialog
2. Click "Create Branch"
3. Enter branch name → select base branch (e.g., main)
4. Click "Create"
```

> [!IMPORTANT]
> Switching branches in a **shared** Git folder switches the branch for all users who share it. Best practice: each developer clones the repo into their **own** workspace folder.

### Branch Operations via Git CLI (Beta)
> Available in Git folders with "Enable Git CLI support" checked (requires Serverless compute)

```bash
cd /Workspace/Users/<your-email>/<your-repo>

# Create and switch to a new branch
git checkout -b feature/my-new-feature

# Switch to existing branch
git checkout main

# List all branches
git branch -a

# Push new branch to remote
git push -u origin feature/my-new-feature

# Delete a local branch
git branch -d feature/old-feature

# Delete a remote branch
git push origin --delete feature/old-feature

# Rebase interactively
git rebase -i main

# Stash uncommitted changes
git stash
git stash pop

# Work with submodules
git submodule update --init --recursive
```

### Switch Branch via API (CI/CD Use Case)
```bash
# After a PR is merged to main, sync the production Git folder:
curl -X PATCH \
  "${DATABRICKS_HOST}/api/2.0/repos/${PROD_REPO_ID}" \
  -H "Authorization: Bearer ${DATABRICKS_TOKEN}" \
  -d '{"branch": "main"}'
```

## 7. Commits — Concepts & Operations

### Commit via UI
1. Make changes to notebooks or files in the Git folder
2. Open Git Dialog → modified files are highlighted
3. Enter a **commit message** (required)
4. Click **Commit & Push** → pushes directly to remote

> [!NOTE]
> Notebook outputs are **NOT** included in commits by default when saved as `.py`, `.scala`, `.sql`, or `.r` source formats. Only `.ipynb` format includes outputs.

### Commit via Git CLI (Beta)
```bash
cd /Workspace/Users/<email>/<repo>

# Stage specific files
git add notebooks/my_notebook.py

# Stage all changes
git add .

# Commit with message
git commit -m "feat: add data transformation notebook"

# Push to remote
git push origin feature/my-feature

# Stage multiple commits locally before pushing
git commit -m "fix: correct schema validation"
git commit -m "refactor: optimize join logic"
git push origin feature/my-feature
```

### Commit Best Practices
- Use **conventional commit** messages: `feat:`, `fix:`, `refactor:`, `docs:`, `chore:`
- Do **NOT** commit build artifacts (`.jar`, `.whl` files) — use Unity Catalog volumes
- Do **NOT** commit credentials — use Databricks Secrets

## 8. Pull Requests & Merge

### Pull Requests — Key Concept
> [!IMPORTANT]
> **Databricks does NOT have a native API for creating or merging Pull Requests.**
> PRs are managed entirely through your Git provider (GitHub, Azure DevOps, GitLab).

### Typical PR Workflow
```
Developer → Feature Branch → Commit & Push → Open PR in GitHub/ADO
     → Code Review → Merge to main → CI/CD triggered
     → Repos API called to sync Databricks workspace
```

### Merge in Git Folders UI
- Uses `git merge` under the hood
- Access via: **Git Dialog → kebab menu (⋮) → Merge**
- Choose the **source branch** to merge INTO the current branch

### Merge Conflict Resolution (UI)
When conflicts are detected during Pull, Merge, or Rebase:
1. Conflicting files are listed in the Git dialog
2. Options:
   - **Keep all current changes** — accept your local version
   - **Take all incoming changes** — accept the remote version
   - **Manual edit** — edit the file, remove conflict markers, click "Mark As Resolved"
3. Click **Continue Merge** (or **Continue Rebase**) to finalize
4. Click **Abort** to cancel and discard all conflict resolution changes

### Rebase (Alternative to Merge)
```
Git Dialog → kebab menu (⋮) → Rebase
```
- Uses `git rebase` to replay commits on top of the target branch
- Creates linear history (no merge commit)
- Requires force push — **use with caution** on shared branches

### Reset (Discard Local Changes)
```
Git Dialog → kebab menu (⋮) → Reset → Hard Reset
```
- Discards all local uncommitted changes
- Syncs local branch to match the remote

## 9. Databricks Asset Bundles (DABs) — CI/CD Standard

> **DABs** are the recommended, modern approach to CI/CD on Databricks. Define your entire project as YAML code.

### Project Structure
```
my-databricks-project/
├── databricks.yml          # Root bundle configuration
├── resources/
│   ├── jobs/
│   │   └── my_job.yml     # Job definition
│   └── pipelines/
│       └── my_pipeline.yml
├── src/
│   ├── notebooks/
│   │   └── main.ipynb
│   └── scripts/
│       └── transform.py
└── .github/
    └── workflows/
        └── deploy.yml      # GitHub Actions CI/CD
```

### Root Bundle Configuration (`databricks.yml`)
```yaml
bundle:
  name: my-databricks-project

# Environment targets
targets:
  dev:
    default: true
    mode: development          # Auto-prefixes resources with username
    workspace:
      host: https://<dev-workspace-url>

  staging:
    mode: production
    workspace:
      host: https://<staging-workspace-url>

  prod:
    mode: production
    workspace:
      host: https://<prod-workspace-url>

# Include modular resource files
include:
  - resources/jobs/*.yml
  - resources/pipelines/*.yml

# Global variables (overridable per target)
variables:
  cluster_node_type:
    default: "i3.xlarge"
    description: "Cluster node type"

# Resource definitions (or reference via include)
resources:
  jobs:
    my_etl_job:
      name: "ETL Job - ${bundle.target}"
      tasks:
        - task_key: transform
          notebook_task:
            notebook_path: ./src/notebooks/main.ipynb
          new_cluster:
            spark_version: "15.4.x-scala2.12"
            node_type_id: ${var.cluster_node_type}
            num_workers: 2
```

### Job Resource File (`resources/jobs/my_job.yml`)
```yaml
resources:
  jobs:
    my_etl_job:
      name: "My ETL Job"
      schedule:
        quartz_cron_expression: "0 0 8 * * ?"
        timezone_id: "America/New_York"
      tasks:
        - task_key: ingest
          notebook_task:
            notebook_path: ./src/notebooks/ingest.ipynb
          job_cluster_key: main_cluster

        - task_key: transform
          depends_on:
            - task_key: ingest
          notebook_task:
            notebook_path: ./src/notebooks/transform.ipynb
          job_cluster_key: main_cluster

      job_clusters:
        - job_cluster_key: main_cluster
          new_cluster:
            spark_version: "15.4.x-scala2.12"
            node_type_id: "i3.xlarge"
            num_workers: 4

      # Target-specific overrides
      targets:
        prod:
          tasks:
            - task_key: ingest
              new_cluster:
                num_workers: 8  # Scale up in prod
```

### Bundle CLI Commands
```bash
# Initialize a new bundle project
databricks bundle init

# Validate configuration (no deployment)
databricks bundle validate

# Validate for a specific target
databricks bundle validate -t prod

# Deploy to the default (dev) target
databricks bundle deploy

# Deploy to a specific target
databricks bundle deploy -t staging
databricks bundle deploy -t prod

# Run a job defined in the bundle
databricks bundle run my_etl_job

# Run for a specific target
databricks bundle run -t prod my_etl_job

# Destroy all deployed resources (careful!)
databricks bundle destroy -t dev

# Generate JSON schema for IDE support
databricks bundle schema > bundle_config_schema.json
```

### Environment-Specific Overrides
```yaml
targets:
  dev:
    mode: development
    workspace:
      host: https://dev-workspace.azuredatabricks.net
    variables:
      cluster_node_type: "Standard_DS3_v2"  # Smaller in dev

  prod:
    mode: production
    workspace:
      host: https://prod-workspace.azuredatabricks.net
    variables:
      cluster_node_type: "Standard_DS5_v2"  # Larger in prod
    run_as:
      service_principal_name: "prod-service-principal"
```

## 10. GitHub Actions CI/CD Pipeline

### Full Production-Grade Workflow (`.github/workflows/deploy.yml`)
```yaml
name: Databricks CI/CD Pipeline

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

env:
  DATABRICKS_HOST: ${{ vars.DATABRICKS_HOST }}

jobs:
  # --- CI: Validate on Pull Request ---
  validate:
    name: Validate Bundle
    runs-on: ubuntu-latest
    if: github.event_name == 'pull_request'
    steps:
      - uses: actions/checkout@v4

      - name: Setup Databricks CLI
        uses: databricks/setup-cli@main

      - name: Validate bundle (staging)
        env:
          DATABRICKS_CLIENT_ID: ${{ secrets.DATABRICKS_CLIENT_ID }}
          DATABRICKS_CLIENT_SECRET: ${{ secrets.DATABRICKS_CLIENT_SECRET }}
        run: databricks bundle validate -t staging

      - name: Run unit tests
        run: |
          pip install pytest
          pytest tests/unit/ -v

  # --- CD: Deploy on merge to main ---
  deploy-staging:
    name: Deploy to Staging
    runs-on: ubuntu-latest
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    environment: staging
    permissions:
      id-token: write   # Required for OIDC (Workload Identity Federation)
      contents: read
    steps:
      - uses: actions/checkout@v4

      - name: Setup Databricks CLI
        uses: databricks/setup-cli@main

      - name: Deploy to Staging
        env:
          DATABRICKS_CLIENT_ID: ${{ secrets.STAGING_CLIENT_ID }}
          DATABRICKS_AUTH_TYPE: github-oidc    # Token-less OIDC auth
        run: |
          databricks bundle validate -t staging
          databricks bundle deploy -t staging

      - name: Sync staging Git folder
        env:
          DATABRICKS_TOKEN: ${{ secrets.STAGING_DATABRICKS_TOKEN }}
        run: |
          curl -X PATCH \
            "${DATABRICKS_HOST}/api/2.0/repos/${{ vars.STAGING_REPO_ID }}" \
            -H "Authorization: Bearer ${DATABRICKS_TOKEN}" \
            -d '{"branch": "main"}'

  # --- CD: Promote to Production (manual approval) ---
  deploy-prod:
    name: Deploy to Production
    runs-on: ubuntu-latest
    needs: deploy-staging
    environment: production   # Requires manual approval in GitHub
    permissions:
      id-token: write
      contents: read
    steps:
      - uses: actions/checkout@v4

      - name: Setup Databricks CLI
        uses: databricks/setup-cli@main

      - name: Deploy to Production
        env:
          DATABRICKS_CLIENT_ID: ${{ secrets.PROD_CLIENT_ID }}
          DATABRICKS_AUTH_TYPE: github-oidc
        run: |
          databricks bundle validate -t prod
          databricks bundle deploy -t prod

      - name: Sync production Git folder
        env:
          DATABRICKS_TOKEN: ${{ secrets.PROD_DATABRICKS_TOKEN }}
        run: |
          curl -X PATCH \
            "${DATABRICKS_HOST}/api/2.0/repos/${{ vars.PROD_REPO_ID }}" \
            -H "Authorization: Bearer ${DATABRICKS_TOKEN}" \
            -d '{"branch": "main"}'
```

### Authentication Methods for CI/CD
| Method | Security | Use Case |
|---|---|---|
| Personal Access Token (PAT) | Low (long-lived) | Development / Testing only |
| Service Principal + Client Secret | Medium | Basic CI/CD |
| **OIDC / Workload Identity Federation** | **High (no stored secrets)** | **Recommended for production** |

### Setting Up OIDC (Token-less Auth)
```yaml
# In your GitHub Actions workflow
permissions:
  id-token: write
  contents: read

steps:
  - name: Deploy
    env:
      DATABRICKS_HOST: ${{ vars.DATABRICKS_HOST }}
      DATABRICKS_CLIENT_ID: ${{ secrets.DATABRICKS_CLIENT_ID }}
      DATABRICKS_AUTH_TYPE: github-oidc   # Uses short-lived OIDC token
    run: databricks bundle deploy -t prod
```

## 11. Branching Strategy & Best Practices

### Recommended: Trunk-Based Development
```
main (protected) ─────────────────────────────────────────→
                    ↑           ↑           ↑
              feature/A    feature/B    hotfix/C
               (short-lived, merged via PR with code review)
```

### Feature Branch Naming Conventions
```
feature/<ticket-id>-short-description
bugfix/<ticket-id>-short-description
hotfix/<ticket-id>-short-description
release/v1.2.0
```

### Developer Workflow
```
1. Each developer clones repo to their OWN workspace folder
   /Workspace/Users/<email>/my-project

2. Create feature branch:
   git checkout -b feature/JIRA-123-add-etl

3. Develop in Databricks → commit changes via Git dialog or CLI

4. Push feature branch to remote:
   git push origin feature/JIRA-123-add-etl

5. Open Pull Request in GitHub/ADO → Code review → Approve

6. Merge PR into main (in Git provider UI)

7. CI/CD pipeline triggers automatically:
   - Validates bundle
   - Deploys to staging
   - (After approval) Deploys to production
   - Syncs workspace Git folders via Repos API
```

### Environment Isolation
| Environment | Workspace | Repo Branch | Git Folder Path |
|---|---|---|---|
| Dev | dev-workspace | `feature/*` | `/Workspace/Users/<email>/project` |
| Staging | staging-workspace | `main` | `/Workspace/Shared/project-staging` |
| Production | prod-workspace | `main` (tagged) | `/Workspace/Shared/project-prod` |

### Best Practices Summary

**Version Control:**
- Version control all notebooks (`.py`, `.sql`), YAML configs, and Python scripts
- Do NOT commit `.jar` or `.whl` build artifacts → use Unity Catalog volumes
- Do NOT commit credentials → use Databricks Secrets (`dbutils.secrets.get(...)`)

**CI/CD:**
- Use **Service Principals** (not personal user accounts) for production pipelines
- Use **OIDC** for token-less authentication in GitHub Actions
- Always run `databricks bundle validate` before deploying
- Use DAB **targets** to parameterize Dev / Staging / Prod environments

**Collaboration:**
- Each developer has their **own Git folder** clone in their user directory
- Only one user should perform Git operations on each shared Git folder
- Use `sparse checkout` for large repos to reduce sync time

**Branch Management:**
- Protect the `main` branch (require PR reviews, status checks)
- Delete feature branches after merge (keeps repo clean)
- Use `git stash` when switching branches with uncommitted changes

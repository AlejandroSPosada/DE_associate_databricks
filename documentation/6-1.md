## 1. Overview — Lakeflow Jobs Observability

Lakeflow Jobs (formerly Databricks Workflows) provides multi-layer observability:

| Layer | Tool | Best For |
|---|---|---|
| Short-term (≤60 days) | Lakeflow Jobs UI — Runs tab, Matrix view | Debugging individual runs, quick status |
| Task-level detail | Task run details panel, Spark UI link | Diagnosing specific task failures |
| Long-term (≤365 days) | System tables (`system.lakeflow`) | Trend analysis, SLAs, cost attribution |
| Custom dashboards | Databricks SQL + system tables | Performance KPIs, failure rate reports |

## 2. Navigating the Jobs UI — Run History

### Getting There
1. In the sidebar, click **Workflows** (or Lakeflow Jobs)
2. Click a job name to open its detail page
3. Click the **Runs** tab to see all run history

### Runs Tab — What You See
- **Run list**: All runs for this job, newest first
- **Run ID**: Unique identifier for each run (useful for API calls)
- **Trigger**: How the run was started — `Scheduled`, `Manual`, `File arrival`, `Repair`, etc.
- **Start time** and **Duration**
- **Result state**: `Succeeded`, `Failed`, `Canceled`, `Timed out`, `Skipped`
- **Last 5 runs**: Visible as mini color-coded indicators on the job list page

### Job List Page — Quick Status
The main Jobs list shows the last 5 run outcomes as colored dots next to each job name, enabling at-a-glance SLA monitoring across all jobs without opening each one.

## 3. Matrix View — Visual Run History

The **matrix view** is a 2D grid showing run history across both time and tasks.

```
              Run 1    Run 2    Run 3    Run 4    Run 5
task_ingest   [green]  [green]  [red]    [green]  [green]
task_transform[green]  [green]  [pink]   [green]  [green]
task_load     [green]  [green]  [pink]   [green]  [green]
```

### How to Read the Matrix
- **Each column** = one job run (newest on the right)
- **Each row** = one task
- **Color** = task status for that run
- Quickly reveals: which task failed, how often, whether failures cascade

### When Matrix View is Most Useful
- Identifying **recurring failures** on the same task across multiple runs
- Distinguishing **isolated failures** from **systematic problems**
- Spotting **cascading failures** (one failed task → downstream tasks skipped)

## 4. Task-Level Status Colors

| Color | Status | Meaning |
|---|---|---|
| **Green** | Succeeded | Task completed successfully |
| **Red** | Failed | Task encountered an error and did not complete |
| **Pink** | Skipped | Task did not run (upstream dependency failed or disabled) |
| **Yellow** | Waiting / Retrying | Task is queued, blocked on a dependency, or waiting to retry |
| **Blue** | Running | Task is currently in progress |
| **Gray** | Canceled | Run was manually canceled before the task ran |

### Why Tasks Get Skipped
A task is **skipped** when:
1. A direct upstream dependency **failed** and the task's `Run if` condition requires `All succeeded`
2. The task itself has been **disabled** (toggled off in the UI or via API)
3. A conditional `Run if` expression evaluates to `false`

### `Run if` Condition Options
| Condition | Behavior |
|---|---|
| `All succeeded` | Runs only if all upstream tasks succeeded |
| `At least one succeeded` | Runs if any upstream task succeeded |
| `None failed` | Runs unless at least one upstream task failed |
| `All done` | Always runs, regardless of upstream status |
| `At least one failed` | Only runs when a failure occurs upstream (e.g., error handler) |

### Disabled Tasks
- Disabling a task removes it from execution without deleting it
- Configuration and run history are preserved
- Downstream tasks still evaluate their `Run if` conditions against the disabled task (treated as skipped)

## 5. Run Detail Page — Task Inspection

Clicking any individual run opens the **Run detail page**:

### Available Information
| Section | Contents |
|---|---|
| **DAG view** | Visual dependency graph of all tasks and their status |
| **Task list panel** | All tasks sorted by execution order with status and duration |
| **Task run details** | On click: output, error message, logs, Spark UI link |
| **Cluster info** | Cluster used, driver/worker node types, start time |
| **Metrics tab** | For streaming tasks: streaming backlog, query metrics |
| **Ganglia / Spark UI** | Link to full cluster metrics for the specific run |

### Task Run Details Panel
- **Output**: Notebook cell outputs or stdout from Python scripts
- **Error**: Full stack trace for failed tasks
- **Logs**: Links to driver logs, event logs
- **Spark UI**: Direct link to the Spark UI for this specific task's cluster session
- **Duration**: Wall-clock time for this task in isolation
- **Attempt**: Which attempt number this was (1 for first try, 2+ for retries)

## 6. Run History Retention

| Storage | Retention Period |
|---|---|
| Jobs UI (Runs tab) | **60 days** |
| System tables (`system.lakeflow`) | **365 days** |

> **Key exam point**: If you need run history older than 60 days — for year-over-year trend analysis, auditing, or SLA reporting — you must query the `system.lakeflow` system tables, not the UI.

## 7. Failure Recovery — Repair Run vs. Retry

Two distinct mechanisms for handling task failures:

| Feature | Repair Run | Task Retry |
|---|---|---|
| **Triggered by** | User manually (after investigating) | Automatic (configured in task settings) |
| **Scope** | Re-runs only failed tasks + their downstream | Re-runs the same individual task |
| **When** | After the full job has completed with failures | During job execution, immediately after failure |
| **Use case** | Transient infrastructure failure, fixed data issue | Transient errors (network flap, API timeout) |
| **Cost efficiency** | High — skips already-succeeded tasks | Lower — only retries one task |

## 8. Repair Run — Concept & Behavior

A **Repair Run** is the recommended recovery mechanism for long-running ETL jobs with multiple tasks.

### What It Does
- Accepts the existing run ID
- Re-executes **only the tasks that failed or were skipped**
- Does NOT re-run tasks that already succeeded
- All downstream tasks of the failed tasks are also re-executed (dependency re-evaluation)

### How to Trigger a Repair Run
**Via UI:**
1. Navigate to the failed run's detail page
2. Click **Repair run** button
3. Optionally select specific tasks to re-run (or keep the default: all failed/skipped)
4. Click **Repair**

**Via CLI:**
```bash
databricks jobs repair-run \
  --run-id 678901 \
  --rerun-tasks '["task_transform", "task_load"]'
```

**Via REST API:**
```bash
POST /api/2.1/jobs/runs/repair
{
  "run_id": 678901,
  "rerun_all_failed_tasks": true
}
```

### Repair Run Requirements
- The original run must be in a terminal state (`Failed`, `Canceled`, `Timed out`)
- The job's cluster must still be available (or a new cluster will be provisioned)
- The code/data must be in a state where re-running is valid (ideally idempotent)

### Idempotency Requirement
For repair runs to be safe, tasks must be **idempotent** — running them twice must produce the same result:

```python
# Safe for repair run — OVERWRITE is idempotent
df.write.mode("overwrite").saveAsTable("gold.summary")

# Safe for repair run — MERGE handles duplicates
spark.sql("""
  MERGE INTO gold.summary t
  USING new_data s ON t.id = s.id
  WHEN MATCHED THEN UPDATE SET *
  WHEN NOT MATCHED THEN INSERT *
""")

# NOT safe for repair run — APPEND creates duplicate rows
df.write.mode("append").saveAsTable("gold.summary")  # Avoid for repairable tasks
```

## 9. Task Retries — Configuration

Configure **automatic retries** in the task definition in `databricks.yml` or the UI:

### In the UI
Task settings → Advanced → **Retries**: set `max retries` and `retry interval`

### In YAML (bundle configuration)
```yaml
resources:
  jobs:
    etl_job:
      tasks:
        - task_key: api_call
          notebook_task:
            notebook_path: ./src/api_ingest.ipynb
          max_retries: 3                 # Retry up to 3 times
          min_retry_interval_millis: 60000  # Wait 60 seconds between retries
          retry_on_timeout: true         # Also retry on timeout, not just error
          timeout_seconds: 1800          # Fail the task if it runs > 30 minutes
```

### Retry Behavior
| Parameter | Description |
|---|---|
| `max_retries` | Max number of automatic retry attempts (0 = no retry) |
| `min_retry_interval_millis` | Minimum wait time between retries in milliseconds |
| `retry_on_timeout` | Whether a timeout counts as a retryable failure |
| `timeout_seconds` | Time limit per task attempt |

> A task with `max_retries: 3` can attempt up to **4 times** total (1 original + 3 retries).

### Retries vs. Repair Run — When to Use Each
| Situation | Use |
|---|---|
| Transient errors (network, API rate limits) | Task retries (automated) |
| Data issue that was fixed manually | Repair run (manual) |
| Infrastructure failure (cluster died) | Repair run (manual) |
| First attempt always fails (code bug) | Fix code, then re-run or repair |

## 10. System Tables for Long-Term Trend Analysis

System tables in the `system.lakeflow` schema record all job and task execution history for up to **365 days**.

### Key System Tables

| Table | Contents |
|---|---|
| `system.lakeflow.job_run_timeline` | One row per job run — start time, end time, duration, result state, trigger type |
| `system.lakeflow.job_task_run_timeline` | One row per task run — task-level duration, state, attempt number |
| `system.lakeflow.jobs` | Job definitions (metadata) |

### Column Reference — `job_run_timeline`

| Column | Description |
|---|---|
| `job_id` | Unique job identifier |
| `run_id` | Unique run identifier |
| `run_name` | Human-readable name of the run |
| `trigger_type` | `PERIODIC`, `ONE_TIME`, `FILE_ARRIVAL`, `REPAIR` |
| `state` | `SUCCEEDED`, `FAILED`, `CANCELED`, `TIMED_OUT` |
| `start_time` | Run start timestamp |
| `end_time` | Run end timestamp |
| `duration` | `end_time - start_time` in seconds |
| `workspace_id` | Workspace identifier (for account-level queries) |

## 11. Key System Table Queries

### Query 1: Average Run Duration per Job (Last 30 Days)
```sql
SELECT
  job_id,
  COUNT(*) AS total_runs,
  AVG(DATEDIFF(second, start_time, end_time)) AS avg_duration_seconds,
  MAX(DATEDIFF(second, start_time, end_time)) AS max_duration_seconds,
  PERCENTILE(DATEDIFF(second, start_time, end_time), 0.95) AS p95_duration_seconds
FROM system.lakeflow.job_run_timeline
WHERE start_time >= CURRENT_TIMESTAMP - INTERVAL 30 DAYS
  AND state = 'SUCCEEDED'
GROUP BY job_id
ORDER BY avg_duration_seconds DESC;
```

### Query 2: Failure Rate per Job (Last 7 Days)
```sql
SELECT
  job_id,
  COUNT(*) AS total_runs,
  SUM(CASE WHEN state = 'FAILED' THEN 1 ELSE 0 END) AS failed_runs,
  ROUND(SUM(CASE WHEN state = 'FAILED' THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 1) AS failure_rate_pct
FROM system.lakeflow.job_run_timeline
WHERE start_time >= CURRENT_TIMESTAMP - INTERVAL 7 DAYS
GROUP BY job_id
HAVING failure_rate_pct > 10    -- Alert on jobs with >10% failure rate
ORDER BY failure_rate_pct DESC;
```

### Query 3: Task-Level Duration Breakdown
```sql
SELECT
  job_id,
  task_key,
  AVG(DATEDIFF(second, start_time, end_time)) AS avg_task_duration_seconds
FROM system.lakeflow.job_task_run_timeline
WHERE start_time >= CURRENT_TIMESTAMP - INTERVAL 14 DAYS
  AND state = 'SUCCEEDED'
GROUP BY job_id, task_key
ORDER BY avg_task_duration_seconds DESC;
```

### Query 4: Daily Run Count by Outcome
```sql
SELECT
  DATE(start_time) AS run_date,
  state,
  COUNT(*) AS run_count
FROM system.lakeflow.job_run_timeline
WHERE job_id = 12345
  AND start_time >= CURRENT_TIMESTAMP - INTERVAL 90 DAYS
GROUP BY DATE(start_time), state
ORDER BY run_date DESC, state;
```

## 12. Notifications & Alerting

Configure job-level email notifications via the UI or YAML:

```yaml
resources:
  jobs:
    etl_job:
      email_notifications:
        on_start:
          - team@company.com
        on_success:
          - team@company.com
        on_failure:
          - oncall@company.com
          - pager@company.com
        no_alert_for_skipped_runs: true   # Don't alert when run is skipped

      # Webhook notifications (Slack, PagerDuty, etc.)
      webhook_notifications:
        on_failure:
          - id: "slack-webhook-id"        # Defined in workspace notification destinations

      # Health rules — alert if SLA breached
      health:
        rules:
          - metric: RUN_DURATION_SECONDS
            op: GREATER_THAN
            value: 7200                   # Alert if run takes > 2 hours
```

## 13. Performance Trend Best Practices

### Core Metrics to Track
| Metric | Source | Action if Degrading |
|---|---|---|
| Average run duration | `system.lakeflow.job_run_timeline` | Profile tasks, check data volume growth |
| Failure rate | `system.lakeflow.job_run_timeline` | Investigate error patterns, add retries |
| P95 task duration | `system.lakeflow.job_task_run_timeline` | Identify slow outlier tasks |
| Skipped task rate | `system.lakeflow.job_task_run_timeline` | Review dependency chain health |

### Optimization Recommendations
- Use **Jobs compute** (not All-purpose clusters) for automated jobs — faster startup, lower cost
- Monitor **Worker CPU Utilization** and **CPU Wait Time** for right-sizing
- Enable **autoscaling** when workload is variable
- Update Databricks Runtime regularly for performance improvements
- Implement **idempotent writes** to make repair runs safe
- Build custom dashboards from `system.lakeflow` tables for SLA tracking

### UI vs. System Tables Decision
| Need | Tool |
|---|---|
| Debug a specific recent failure | Jobs UI → Runs tab → Task run details |
| Check last 5 run outcomes at a glance | Job list view (mini dots) |
| Visual pattern across runs | Matrix view |
| 30-day duration trend | System tables query |
| Year-over-year failure rate report | System tables (365-day retention) |
| Real-time streaming task metrics | Task run details → Metrics tab |

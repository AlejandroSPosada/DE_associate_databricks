## 1. Unity Catalog Security Model Overview

Unity Catalog (UC) uses a **hierarchical, grant-based security model**:

```
Metastore (root)
    └── Catalog
        └── Schema (Database)
            └── Table / View / Volume / Function / Model
                                    ↑
             Privileges flow DOWN this hierarchy
```

### Key Principles
1. **Deny-by-default**: If no privilege is granted, access is denied
2. **Hierarchical inheritance**: Granting on a parent cascades to children
3. **Explicit DENY overrides GRANT**: A DENY takes precedence at any level
4. **Ownership = all privileges**: The owner has full control over their object
5. **USE privileges required**: To access any object, USE must be granted on all parent containers

## 2. Principals — Users, Groups, Service Principals

A **principal** is who receives a privilege.

| Principal Type | Description | How Referenced in SQL |
|---|---|---|
| **User** | Individual Databricks user (email address) | `` `frank@company.com` `` |
| **Group** | Collection of users managed in UC | `` `data-engineers` `` |
| **Service Principal** | Non-human identity for apps/automation | `` `my-service-principal` `` or by application ID |
| **account users** | Special group representing all users in the account | `` `account users` `` |

### Best Practice: Use Groups
Always grant privileges to **groups** rather than individual users:
- When a user joins/leaves, only their group membership changes
- Privileges don't need to be individually re-granted or revoked

```sql
-- Good: grant to a group
GRANT SELECT ON TABLE catalog.schema.orders TO `data-analysts`;

-- Avoid: grant to individual users (harder to maintain)
GRANT SELECT ON TABLE catalog.schema.orders TO `user1@company.com`;
```

## 4. Privilege Types by Object

### Catalog-Level Privileges
| Privilege | Description |
|---|---|
| `USE CATALOG` | Required gateway privilege to access any object in the catalog |
| `CREATE SCHEMA` | Create new schemas inside the catalog |
| `APPLY TAG` | Apply tags to catalog objects |
| `BROWSE` | View catalog metadata without data access |

### Schema-Level Privileges
| Privilege | Description |
|---|---|
| `USE SCHEMA` | Required gateway to access any object in the schema |
| `CREATE TABLE` | Create new tables in the schema |
| `CREATE VIEW` | Create views in the schema |
| `CREATE VOLUME` | Create volumes in the schema |
| `CREATE FUNCTION` | Create functions in the schema |
| `SELECT` | Query tables/views within the schema (inherited) |
| `MODIFY` | Insert, update, delete data in tables (inherited) |
| `EXECUTE` | Call functions in the schema |

### Table / View Level Privileges
| Privilege | Description |
|---|---|
| `SELECT` | Read data from the table or view |
| `MODIFY` | Insert, update, delete rows |
| `APPLY TAG` | Apply tags to the table |
| `MANAGE` | Manage privileges and ownership |

### External Location Privileges
| Privilege | Description |
|---|---|
| `READ FILES` | Read files from the cloud storage path |
| `WRITE FILES` | Write files to the cloud storage path |
| `CREATE EXTERNAL TABLE` | Create external tables using this location |
| `BROWSE` | View external location metadata |

### Volume Privileges
| Privilege | Description |
|---|---|
| `READ VOLUME` | Read files from the volume |
| `WRITE VOLUME` | Write files to the volume |

## 5. USE Privileges — The Gateway Pattern

To access any UC object, a user must have `USE` privileges on **all parent containers** in the hierarchy.

### Example: Reading from a Table
```sql
-- To run: SELECT * FROM catalog.schema.my_table
-- The user needs ALL THREE of these:
GRANT USE CATALOG ON CATALOG my_catalog TO `analysts`;
GRANT USE SCHEMA  ON SCHEMA  my_catalog.my_schema TO `analysts`;
GRANT SELECT      ON TABLE   my_catalog.my_schema.my_table TO `analysts`;

-- Without USE CATALOG → "Catalog not found" error
-- Without USE SCHEMA → "Schema not found" error
-- Without SELECT → "User does not have SELECT on table" error
```

### Why USE Privileges Are Separate
- `USE CATALOG` does NOT grant read access to any data — it only allows navigation
- `USE SCHEMA` does NOT grant read access to any table — it only allows schema navigation
- This separation enables fine-grained control over which tables within a schema are accessible

## 6. GRANT Syntax

```sql
GRANT <privilege> ON <securable_type> <securable_name> TO <principal>;

-- Grant multiple privileges in one statement
GRANT SELECT, MODIFY ON TABLE catalog.schema.orders TO `data-engineers`;

-- Grant on a catalog (cascades to all schemas and tables within)
GRANT USE CATALOG ON CATALOG prod_catalog TO `all-analysts`;

-- Grant on a schema (cascades to all tables within)
GRANT SELECT, MODIFY ON SCHEMA prod_catalog.sales TO `sales-engineers`;

-- Grant on a single table
GRANT SELECT ON TABLE prod_catalog.sales.orders TO `data-analysts`;

-- Grant on an external location
GRANT READ FILES ON EXTERNAL LOCATION raw_data TO `etl-service`;

-- Grant CREATE TABLE on a schema
GRANT CREATE TABLE ON SCHEMA prod_catalog.staging TO `data-engineers`;

-- Grant to all users in the account
GRANT SELECT ON TABLE catalog.schema.public_data TO `account users`;
```

## 7. REVOKE Syntax

```sql
REVOKE <privilege> ON <securable_type> <securable_name> FROM <principal>;

-- Revoke a specific privilege
REVOKE MODIFY ON TABLE catalog.schema.orders FROM `junior-analysts`;

-- Revoke multiple privileges
REVOKE SELECT, MODIFY ON SCHEMA catalog.staging FROM `external-user@vendor.com`;

-- Revoke ALL PRIVILEGES on a table
REVOKE ALL PRIVILEGES ON TABLE catalog.schema.orders FROM `old-team`;
```

### REVOKE Behavior
- `REVOKE` succeeds even if the privilege was never granted (idempotent)
- Revoking from a group removes access for all members of that group
- Revoking at a parent level (schema) does NOT automatically revoke explicit grants at child level (table) — each must be revoked individually

## 8. DENY Privilege — Override Behavior

In Unity Catalog, `DENY` explicitly blocks access, overriding any inherited or explicit `GRANT`.

### How DENY Works
```sql
-- Scenario: A user is in both `all-analysts` and `finance-restricted`
-- all-analysts has SELECT on finance schema (inherited)
-- We want to explicitly block this user from finance.payroll table

DENY SELECT ON TABLE catalog.finance.payroll TO `contractor@vendor.com`;
-- → Despite being in all-analysts (which has SELECT), DENY wins
```

### DENY Precedence Rules
| Situation | Result |
|---|---|
| GRANT exists, no DENY | Access allowed |
| No GRANT, no DENY | Access denied (default) |
| DENY exists (any level) | Access **denied, DENY wins** |
| GRANT at table, DENY at schema | **DENY wins** — schema-level DENY blocks table access |
| DENY at catalog | Denies access to all schemas and tables in that catalog |

### DENY Inheritance
- DENY on a catalog → implicitly denies on all schemas and tables within
- DENY on a schema → implicitly denies on all tables/views within
- DENY is the **strongest** permission state

### DENY vs. Not Granting
| Approach | Effect | When to Use |
|---|---|---|
| Simply don't grant | No access (default deny) | Normal case — just don't give access |
| Explicit DENY | No access, even if future GRANT is added | Block a specific principal even when they're in a group with access |

## 9. ALL PRIVILEGES — Special Keyword

`ALL PRIVILEGES` grants all applicable privileges for that object type in a single statement.

```sql
-- Grant all applicable privileges on a table
GRANT ALL PRIVILEGES ON TABLE catalog.schema.orders TO `table-owner-group`;

-- Grant all applicable privileges on a schema
GRANT ALL PRIVILEGES ON SCHEMA catalog.schema TO `schema-admins`;

-- Grant all privileges on an external location
GRANT ALL PRIVILEGES ON EXTERNAL LOCATION raw_data TO `admin-group`;
```

### What ALL PRIVILEGES Includes (and Excludes)

| Object | Included | NOT Included |
|---|---|---|
| Table | SELECT, MODIFY, APPLY TAG | MANAGE |
| Schema | USE SCHEMA, SELECT, MODIFY, CREATE TABLE, etc. | MANAGE |
| External Location | READ FILES, WRITE FILES, CREATE EXTERNAL TABLE | MANAGE |

> **Critical**: `ALL PRIVILEGES` does **NOT** include `MANAGE`. This prevents accidental privilege escalation — you cannot accidentally give someone admin control of the object via `ALL PRIVILEGES`.

## 10. Privilege Inheritance — Top-Down Flow

Privileges granted on a parent object **cascade** to all current and future child objects.

### Inheritance Examples
```sql
-- Grant SELECT on the schema → inherits to all tables in the schema
GRANT SELECT ON SCHEMA catalog.analytics TO `analysts`;
-- Effect: analysts can SELECT from any table in catalog.analytics
--         including tables created in the future

-- Grant USE CATALOG on catalog → all schemas/tables navigate but no data access
GRANT USE CATALOG ON CATALOG prod TO `analysts`;
-- Effect: analysts can navigate the catalog but cannot read any table

-- Grant SELECT on a specific table → does NOT inherit to schema/catalog
GRANT SELECT ON TABLE catalog.analytics.orders TO `analysts`;
-- Effect: analysts can ONLY read orders — not other tables in the schema
-- Additional: they still need USE CATALOG + USE SCHEMA explicitly!
```

### Inheritance Flow (Top-Down)
```
GRANT SELECT ON CATALOG my_catalog
    └── All SCHEMAS in my_catalog → SELECT inherited
        └── All TABLES/VIEWS in each SCHEMA → SELECT inherited

GRANT SELECT ON SCHEMA my_catalog.my_schema
    └── All TABLES/VIEWS in my_schema → SELECT inherited
    (Does NOT affect sibling schemas)

GRANT SELECT ON TABLE my_catalog.my_schema.my_table
    └── Only this specific table
    (Does NOT affect other tables or parent objects)
```

## 11. Ownership — Object Owners and Their Powers

Every UC object has exactly one **owner** (a user, group, or service principal).

### What Owners Can Do
- Perform **all** operations on the object (even without explicit grants)
- Grant/revoke privileges to other principals
- Transfer ownership to another principal
- Manage child objects (schemas within a catalog, tables within a schema)

### Ownership Does NOT Automatically Cascade
- Owning a catalog does not make you the owner of all schemas within it
- Each object has its own owner

### Transferring Ownership
```sql
-- Transfer table ownership to a group
ALTER TABLE catalog.schema.orders
SET OWNER TO `data-engineering-team`;

-- Transfer schema ownership
ALTER SCHEMA catalog.my_schema
SET OWNER TO `platform-team`;

-- Transfer catalog ownership
ALTER CATALOG prod_catalog
SET OWNER TO `catalog-admin-group`;
```

### Who Can Grant Privileges
1. The **owner** of the object
2. The **owner** of the parent catalog or schema
3. A **metastore admin**
4. A user with the **MANAGE** privilege on the object

## 12. MANAGE Privilege — Delegated Administration

`MANAGE` allows a principal to manage privileges on an object **without being the owner**.

```sql
-- Grant MANAGE on a schema to a team lead
GRANT MANAGE ON SCHEMA catalog.analytics TO `analytics-lead`;
-- Now analytics-lead can:
-- GRANT/REVOKE privileges to others on catalog.analytics
-- But cannot: transfer ownership, create objects, read data
```

### MANAGE vs. Ownership
| Action | Owner | MANAGE Holder |
|---|---|---|
| Grant/revoke privileges | Yes | Yes |
| Transfer ownership | Yes | No |
| Create child objects | Yes | No (need CREATE TABLE separately) |
| Read data | Yes | No (need SELECT separately) |
| Delete the object | Yes | No |

> `MANAGE` is NOT included in `ALL PRIVILEGES` — must be granted explicitly.

## 13. SHOW GRANTS — Auditing Permissions

### Syntax
```sql
-- Show all grants on an object (requires MANAGE or ownership)
SHOW GRANTS ON TABLE catalog.schema.orders;

-- Show grants for a specific principal on an object
SHOW GRANTS `frank@company.com` ON TABLE catalog.schema.orders;

-- Show grants on a schema
SHOW GRANTS ON SCHEMA catalog.analytics;

-- Show grants on a catalog
SHOW GRANTS ON CATALOG prod_catalog;

-- Show grants on external location
SHOW GRANTS ON EXTERNAL LOCATION raw_data;
```

### SHOW GRANTS Output Columns
| Column | Description |
|---|---|
| `Principal` | User, group, or service principal |
| `ActionType` | The privilege type (SELECT, MODIFY, etc.) |
| `ObjectType` | TABLE, SCHEMA, CATALOG, etc. |
| `ObjectKey` | Full name of the securable object |

### Who Can Run SHOW GRANTS
- Metastore admin
- Object owner
- User with `MANAGE` privilege on the object (plus USE on parent containers)

## 14. Working with Groups

### Creating and Managing Groups (Admin)
Groups are managed at the account level (not workspace-specific).

```sql
-- Grant to a group (all members get the privilege)
GRANT SELECT ON TABLE catalog.schema.orders TO `data-analysts`;

-- When a new user joins data-analysts group, they automatically get SELECT
-- When a user leaves data-analysts group, they automatically lose SELECT

-- Nested groups: members of child groups inherit parent group's privileges
-- If user is in `junior-engineers` which is nested under `engineers`:
-- GRANT SELECT TO `engineers` → junior-engineers members get SELECT too
```

### Account Users Group
```sql
-- Grant access to ALL users in the Databricks account
GRANT SELECT ON TABLE catalog.schema.public_reference TO `account users`;
```

## 15. Service Principals — Programmatic Access

Service principals represent **non-human identities** used by:
- CI/CD pipelines (Databricks Jobs, Lakeflow)
- DABs (Declarative Automation Bundles)
- External applications
- Terraform automation

### Granting Privileges to Service Principals
```sql
-- Grant to a service principal by name
GRANT SELECT ON TABLE catalog.schema.orders TO `etl-pipeline-sp`;

-- Grant to a service principal by application ID
GRANT CREATE TABLE ON SCHEMA catalog.staging TO `my-sp-application-id`;
```

### Common Service Principal Grants for Pipelines
```sql
-- ETL pipeline service principal needs:
GRANT USE CATALOG ON CATALOG prod       TO `etl-sp`;
GRANT USE SCHEMA  ON SCHEMA prod.raw    TO `etl-sp`;
GRANT SELECT      ON SCHEMA prod.raw    TO `etl-sp`;
GRANT USE SCHEMA  ON SCHEMA prod.silver TO `etl-sp`;
GRANT MODIFY      ON SCHEMA prod.silver TO `etl-sp`;
GRANT CREATE TABLE ON SCHEMA prod.silver TO `etl-sp`;
```

## 16. Minimum Required Privileges — Common Scenarios

### Scenario 1: Read-Only Analyst
```sql
-- Analyst can only read from a specific table
GRANT USE CATALOG ON CATALOG prod TO `analyst@company.com`;
GRANT USE SCHEMA  ON SCHEMA prod.analytics TO `analyst@company.com`;
GRANT SELECT      ON TABLE prod.analytics.orders TO `analyst@company.com`;
```

### Scenario 2: Data Engineer — Full Schema Access
```sql
-- Engineer can read/write all tables in a schema
GRANT USE CATALOG  ON CATALOG prod TO `data-engineers`;
GRANT USE SCHEMA   ON SCHEMA prod.staging TO `data-engineers`;
GRANT SELECT       ON SCHEMA prod.staging TO `data-engineers`;
GRANT MODIFY       ON SCHEMA prod.staging TO `data-engineers`;
GRANT CREATE TABLE ON SCHEMA prod.staging TO `data-engineers`;
```

### Scenario 3: Schema Admin (with delegated management)
```sql
-- Schema admin can manage permissions for others
GRANT USE CATALOG ON CATALOG prod TO `schema-admin`;
GRANT USE SCHEMA  ON SCHEMA prod.analytics TO `schema-admin`;
GRANT MANAGE      ON SCHEMA prod.analytics TO `schema-admin`;
```

### Scenario 4: External ETL Tool — Read Raw Files
```sql
-- External tool needs to read files from external location
GRANT READ FILES ON EXTERNAL LOCATION raw_landing TO `external-etl-sp`;
GRANT CREATE EXTERNAL TABLE ON EXTERNAL LOCATION raw_landing TO `external-etl-sp`;
```

### Scenario 5: Restrict Specific User from Inherited Access
```sql
-- All analysts have SELECT on prod.finance schema
-- But restrict contractor from the payroll table
GRANT SELECT ON SCHEMA prod.finance TO `data-analysts`;
-- contractor@vendor.com is in data-analysts but should not see payroll
DENY SELECT ON TABLE prod.finance.payroll TO `contractor@vendor.com`;
-- DENY wins over the inherited GRANT from the schema level
```

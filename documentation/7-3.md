## 1. Overview — Fine-Grained Access Control in Unity Catalog

Unity Catalog provides two enforcement-point mechanisms for controlling **what data users can see within a table**:

| Mechanism | What It Controls | Level |
|---|---|---|
| **Column Masking** | Controls which value a user sees in a specific column | Column-level |
| **Row-Level Security (Row Filter)** | Controls which rows a user can see | Row-level |

Both work by:
1. Defining a **SQL UDF (User-Defined Function)** that contains the access logic
2. **Attaching** the UDF to a table column (for masks) or to the table (for row filters)
3. The policy is **transparent** — queries look normal but results are filtered/masked at query time

### Key Identity Functions
- `current_user()` — returns the email/username of the executing user
- `is_account_group_member('group-name')` — returns TRUE if the current user is in the group

## 2. Column Masking — Core Concept

A **column mask** replaces the actual column value with a masked/transformed value for users who don't have privilege to see the real data.

### How It Works
```
User queries: SELECT ssn FROM users;

Without mask:         With mask (non-admin user):
ssn                   ssn
-----------           -----------
123-45-6789    →      ***-**-****
987-65-4321           ***-**-****

With mask (admin user):
ssn
-----------
123-45-6789      ← sees real value
987-65-4321
```

### Key Properties
- The mask function runs **per-row** at query time
- The original data is **never changed** in storage — masking is purely at query time
- Column masks are **invisible to users** — the query looks normal, results are masked
- The masked column **still appears** in the schema — users can see the column exists

## 3. Creating a Column Mask UDF

A column mask function is a **scalar SQL UDF** that:
- Takes the column's value as input (same data type as the column)
- Returns the same data type as the column
- Uses identity functions to determine what to return based on who is querying

### Example 1: SSN Masking (admin sees full, others see masked)
```sql
CREATE OR REPLACE FUNCTION catalog.schema.ssn_mask(ssn STRING)
RETURNS STRING
RETURN CASE
  WHEN is_account_group_member('pii-authorized')
  THEN ssn                     -- authorized group sees real value
  ELSE '***-**-****'           -- everyone else sees masked value
END;
```

### Example 2: Email Domain Masking
```sql
CREATE OR REPLACE FUNCTION catalog.schema.email_mask(email STRING)
RETURNS STRING
RETURN CASE
  WHEN is_account_group_member('data-admins') THEN email
  WHEN current_user() = 'frank@company.com' THEN email  -- owner always sees real
  ELSE CONCAT('****@', SPLIT(email, '@')[1])  -- show domain only
END;
```

### Example 3: Salary — Show Only If Manager
```sql
CREATE OR REPLACE FUNCTION catalog.hr.salary_mask(salary DOUBLE)
RETURNS DOUBLE
RETURN CASE
  WHEN is_account_group_member('hr-managers') THEN salary
  ELSE NULL   -- non-managers see NULL for salary
END;
```

## 4. Applying a Column Mask — ALTER TABLE SET MASK

### Syntax
```sql
ALTER TABLE <table_name>
ALTER COLUMN <column_name>
SET MASK <function_name>
[ USING COLUMNS (<additional_col1>, <additional_col2>, ...) ];
```

### Examples
```sql
-- Apply SSN mask to the ssn column
ALTER TABLE catalog.schema.users
ALTER COLUMN ssn
SET MASK catalog.schema.ssn_mask;

-- Apply email mask
ALTER TABLE catalog.schema.customers
ALTER COLUMN email
SET MASK catalog.schema.email_mask;

-- Apply salary mask (only managers see real value)
ALTER TABLE catalog.hr.employees
ALTER COLUMN salary
SET MASK catalog.hr.salary_mask;
```

### Using USING COLUMNS (Multi-Column Context)
When the mask function needs to look at OTHER columns to decide how to mask:
```sql
-- Mask function needs to check the user's department to decide masking
CREATE OR REPLACE FUNCTION catalog.hr.dept_salary_mask(salary DOUBLE, dept STRING)
RETURNS DOUBLE
RETURN CASE
  WHEN is_account_group_member('hr-managers') THEN salary
  WHEN current_user() = CONCAT(dept, '-manager@company.com') THEN salary
  ELSE NULL
END;

-- Apply the mask, passing the 'department' column as additional context
ALTER TABLE catalog.hr.employees
ALTER COLUMN salary
SET MASK catalog.hr.dept_salary_mask
USING COLUMNS (department);  -- 'department' column value passed as 2nd arg
```

## 5. Removing a Column Mask

```sql
-- Remove the mask from a specific column
ALTER TABLE catalog.schema.users
ALTER COLUMN ssn
DROP MASK;

-- After this, all users see the real SSN value
```

## 6. Row-Level Security — Core Concept

A **row filter** limits which rows a user sees when querying a table. Rows that don't pass the filter are completely invisible — the user can't see them or know they exist.

### How It Works
```
Table: orders (5 rows)
region  | order_id | amount
--------+----------+--------
US      | 1001     | 500
EU      | 1002     | 300
US      | 1003     | 750
APAC    | 1004     | 200
EU      | 1005     | 450

Query by EU analyst: SELECT * FROM orders;
region  | order_id | amount
--------+----------+--------
EU      | 1002     | 300    ← only EU rows visible
EU      | 1005     | 450

Query by admin: SELECT * FROM orders;
→ All 5 rows visible
```

### Key Properties
- Row filter function must return **BOOLEAN**
- Rows where the function returns `FALSE` or `NULL` are **excluded**
- Only rows where function returns `TRUE` are returned
- The filter is **transparent** — users see no indication that rows are hidden

---

## 7. Creating a Row Filter UDF

A row filter function is a **scalar SQL UDF** that:
- Returns **BOOLEAN** (`TRUE` = include row, `FALSE`/`NULL` = exclude row)
- Can take column values as arguments for context
- Uses identity functions to make user-aware decisions

### Example 1: Region-Based Row Filter
```sql
CREATE OR REPLACE FUNCTION catalog.schema.region_filter(region STRING)
RETURNS BOOLEAN
RETURN is_account_group_member('admin')              -- admins see all
    OR is_account_group_member(CONCAT(region, '-team')); 
    -- EU users in 'EU-team' group see only EU rows
    -- US users in 'US-team' group see only US rows
```

### Example 2: My Own Records Filter
```sql
CREATE OR REPLACE FUNCTION catalog.schema.owner_filter(record_owner STRING)
RETURNS BOOLEAN
RETURN is_account_group_member('admin')
    OR current_user() = record_owner;
-- Each user sees only their own records (and admins see all)
```

### Example 3: Time-Based Data Access
```sql
CREATE OR REPLACE FUNCTION catalog.schema.data_tier_filter(tier STRING)
RETURNS BOOLEAN
RETURN is_account_group_member('premium-users')  -- premium sees all tiers
    OR tier = 'public';                           -- free users see only public
```

---

## 8. Applying a Row Filter — ALTER TABLE SET ROW FILTER

### Syntax
```sql
ALTER TABLE <table_name>
SET ROW FILTER <function_name>
ON ( <column_name1> [, <column_name2>, ...] );
-- The ON clause specifies which columns are passed to the function
```

### Examples
```sql
-- Apply region filter, passing the 'region' column to the function
ALTER TABLE catalog.sales.orders
SET ROW FILTER catalog.sales.region_filter
ON (region);

-- Apply owner filter, passing 'created_by' column
ALTER TABLE catalog.tasks.user_tasks
SET ROW FILTER catalog.tasks.owner_filter
ON (created_by);

-- Filter with no column arguments (filter is purely based on identity)
-- Function takes no parameters — returns TRUE for admins only
CREATE OR REPLACE FUNCTION catalog.schema.admin_only()
RETURNS BOOLEAN
RETURN is_account_group_member('admins');

ALTER TABLE catalog.schema.sensitive_table
SET ROW FILTER catalog.schema.admin_only ON ();
```

---

## 9. Row Filter at Table Creation Time

Row filters can also be applied when creating a table (prevents initial data exposure):

```sql
CREATE TABLE catalog.schema.orders (
  order_id    BIGINT,
  region      STRING,
  amount      DECIMAL(10,2),
  created_by  STRING
)
WITH ROW FILTER catalog.schema.region_filter ON (region);
```

---

## 10. Removing a Row Filter

```sql
-- Remove the row filter from the table
ALTER TABLE catalog.sales.orders
DROP ROW FILTER;

-- After this, all users see all rows
```

## 11. Identity Functions — current_user() & is_account_group_member()

These are the core building blocks for dynamic, user-aware masks and filters.

### `current_user()`
```sql
-- Returns the current executing user's identity (email or username)
SELECT current_user();
-- Returns: 'frank@company.com'

-- Use in UDF:
CASE WHEN current_user() = 'admin@company.com' THEN real_value ELSE masked END
```

### `is_account_group_member('group-name')`
```sql
-- Returns TRUE if current user is a member of the specified ACCOUNT-level group
SELECT is_account_group_member('data-admins');
-- Returns: TRUE or FALSE

-- Use in UDF:
CASE WHEN is_account_group_member('pii-authorized') THEN ssn ELSE '***' END
```

### `is_account_group_member()` vs. `is_member()` (Legacy)
| Function | Scope | Use In UC |
|---|---|---|
| `is_account_group_member()` | Account-level groups | **Preferred** for Unity Catalog |
| `is_member()` | Workspace-local groups | Legacy — may have scope issues in UC |

> Always use `is_account_group_member()` in Unity Catalog environments.

### Performance Considerations
- Do **NOT** mark UDFs using these functions as `DETERMINISTIC` — they are session-stable but not deterministic across sessions
- Keep UDF logic simple (`CASE` statements) to allow the engine to perform predicate pushdown
- Complex UDFs with subqueries or joins can significantly degrade query performance (run per-row)

## 12. USING COLUMNS — Passing Context to Mask/Filter Functions

When a mask or filter function needs data from **other columns** in the same row, use `USING COLUMNS`:

### For Column Masks
```sql
-- Function signature: mask(column_to_mask, context_column)
CREATE OR REPLACE FUNCTION catalog.schema.conditional_mask(
  value STRING,
  sensitivity_level STRING  -- from another column
)
RETURNS STRING
RETURN CASE
  WHEN sensitivity_level = 'PUBLIC' THEN value           -- show if public
  WHEN is_account_group_member('data-stewards') THEN value  -- stewards see all
  ELSE REGEXP_REPLACE(value, '.', '*')   -- mask others
END;

ALTER TABLE catalog.schema.documents
ALTER COLUMN content
SET MASK catalog.schema.conditional_mask
USING COLUMNS (sensitivity_level);  -- passes the 'sensitivity_level' column value
```

### For Row Filters (Using ON Clause)
```sql
-- ON clause passes column values to the filter function
ALTER TABLE catalog.sales.orders
SET ROW FILTER catalog.sales.region_filter
ON (region);       -- 'region' column value is passed to region_filter(region STRING)
```

## 13. Required Privileges for Masks & Filters

To **apply** a column mask or row filter to a table:

| Requirement | Detail |
|---|---|
| **On the UDF function** | `USE CATALOG` + `USE SCHEMA` + `EXECUTE` on the function |
| **On the table** | Must be the table **owner** OR have `MANAGE` privilege on the table |

```sql
-- Granting the EXECUTE privilege on the mask function
GRANT EXECUTE ON FUNCTION catalog.schema.ssn_mask TO `data-engineers`;

-- The user applying the mask also needs table ownership or MANAGE
```

To **query** a table with a mask/filter:
- Users only need `SELECT` on the table — they don't need `EXECUTE` on the UDF
- The mask/filter runs automatically — users are unaware it exists

## 14. ABAC Policies — Scalable Governance (2026)

For large organizations managing hundreds of tables, **ABAC (Attribute-Based Access Control) policies** provide scalable row/column security via governed tags.

### How ABAC Differs from Table-Level Policies
| Feature | Table-Level (ALTER TABLE) | ABAC Policy (CREATE POLICY) |
|---|---|---|
| Scope | Single table | Catalog or Schema |
| Configuration | Per-table ALTER command | Governed tags → automatic |
| Scalability | Low (manage per table) | High (policy applies to all tagged objects) |
| DBR Requirement | DBR 15.4+ | DBR 16.4+ |

### ABAC Column Mask Example
```sql
-- Create a policy that applies to any column tagged with 'sensitivity = PII'
CREATE POLICY mask_pii_policy ON CATALOG my_catalog
COLUMN MASK ssn_mask_function
TO `account users`
WHEN COLUMN TAG('sensitivity') = 'PII';
-- Any table in my_catalog with a column tagged sensitivity=PII gets masked
```

### ABAC Row Filter Example
```sql
-- Apply row filter to all tables tagged as 'access = restricted'
CREATE POLICY restrict_rows_policy ON SCHEMA my_catalog.finance
ROW FILTER finance_access_filter
WHEN TABLE TAG('access') = 'restricted';
```

## 15. Limitations & Compatibility

### Feature Compatibility

| Feature | Compatible with Masks/Row Filters? |
|---|---|
| **Time Travel** (`VERSION AS OF`, `TIMESTAMP AS OF`) | Generally NOT supported |
| **DEEP CLONE / SHALLOW CLONE** | NOT supported on tables with active masks/filters |
| **Delta Sharing** | Standard sharing NOT supported; ABAC has special considerations |
| **Generated columns** | Column masks cannot be applied to columns referenced by generated columns |
| **Views** | Cannot apply policies directly to views; but base table policies apply when views are queried |
| **Streaming (Structured Streaming)** | Limited support — check DBR version |

### Compute Requirements
| Policy Type | Minimum DBR |
| `ALTER TABLE SET ROW FILTER / SET MASK` | DBR 15.4 LTS |
| ABAC policies | DBR 16.4+ |

### Quotas (per Metastore)
| Object | Quota |
|---|---|
| Total policies | 10,000 |
| Policies per catalog/schema | 100 |
| Policies per table | 50 |

### Performance Considerations
- Row filter UDFs execute as a hidden `WHERE` clause on **every query** — keep them simple
- Column mask UDFs execute on **every row** for the masked column — avoid subqueries
- Complex UDFs with joins can cause significant query slowdown

## 16. Column Masking vs. Row Filtering — Summary

| Aspect | Column Masking | Row Filtering |
|---|---|---|
| **Purpose** | Hide/transform column values | Hide entire rows |
| **UDF return type** | Same type as the column (STRING, DOUBLE, etc.) | BOOLEAN |
| **Applied with** | `ALTER TABLE ... ALTER COLUMN ... SET MASK` | `ALTER TABLE ... SET ROW FILTER` |
| **Column creation** | `ALTER TABLE ... ALTER COLUMN ...` | `CREATE TABLE ... WITH ROW FILTER` |
| **Removed with** | `ALTER TABLE ... ALTER COLUMN ... DROP MASK` | `ALTER TABLE ... DROP ROW FILTER` |
| **Context args** | `USING COLUMNS (col1, col2)` | `ON (col1, col2)` |
| **FALSE/NULL behavior** | Returns masked value | Row is excluded |
| **Row visibility** | All rows visible (column shows masked value) | Excluded rows are completely hidden |
| **Identity functions** | `current_user()`, `is_account_group_member()` | `current_user()`, `is_account_group_member()` |
| **Apply privilege** | Table owner or MANAGE + EXECUTE on function | Table owner or MANAGE + EXECUTE on function |
| **Query privilege** | SELECT on table only | SELECT on table only |
